# Эпик 4 — Каналы роста (Онлайн-запись, Лояльность, Уведомления)

> Контур привлечения и удержания поверх реальных данных Эпиков 2–3: публичная запись клиентов, программы лояльности с живыми баллами, уведомления с одним настоящим каналом доставки.
> Источник: [`docs/ROADMAP.md`](ROADMAP.md), Эпик 4.

## Что уже есть в коде (точка старта)

- **Модели БД полностью готовы, миграции не нужны:** `back/models/loyalty.py` — 11 таблиц под все 5 программ (StudioLoyaltyConfig, LoyaltyLevel, ClientLoyaltyCard, LoyaltyPointTransaction, StudioDiscountConfig, StudioCertificateConfig, GiftCertificate, StudioSubscriptionProgramConfig, SubscriptionPackage, StudioReferralConfig, ReferralRecord); `back/models/settings.py` — StudioBookingSettings (правила записи + брендинг виджета), BookingChannelConfig (каналы, config JSON), StudioNotificationSettings, NotificationEventToggle (матрица «роль × событие × канал»). У Reservation уже есть `booking_channel`; у Client — `tg_id`, `notifs_enabled`, `reminders_enabled`.
- **Роутеров нет:** `back/routers/booking/` и `back/routers/loyalty/` не существуют; `back/routers/settings/*` — все заглушки; `back/routers/clients/loyalty.py` — заглушка. Префиксы `/booking` и `/loyalty` не зарегистрированы в `main.py`.
- **Схемы — заглушки:** `back/schemas/loyalty/*`, `back/schemas/settings/booking.py`; схем уведомлений нет вообще.
- **Фронтовый контракт уже задан:** `front/src/api/booking|loyalty|notifications/*` — полные типизированные api-модули (bookingApi: settings + channels; loyaltyApi: config/levels/cards/certificates/subscriptions; notificationsApi: settings + events под префиксом `/settings/notifications`). Ни одна страница их пока не вызывает.
- **UI на 100% моках:** `pages/dashboard/Booking/` (каналы захардкожены, тумблеры uncontrolled, превью StudioMockup недостижимо), `Loyalty/` (5 программ, дровер-формы ничего не сохраняют), `Notifications/` (матрица в локальном стейте, события с id `c1/t1/...` в constants.ts). Кнопки в карточках: Клиенты («Позвонить»/«Сообщение»/«Записать»/«Бонус» в `ClientProfileSlider.tsx` + `useClientActions.ts`) и Сотрудники («Написать»/«Позвонить» в `Staff.tsx`) — тост-заглушки.
- **Отправки нет никакой:** email-код регистрации — `print()` в `back/routers/auth/register.py`; aiogram числится в requirements, но не используется; SMTP-кода нет.
- **Паттерн публичного эндпоинта** (без oauth2_scheme) — `back/routers/auth/register.py`; готовая логика брони с лимитом мест и запретом дублей — `book_lesson` в роутере клиентов.
- **Из Эпика 1:** `get_studio_context` и `require_role`. **Из Эпика 2:** занятия (без них публичной записи не на что записывать). **Из Эпика 3:** `POST /finances/operations` — точка триггера начисления баллов.

## Жёсткие правила эпика (MVP)

1. **Единственный реальный канал доставки — Email (SMTP).** У клиентов в базе уже есть email, а `tg_id` пуст (Telegram-бота нет — клиенту неоткуда было написать боту). Один модуль `back/services/mailer.py` — им же заменяем `print()` в регистрации. Telegram/WhatsApp/Instagram/SMS/Push — только сохранение конфигурации без отправки.
2. **Публичная запись — только API** (`/public/booking/*`, без JWT, studio_id в пути): каталог занятий → бронь по имени и телефону (find-or-create Client, `booking_channel="web"`). UI мини-приложения — post-MVP (раздел 17 ТЗ).
3. **Триггеры лояльности и уведомлений** — обычные вызовы хелперов из существующих эндпоинтов. Никаких очередей и фоновых воркеров; отправка письма — await с try/except: ошибка отправки логируется и не валит запрос.
4. **Диспетчер уведомлений** — одна функция `notify(studio_id, role, event_id, context)`: читает матрицу; включён email — шлёт; остальные каналы молча пропускает. Канонические event_id — строки из фронтовых `NOTIF_EVENTS` (`c1`, `t1`, ...).
5. **Доступ:** все `/loyalty/*`, `/booking/*` (настройки), `/settings/notifications/*` — `require_role("owner")`; `/public/booking/*` — без авторизации.
6. **Контракт-фиксы в задаче схем:** `LoyaltyConfig.expiry_period` — бэкенд диктует строку (фронт-тип `number|null` поправить); короткие ключи NotificationSettings (`telegram`, `email`, ...) ↔ колонки `*_notifications` — маппинг alias-ами в Pydantic-схеме.

**Обозначения сложности:** 🟢 простая · 🟡 средняя · 🔴 сложная, делать внимательно.

## Что специально НЕ входит в Эпик 4

- **Реальная отправка в Telegram/WhatsApp/Instagram/SMS/Push** — только сохранение конфигурации; живой канал один — Email.
- **UI клиентского мини-приложения/виджета** — post-MVP (раздел 17 ТЗ); публичный API готовится под него уже сейчас; StudioMockup остаётся мок-превью.
- **Модель лога уведомлений** — её нет в БД; проверка сквозного потока — по факту получения письма; лог — post-MVP.
- **Скидки и сертификаты в кассовом потоке** (применение скидки к операции; оплата сертификатом) — конфиги сохраняются; интеграция с оплатами — Эпик 5.

## Как проверить (Verification)

1. **Лояльность:** Swagger — включить программу (PATCH /loyalty/config) → создать доходную операцию 1000 ₽ с client_id (Эпик 3) → у клиента +10 баллов (курс 100); кнопка «Бонус» в карточке добавляет ещё; /loyalty/stats отражает.
2. **Уведомления:** заполнить SMTP в .env → включить галочку «подтверждение записи × email» → записать клиента с реальным email → письмо пришло; выключить галочку → не приходит.
3. **Онлайн-запись:** сохранить правила (запись за 2 часа; окно 7 дней) → `curl` без токена: `/public/booking/{id}/services` → `/slots` (слоты уважают правила) → POST `/reserve` → запись видна в Журнале с booking_channel=web; новый клиент появился в Клиентах с source=online.
4. **Фронт:** `cd front && npm run dev` — все три страницы переживают F5; `npm run build && npm run lint` — чисто.
