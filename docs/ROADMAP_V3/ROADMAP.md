# Velora CRM — Roadmap V3 (аудиты страниц, продолжение)

Продолжение серии аудитов из V2 (`docs/ROADMAP_V2/ROADMAP.md`, Аудит 1: Сотрудники).
Формат тот же: проходим страницу, фиксируем что лжёт/не работает/выглядит плохо,
принимаем решения и превращаем их в задачи.

**Принятые здесь общие решения (действуют на всё приложение):**
1. **TanStack Query** — кэш данных на весь фронт: страницы рисуются мгновенно из кэша
   и тихо обновляются в фоне; после любой мутации инвалидация ключа обновляет все места,
   где данные показаны. Решает «данные без F5» и «лаг при переключении страниц» разом.
2. **Единый UI-кит** в `front/src/components/ui/`: Toast, ConfirmModal, Modal-shell,
   Select — по одному компоненту на всё приложение. Внедряем сейчас в Сотрудниках и
   Каталоге, дальше поэтапно (уточняет план V2-2: разрешено, обсуждено 2026-07-14).
3. Контакты филиала (телефон/email) **могут совпадать** с контактами сотрудников и
   владельца — это контакты для клиентов, не логин. Проверяем только формат.
   Уникальность остаётся только у email аккаунта пользователя.

## Аудит 2: Каталог (Catalog) — обсуждено 2026-07-14

### 1. Языки: страница целиком на выбранном языке

**Сейчас:** сама страница (`Catalog.tsx`, `StudioSection.tsx`, `ServiceSection.tsx`)
корректно ходит через `t()`. Но **все четыре модалки — сплошной русский хардкод**:
- `AddStudioModal.tsx` — весь 3-шаговый визард: заголовки шагов, подписи, trust-лейблы,
  кнопки («Продолжить», «Далее — фото», «Создать студию», «Пропустить фото и создать»),
  тексты в SVG-иллюстрациях («Контакты», «Новый филиал», «Город»), зона загрузки фото
  («Нажмите для загрузки», «JPG, PNG до 10 МБ», «Заменить», «Удалить»), блок «Итог»,
  текст ошибки загрузки фото.
- `EditStudio.tsx` (EditStudioModal + HallModal) — «Редактировать филиал», «Новый зал»,
  все label'ы полей, «Отмена»/«Сохранить»/«Создать», «Онлайн-зал», «Цвет».
- `EditService.tsx` — «Редактировать услугу», «Новая услуга», «Категория», «Тип»,
  «Групповая»/«Индивидуальная», «Цена, ₽», «Макс. клиентов», «Описание» и т.д.
- `constants.ts`: `SERVICE_CATEGORIES = ['Йога', 'Пилатес', …]` — русские строки,
  которые ещё и **сохраняются в БД как значение** `category` (см. п.10).
- Ошибки бэка приходят русским текстом («Филиал не найден») и показываются как есть
  на любом языке интерфейса.

**Решение:**
- Все строки модалок → `catalog.json` (ru + en), включая тексты внутри SVG-иллюстраций.
- Категории услуг → ключи (см. п.10).
- Ошибки API: известные статусы (403, 404, 500, сеть) фронт переводит через `t()`;
  осмысленный `detail` (409-дубли и валидация) показываем как есть.
- Смежное: `₽` захардкожен по всей странице (статистика, «Цена, ₽», «₽/ч») —
  брать символ валюты из настроек студии (то же решение, что в Аудите 1).

### 2. Данные обновляются сами, без F5 — TanStack Query

**Сейчас:** каждый хук (`useStudioList`, `useBranchDetail`, `useServiceList`) держит
локальный стейт и рефетчит после своих мутаций, но:
- CRUD зала рефетчит только деталь филиала — **счётчик залов в левом списке
  (`hall_count`) остаётся старым** до F5 или переключения вкладки;
- переключение вкладок Студии ↔ Услуги размонтирует секции и грузит всё заново
  (пустой экран, мигание);
- между страницами синхронизации нет вообще (изменил зал в Каталоге — Журнал не в курсе).

**Решение (принято 2026-07-14): внедряем `@tanstack/react-query`.**
- `QueryClientProvider` в корне приложения; ключи: `['branches']`, `['branch', id]`,
  `['services']` (дальше — `['staff']`, `['clients']` и т.д. по мере миграции страниц).
- Мутации инвалидируют связанные ключи: создание зала → `['branch', id]` **и**
  `['branches']` — левый список и счётчики обновляются сами.
- `staleTime` небольшой (например 30 с) + refetch on window focus — данные подтягиваются
  сами и после действий в другой вкладке.
- Хуки Каталога переписать на useQuery/useMutation, внешний интерфейс для компонентов
  сохранить (рефакторинг без изменения поведения UI).

### 3. Модалки: один премиальный стиль на всё приложение

**Сейчас — три разных мира в одном Каталоге:**
- `AddStudioModal` — 860px двухпанельный визард на инлайн-стилях (свой стиль);
- `EditStudioModal`/`HallModal` — простенький `ModalShell` 460px, «просто треш»
  по сравнению с остальным продуктом;
- `EditService` — **копипаста того же shell'а** ещё раз, 480px, со своими стилями
  `<select>`.

**Решение:**
- Общий модальный кит в `front/src/components/ui/modal/`: overlay + shell (анимации
  появления/закрытия, закрытие по Esc/клику мимо), Header, Footer (Ghost + Primary),
  поля (Input, Select, ColorPicker, PhotoUpload) — в премиум-стиле дизайн-системы
  (CLAUDE.md §6: 2-колоночная сетка, качественные анимации).
- Модалки Каталога (редактирование филиала, зал создание/редактирование, услуга
  создание/редактирование) пересобрать на ките: 2 колонки, левая панель с
  иллюстрацией/превью — уровень `AddStudioModal`, не хуже.
- `AddStudioModal` остаётся визардом, но забирает общие поля/кнопки из кита.
- Дальше кит раскатывается на остальные страницы (очередь — V2-2).

### 4. Контакты филиала: телефон или email обязателен

**Сейчас:** в `AddStudioModal` телефон и email — оба необязательные; в
`EditStudioModal` тоже. Валидации формата нет ни на фронте, ни на бэке
(в `BranchCreate`/`BranchUpdate` это просто `str`).

**Решение (принято 2026-07-14):**
- При создании и редактировании филиала: **заполнено хотя бы одно из двух** —
  телефон или email. Подсветка полей + сообщение, кнопка «Далее» блокируется.
- Формат: email через `EmailStr` (бэк) + проверка на фронте; телефон уже
  форматируется `PhoneField`.
- **Совпадение с контактами сотрудников/владельца — разрешено** (решение выше):
  никаких проверок уникальности для контактов филиала, админ может указать свой
  номер как номер студии.

### 5. Тосты: один компонент, плавные анимации появления и исчезновения

**Сейчас:** у Каталога свой локальный тост (`cat-toast` в `Catalog.css` + стейт в
`Catalog.tsx`) — **пятая** реализация тостов в проекте (Finances, Settings, Profile,
inline в Staff). Анимация исчезновения сломана: `setToastMsg(null)` убирает текст
одновременно со снятием класса `.show` — тост схлопывается мгновенно, а не
растворяется. Повторный показ не сбрасывает старый `setTimeout` — новый тост может
исчезнуть раньше времени.

**Решение (уточняет V2-1 п.8 — внедряем сразу в Сотрудниках И Каталоге):**
- Один глобальный `Toast` + `ToastProvider`/`useToast` в `front/src/components/ui/`,
  монтируется в `DashboardLayout`. Варианты success / error.
- Анимации: появление и исчезновение полностью плавные (текст живёт до конца
  exit-анимации; таймер сбрасывается при новом показе; очередь, если тостов несколько).
- Каталог и Сотрудники переводятся на него сейчас; `cat-toast`, inline-тост Staff —
  удалить. Finances/Settings/Profile — следующими итерациями (план V2-2 в силе).

### 6. Фото в шапке: студия — фото студии, зал — фото зала

**Сейчас:** `photo_url` филиала загружается при создании и приходит в `BranchDetail`,
но **нигде не показывается** — `.cat-hero-bg` это просто персиковая заливка. У залов
фото нет вообще (ни в модели, ни в API).

**Решение:**
- Выбран филиал → в фоне шапки его фото (затемнение/градиент поверх, чтобы кнопки
  и название читались); нет фото — текущая заливка.
- Выбран зал → фото зала; **нет фото зала → фото студии** (решили: лучше с фото);
  нет ни того ни другого — заливка.
- Плавная смена фона при переключении студия ↔ зал.

### 7. Фото зала: загрузка при создании и редактировании

**Сейчас:** невозможно — у зала нет поля.

**Решение:**
- Бэк: `photo_url` в модель `Hall` (+ миграция), в `HallCreate`/`HallUpdate`/`HallBrief`;
  эндпоинт загрузки — обобщить существующий `/upload-branch-photo` (`media.py`)
  или добавить `/upload-hall-photo` в ту же папку логики.
- Фронт: зона загрузки фото в `HallModal` (создание и редактирование) — тот же
  компонент PhotoUpload из модального кита (превью, заменить, удалить), что в
  `AddStudioModal` шаг 3.

### 8. График работы филиала: по умолчанию при создании + редактирование

**Сейчас — витрина без товара:** таблица `branch_working_hours` существует, сетка
часов рисуется в обзоре филиала, но **строки в таблицу никто никогда не пишет**
(`BranchWorkingHours(` не инстанцируется нигде в бэке — онбординг заполняет только
`StudioWorkingHours` студии). Сетка всегда пустая. Эндпоинта редактирования нет.

**Решение:**
- При создании филиала автоматически заводить график по умолчанию (7 дней;
  скопировать часы студии из онбординга, если есть, иначе пн–пт 09:00–21:00,
  сб–вс выходной). В визарде создания график не спрашиваем — он «необязательный,
  но по умолчанию есть».
- Редактирование: секция «Часы работы» в модалке редактирования филиала
  (7 строк: выходной-тумблер + время с/до — переиспользовать паттерн из
  Settings → HoursTab).
- Бэк: `working_hours` в `BranchUpdate` (upsert по `(branch_id, day_of_week)`);
  сидинг дефолта в `create_branch`; разовая миграция/бэкфилл для уже существующих
  филиалов без графика.

### 9. Кнопки шапки «Исправить»/«Удалить» работают с тем, что выбрано

**Сейчас:** в `StudioSection.tsx` кнопки в hero всегда редактируют/удаляют **филиал**,
даже когда выбран зал (`activeHallId != null`). При этом внизу деталей зала есть свои
кнопки «Редактировать зал»/«Удалить зал» — дублирование и путаница.

**Решение:**
- Выбран зал → кнопки шапки открывают `HallModal` с залом / подтверждение удаления зала.
- Выбрана вкладка «Информация» → работают с филиалом (как сейчас).
- Дублирующие кнопки внизу деталей зала убрать.

### 10. Услуги: язык, дизайн, честные данные

**Сейчас:**
- Модалка услуги — русский хардкод и третья копия shell'а (закрывается п.1 и п.3).
- `SERVICE_CATEGORIES` — русские строки как значения в БД: на en-интерфейсе категории
  остаются русскими (перевод через `defaultValue` не срабатывает для данных).
- **Сетка «Расписание» всегда пустая** — `schedule: []` захардкожен в
  `useCatalogList.ts` (бэкенд слоты не отдаёт), рисуем пустую таблицу-декорацию.
- «Записей/мес» = `bookings_total / 6` — выдумка (магическая шестёрка).

**Решение:**
- Категории → ключи `yoga / pilates / stretching / individual` + перевод в locales;
  миграция существующих русских значений в БД на ключи; в модалке — Select из кита.
- Сетка расписания: строить из **реальных занятий** услуги (занятия ближайшей/текущей
  недели по `service_id` — данные в `lessons` уже есть); если бэку нечего отдать —
  секцию скрывать, пустую декорацию не рисовать.
- «Записей/мес» считать честно от реального периода (первое занятие → сейчас),
  либо убрать метрику.
- Модалка услуги — на модальном ките, полный i18n (п.1, п.3).

### 11. Переключение страниц без «лага» — мгновенно и плавно

**Сейчас:** каждая страница на входе грузит данные с нуля: пустой экран/`null`
(в Каталоге при загрузке левая панель просто пустая — `isLoading ? null : …`),
потом контент впрыгивает. Выглядит как лаг.

**Решение (общее, принято 2026-07-14 — TanStack Query, см. решение №1):**
- Повторный заход на страницу рисуется **мгновенно из кэша**, обновление — тихо в фоне.
- Первый заход — лёгкие скелетоны в стиле дизайн-системы вместо пустоты
  (список: 3–4 строки-плейсхолдера; hero: приглушённый блок).
- Дёшево и приятно: префетч данных раздела по hover на пункт бокового меню.
- Внедряем в Каталоге (+ Сотрудники при миграции на общий тост), дальше — по страницам.

### 12. Ошибки видимы всегда: тост, подтверждения, селекты — единые

**Сейчас:**
- Ошибки **загрузки** списков глотаются: в `useCatalogList.ts` все `catch` →
  `console.error` — пользователь видит просто пустую страницу (при 403/500/сети).
- Ошибки мутаций в Каталоге показываются, но локальным тостом, без деталей валидации.
- Валидации на фронте нет: кривой email спокойно уходит на сервер.
- Удаления (филиал, зал, услуга) — через нативный `window.confirm`.
- Селекты: в модалке услуги нативный `<select>`; в AI и Booking — два разных
  собственных `CustomSelect`.

**Решение:**
- Все ошибки API (401, 403, 404, 409, 422, 500, сеть) → **глобальный error-тост**
  с текстом от сервера или переводом по статусу; тост один на всё приложение (п.5).
  Ошибки загрузки — тоже тост + состояние «не удалось загрузить, повторить».
- Клиентская валидация до отправки: формат email, обязательные поля — подсветка
  поля + сообщение под ним (в Каталоге: филиал, зал, услуга).
- Единый `ConfirmModal` в `front/src/components/ui/` (стиль дизайн-системы) —
  заменить `window.confirm` в Каталоге; дальше по очереди V2-2.
- Единый `Select` в `front/src/components/ui/` — внедрить в модалке услуги;
  `CustomSelect` из AI/Booking мигрируют на него следующими итерациями.
- Внедряем сейчас в **Сотрудниках и Каталоге** (уточнение плана V2 — согласовано).

### Бонус-находки аудита

- **`is_online` зала молча сбрасывается при каждом редактировании:** `HallBrief` не
  отдаёт `is_online`, `HallModal` стартует с `false` и шлёт его в PATCH. Онлайн-зал
  после любого сохранения перестаёт быть онлайн. Фикс: `is_online` в `HallBrief`,
  чекбокс инициализировать из зала.
- `is_online` зала вообще не показывается в UI (ни бейджа в списке, ни в деталях) —
  добавить бейдж «Онлайн» в карточку/таблицу залов.
- `getBranches`/`get_branch` требуют роль `owner` — ок по ТЗ (Каталог только для
  владельца), но Журнал тоже использует залы — проверить, что у него свой доступ.
- Оборудование зала — ввод «через запятую» одной строкой; в модальном ките заменить
  на чипы-теги (добавить/удалить).
