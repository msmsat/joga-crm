# Эпик V3-1 — Данные без F5: TanStack Query

Цель эпика простыми словами: сейчас каждая страница держит данные в локальном стейте и грузит их с нуля при каждом заходе — поэтому переключение страниц «лагает» (пустой экран → впрыгивание контента), а изменения в одном месте не видны в другом без F5 (создал зал — счётчик залов в левом списке старый). Внедряем **TanStack Query**: один кэш на всё приложение, мгновенная отрисовка из кэша с тихим обновлением в фоне, инвалидация ключа после мутации обновляет все места, где данные показаны. В этом эпике мигрирует Каталог; остальные страницы — по мере своих аудитов, но провайдер и правила закладываются здесь.

Источник: аудит 2 в `docs/ROADMAP_V3/ROADMAP.md`, пп. 2 и 11 (обсуждено 2026-07-14, решение «TanStack Query» принято владельцем).

**Важно понимать про старт:**
- Кэша на фронте нет вообще: все хуки — `useState` + `useEffect` + ручной `refetch` (`front/src/pages/dashboard/Catalog/hooks/useCatalogList.ts` — типичный образец).
- Подтверждённый баг рассинхрона: CRUD зала рефетчит только деталь филиала (`useBranchDetail.refetch`), а `hall_count` в левом списке живёт в другом хуке (`useStudioList`) и остаётся старым до F5 или переключения вкладки.
- Ошибки загрузки сейчас глотаются: `catch → console.error` (`useCatalogList.ts:38, 78, 118`) — Query отдаст их наружу, показ возьмёт на себя общий тост (эпик V3-2).
- api-клиент уже нормализует ошибки (`ApiError` в `front/src/api/client.ts`) — Query его просто пробрасывает, ничего переделывать не надо.
- Внешний интерфейс хуков для компонентов сохраняем — `StudioSection`/`ServiceSection` не должны заметить подмену (рефакторинг = реорганизация, поведение UI не меняется, кроме исчезновения «лага»).

**Обозначения сложности:** 🟢 простая · 🟡 средняя · 🔴 сложная, делать внимательно.

## Фаза 0 — Фундамент

### 1. Установка, QueryClientProvider, дефолты · 🟢 · 0:45 · ✅ сделано (проверено при старте V4-2, 2026-07-18)
**Простыми словами:** подключить библиотеку и обернуть приложение провайдером с настройками по умолчанию, которые дают нужное поведение «мгновенно из кэша + тихое обновление в фоне».
**Зачем:** это единственная новая зависимость всего roadmap V3 — согласована владельцем 2026-07-14.
**Где:** `front/package.json`, точка входа приложения (`front/src/main.tsx` / `App.tsx` — там, где сейчас роутер и провайдеры).
**Что конкретно сделать:**
1. `npm i @tanstack/react-query` (только ядро; devtools не ставим — YAGNI).
2. `QueryClientProvider` поверх роутера; один `QueryClient` с дефолтами: `staleTime: 30_000` (полминуты данные считаются свежими — переключения страниц мгновенные), `refetchOnWindowFocus: true` (вернулся во вкладку — данные подтянулись), `retry: 1` (одна повторная попытка, не бесконечные).
3. Убедиться, что `ApiError` из `front/src/api/client.ts` доходит до `error` квери без обёрток.
**Готово, когда:** `npm run build` чистый, приложение работает как раньше (провайдер пока никто не использует).

## Фаза 1 — Каталог на Query

### 2. Хуки Каталога → useQuery · 🟡 · 1:30
**Простыми словами:** три хука в `useCatalogList.ts` (`useStudioList`, `useBranchDetail`, `useServiceList`) переписать с ручного `useState`+`refetch` на `useQuery`. Компоненты (`StudioSection`, `ServiceSection`) продолжают получать те же поля — их не трогаем.
**Зачем:** после этого повторный заход на Каталог рисуется мгновенно из кэша; загрузка «пустой левой панели» (`StudioSection.tsx:92` — `isLoading ? null : …`) уходит в прошлое вместе с задачей 4.
**Где:** `front/src/pages/dashboard/Catalog/hooks/useCatalogList.ts`.
**Что конкретно сделать:**
1. Ключи: `['branches']` (список филиалов), `['branch', branchId]` (деталь с залами и часами), `['services']` (услуги). Ключи — строковые константы в одном месте (`front/src/api/queryKeys.ts`), чтобы инвалидация не разъехалась с опечаткой.
2. `useBranchDetail`: `enabled: branchId != null` (сейчас ручной `if (!branchId) return`).
3. Маппер `toUiService` остаётся как `select`-функция квери — трансформация на лету, как и было.
4. Возвращаемый интерфейс хуков сохранить: `{ studios, isLoading, … }` — компоненты не меняются.
**⚠️ Подводные камни:** `useEffect` в `StudioSection.tsx:22–26` (автовыбор первой студии) завязан на приход данных — проверить, что срабатывает и при мгновенной отдаче из кэша.
**Готово, когда:** Каталог работает как раньше; повторный заход на страницу — без «мигания» пустотой; `console.error`-заглушки из хуков удалены (ошибка уходит в `error` квери).

### 3. Мутации → useMutation + инвалидация · 🟡 · 1:00
**Простыми словами:** каждое создание/правка/удаление после успеха инвалидирует **все** ключи, где эти данные видны — и экран обновляется сам, целиком, без F5.
**Зачем:** лечит подтверждённый баг: создал/удалил зал → `hall_count` в левом списке остаётся старым (`useCatalogList.ts`: `createHall` рефетчит только деталь).
**Где:** `front/src/pages/dashboard/Catalog/hooks/useCatalogList.ts`.
**Что конкретно сделать:**
1. Филиалы: create/update/delete → инвалидация `['branches']` и `['branch', id]`.
2. Залы: create/update/delete → инвалидация `['branch', branchId]` **и** `['branches']` (счётчик залов в списке).
3. Услуги: create/update/delete → инвалидация `['services']`.
4. Интерфейс мутаций для компонентов сохранить (это по-прежнему async-функции, которые кидают `ApiError` — обработка ошибок в компонентах не меняется до эпика V3-2).
**Готово, когда:** создание зала мгновенно обновляет и вкладки залов, и счётчик «N залов» в левом списке; удаление филиала убирает его из списка без F5.

## Фаза 2 — Плавность

### 4. Скелетоны Каталога вместо пустоты · 🟡 · 1:00
**Простыми словами:** при **первом** заходе (кэш пустой) вместо пустой панели и впрыгивающего контента — лёгкие серые плейсхолдеры в стиле дизайн-системы. При повторных заходах скелетоны не появляются вообще — данные уже в кэше.
**Зачем:** п. 11 аудита: «переключаешься на страницу — оно начинает данные грузить, выглядит будто лагает».
**Где:** `front/src/pages/dashboard/Catalog/components/StudioSection.tsx:92` (сейчас `isLoading ? null`), `ServiceSection.tsx` (аналогично), `Catalog.css`.
**Что конкретно сделать:**
1. Скелетон левой панели: 3–4 строки-плейсхолдера (иконка + две полоски) с мягким shimmer.
2. Скелетон правой части: приглушённый hero-блок + три статкарточки.
3. Показывать только когда `isLoading && !data` (первая загрузка); фоновое обновление из кэша скелетон не показывает.
4. Стиль: цвета `rgba(26,26,26,0.04–0.07)`, радиусы 12–16px, без резких контрастов (CLAUDE.md §6).
**Готово, когда:** первый заход — плавные скелетоны → контент через `cat-fade`; повторный — мгновенный контент без промежуточных состояний.

### 5. Префетч раздела по hover в боковом меню · 🟢 · 0:45
**Простыми словами:** пока пользователь ведёт мышь к пункту «Каталог», данные уже летят — к моменту клика страница рисуется с готовыми данными. Дёшево (одна строка на пункт) и даёт wow-эффект «всё мгновенно».
**Где:** боковое меню в `front/src/layouts/DashboardLayout.tsx`.
**Что конкретно сделать:**
1. На `onMouseEnter` пункта меню — `queryClient.prefetchQuery` ключей раздела (для Каталога: `['branches']`, `['services']`).
2. Подключить для разделов, уже живущих на Query (пока только Каталог); карта «пункт меню → ключи» расширяется по мере миграции страниц.
3. Благодаря `staleTime` повторные hover не спамят сервер.
**Готово, когда:** заход в Каталог после наведения на пункт меню рисует контент сразу, без скелетонов (проверить с заторможенной сетью в devtools).
