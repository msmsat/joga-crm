# Эпик V3-2 — UI-кит: Toast, ConfirmModal, Select, модальный каркас

Цель эпика простыми словами: одинаковые вещи должны существовать в одном экземпляре и выглядеть одинаково. Сейчас у Каталога **пятая** в проекте реализация тоста (и её анимация исчезновения сломана), удаления идут через голый `window.confirm`, селекты — нативные с местными стилями, а модалки свёрстаны трижды по-разному. Этот эпик строит общие компоненты в `front/src/components/ui/` и подключает их в Каталоге (и Сотрудниках — по тостам); пересборка самих модалок Каталога на новом каркасе — эпик V3-3.

Источник: аудит 2 в `docs/ROADMAP_V3/ROADMAP.md`, пп. 3, 5, 12 (обсуждено 2026-07-14).

**Важно понимать про старт:**
- **Координация с V2:** общий Toast уже создан эпиком V2-1 (Сотрудники) — здесь только доводка. ConfirmModal и Select создаются **здесь** (эпик V2-2 расформирован; его инвентаризация дублей — чек-лист миграции — в `docs/BACKLOG/EPIC_V2_2_BACKLOG.md`). Дублей быть не должно.
- Сломанный тост Каталога — конкретика: `Catalog.tsx:13–16` чистит `toastMsg` одновременно со снятием класса `.show` — текст исчезает мгновенно, «растворяется» уже пустая плашка; повторный показ не сбрасывает предыдущий `setTimeout` — второй тост гаснет раньше времени. Стили: `Catalog.css:657–680`.
- Эталоны стиля уже есть в проекте: анимации и левая панель — `AddStudioModal.tsx`; плашка тоста — премиум-тост Staff; поля с glow-фокусом — `FocusInput` в `AddStudioModal.tsx:224–247`.
- Каркас модалок — самая ответственная задача (🔴): на нём в V3-3 пересобираются четыре модалки Каталога, а дальше — всё приложение. Спешить нельзя.

**Обозначения сложности:** 🟢 простая · 🟡 средняя · 🔴 сложная, делать внимательно.

## Фаза 1 — Тосты

### 1. Общий Toast: плавные анимации входа И выхода, очередь · 🟡 · 1:30
**Простыми словами:** один `Toast` + `ToastProvider`/`useToast()` на всё приложение, монтируется в `DashboardLayout`. Главное требование владельца — **анимация исчезновения плавная до конца**: тост растворяется с текстом, а не схлопывается пустым.
**Зачем:** п. 5 аудита («пусть все тосты будут везде одинаковы… анимация исчезновения до конца плавная»); в проекте уже 5 реализаций.
**Где:** новый `front/src/components/ui/Toast.tsx` (+ `ToastProvider`), монтирование в `front/src/layouts/DashboardLayout.tsx`. Если Toast уже создан эпиком V2-1 — здесь только доводка анимаций и очереди по этой спеке.
**Что конкретно сделать:**
1. `useToast()` → `{ success(msg), error(msg) }`; варианты: success (плашка `#1A1A1A`, как у Staff) и error (акцент `#D88C9A`).
2. Жизненный цикл: enter (подъём + fade, ~0.3s, пружинящий cubic-bezier) → пауза ~3s → exit (fade + лёгкий уход вниз, ~0.25s). **Сообщение удаляется из DOM только по завершении exit-анимации** — текст живёт до конца.
3. Новый показ при видимом тосте: сбросить таймер, обновить контент (или очередь, если сообщений несколько подряд — не терять ни одного).
4. Тёмная тема сразу (перекраска через переменные, не хардкод).
**Готово, когда:** быстрые повторные тосты не «моргают» и не гаснут раньше времени; исчезновение плавное с текстом; компонент один, живёт в `components/ui`.

### 2. Каталог → общий Toast; ошибки загрузки перестают глотаться · 🟡 · 1:00
**Простыми словами:** выкинуть локальный тост Каталога и пробросить в общий; заодно ошибки **загрузки** списков (сейчас `console.error` — пользователь видит просто пустую страницу) начинают всплывать error-тостом с возможностью повторить.
**Зачем:** п. 12 аудита: «ошибки пусть выскакивают все… 403 или ещё какая-то — пусть выпадает тост».
**Где:** `front/src/pages/dashboard/Catalog/Catalog.tsx:11–16, 48` (локальный тост — удалить), `Catalog.css:657–680` (стили — удалить), `StudioSection.tsx` / `ServiceSection.tsx` (проп `showToast` заменить на `useToast`), ошибки загрузки — из `error` квери (после V3-1) или из `catch` хуков.
**Что конкретно сделать:**
1. Убрать `showToast`-проп из секций; успехи → `toast.success(…)`, ошибки мутаций → `toast.error(detail из ApiError или перевод по статусу)`.
2. Ошибка загрузки списка (филиалы, деталь, услуги) → error-тост + в панели состояние «Не удалось загрузить — Повторить» (кнопка дёргает refetch).
3. Известные статусы без осмысленного `detail` (403, 404, 500, сеть) — переводить через `t('common:errors.<код>')` (ru + en).
**Готово, когда:** выключенный бэкенд превращает заход в Каталог в понятный тост + кнопку «Повторить», а не в тишину; `cat-toast` не существует.

**Простыми словами:** одна модалка «Точно удалить X? — Отмена / Удалить» на всё приложение, в стиле дизайн-системы; в Каталоге сейчас три нативных `window.confirm` (браузерное серое окно — против «premium minimalism»).
**Зачем:** п. 12 аудита («модалки удалить/подтвердить — везде одинаково»); эталон — `Staff/components/modals/DeleteConfirmModal.tsx` (у него danger-стиль зашит — добавить и нейтральный вариант, см. чек-лист в `docs/BACKLOG/EPIC_V2_2_BACKLOG.md`).
**Где:** новый `front/src/components/ui/ConfirmModal.tsx`; замены: `StudioSection.tsx:51` (удаление филиала), `StudioSection.tsx:63` (удаление зала), `ServiceSection.tsx:40` (удаление услуги).
**Что конкретно сделать:**
1. Пропсы: `title`, `message`, `confirmText`, `danger` (красная кнопка `#D88C9A`), `onConfirm`/`onClose`; кнопки по умолчанию из `common.json` (`buttons.cancel`/`buttons.delete`).
2. Анимации: blur-подложка + scaleUp, закрытие по Esc и клику мимо — те же, что у каркаса задачи 5 (переиспользовать overlay).
3. Заменить все три `window.confirm` Каталога; тексты подтверждений уже в `catalog.json` (`confirmDelete`, `confirmDeleteHall`, `confirmDelete` услуг) — переиспользовать.
4. Кнопка «Удалить» показывает спиннер/disabled, пока летит запрос (сейчас confirm закрывается мгновенно, а запрос может упасть).
**Готово, когда:** ни одного `window.confirm` в Каталоге; удаление филиала/зала/услуги — через общую модалку с danger-стилем.
