# EPIC R9 — Подсказки: человеческий язык и попапы, которые всегда видно

**Цель:** владелец студии (не аналитик) понимает каждую цифру на экране.
Технические формулы вида `Σ Operation.amount, type='in', по op_date периода`
заменяются на объяснение бизнес-языком; подсказки появляются у всех блоков, где
их нет; поповер больше никогда не уезжает за край экрана и не пропадает под
соседней карточкой.

**Аудит:** пункты 4, 5, 18.
**Зависимости:** нет. **Оценка: ~5:30.**

## Frontend Architecture

### Задача 1. Новые тексты подсказок

**Файлы:** `front/src/locales/ru/reports.json` и `en/reports.json`, блок `formulas`.

Правила копирайтинга (обязательны для всех новых ключей):
- никаких имён таблиц, полей, `Σ`, `count`, `distinct`, `status='attended'`;
- 1–2 предложения, второе — зачем эта цифра владельцу;
- обращение на «вы», без канцелярита;
- если цифра приблизительная — сказать об этом прямо.

`title` остаётся прежним (это заголовок метрики), меняется `text`:

| Ключ | Новый `text` (ru) |
|---|---|
| `revenue` | Все деньги, поступившие в студию за выбранный период — по дате оплаты. Абонементы, разовые занятия, сертификаты и товары вместе. |
| `profit` | Сколько осталось у студии после расходов периода: аренды, зарплат, закупок. Это и есть ваш реальный заработок. |

| `attendance` | Сколько визитов клиенты действительно совершили. Считаются только занятия, где посещение отмечено. |
| `bookings` | Сколько записей на занятия сделали клиенты. Отменённые записи не учитываются. |
| `noshow` | Клиент записался, занятие прошло, а он не пришёл. Каждая такая запись — потерянное место, которое мог занять кто-то другой. |
| `activeClients` | Сколько разных людей пришли к вам хотя бы раз за период. Один клиент считается один раз, сколько бы визитов у него ни было. |
| `occupancy` | Насколько полно заполнены ваши занятия: сколько мест занято из тех, что вы поставили в расписание. 100 % — свободных мест не осталось. |
| `avgCheck` | Сколько в среднем приносит одна оплата. Растёт, когда клиенты берут абонементы подлиннее или покупают дополнительно. |
| `retention` | Сколько клиентов прошлого периода вернулись в этом. Показывает, удерживаете вы людей или каждый месяц набираете новых с нуля. |
| `newClients` | Люди, которые впервые появились в базе студии за выбранный период. |
| `returning` | Клиенты, которые ходили к вам и раньше, и снова пришли в этом периоде. |
| `lost` | Ходили в прошлом периоде и ни разу не пришли в этом. Их ещё реально вернуть — начните с этого списка. |
| `clientValue` / `avgValue` | Сколько в среднем оставляет у вас один клиент за период. |
| `repeatPurchaseShare` | Доля клиентов, которые купили у вас больше одного раза. Чем выше — тем крепче привычка ходить именно к вам. |
| `renewalRate` | Сколько клиентов купили новый абонемент после того, как закончился прежний. |
| `revenuePerHour` | Сколько денег приносит один час работы тренера. Помогает сравнивать тренеров честно: у кого-то занятий меньше, но каждое дороже. |
| `trainerReturnRate` | Доля клиентов тренера, которые записались к нему повторно. |
| `missedRevenue` | Деньги, которых студия не получила из-за пустых мест на прошедших занятиях: свободные места, умноженные на цену занятия. |
| `lessonRevenue` | Примерная выручка занятия: сколько человек пришло, умноженное на цену. Оплата не привязана к конкретному занятию, поэтому это близкая оценка, а не точная сумма. |
| `filtersScope` | Фильтры «Филиал» и «Зал» действуют только на показатели по занятиям. У платежей привязки к залу нет, поэтому деньги показываются по всей студии. |
| `cancelNoshow` | Доля сорванных занятий: отменённые занятия и клиенты, которые не пришли, — от всего запланированного. |
| `avgRating` | Средняя оценка, которую клиенты поставили занятиям этого тренера. |
| `teamScope` | Здесь учитывается только выручка, привязанная к конкретному тренеру. Поэтому сумма по всем тренерам может быть меньше общей выручки студии. |
| `lessonsCount` | Сколько занятий провели тренеры за выбранный период. |

### Задача 2. Поповеры, привязанные к viewport

**`front/src/components/ui/InfoHint.tsx`** — переписать позиционирование:
поповер рендерится в `createPortal(document.body)` с `position: fixed` и
координатами, посчитанными от `rect` кнопки.

```tsx
// Логика (общая для InfoHint и Tooltip) — front/src/components/ui/popoverPosition.ts
export function placePopover(
  anchor: DOMRect, size: { w: number; h: number },
  side: Side, gap = 10, pad = 8,
): { top: number; left: number; side: Side } {
  // 1. Не влезает по выбранной стороне — переворачиваем (top ⇄ bottom, left ⇄ right).
  // 2. По второй оси кламп: left = clamp(ideal, pad, innerWidth - size.w - pad).
  // 3. Возвращаем реальную сторону — стрелка рисуется по ней, а не по запрошенной.
}
```

Реализация в компоненте:
1. `useLayoutEffect` при `open`: измерить `buttonRef.getBoundingClientRect()` и
   `popoverRef.getBoundingClientRect()`, посчитать координаты, положить в стейт;
2. поповер — `position: fixed; top; left; zIndex: 1200`, ширина
   `min(260px, calc(100vw - 16px))`;
3. стрелка смещается к якорю: `left = anchorCenterX - popoverLeft - 7`, тоже с
   клампом `[14, width - 14]` — иначе она уезжает за скруглённый угол;
4. пересчёт на `scroll` и `resize` (passive-слушатели, только пока открыт);
   при скролле дальше 40px от места открытия — просто закрываем, это дешевле и
   не «пляшет».

Портал заодно решает проблему контекста наложения от `Card`'s `transform`
(поповер больше не может оказаться под соседней карточкой).

**`front/src/components/ui/Tooltip.tsx`** — тот же `placePopover`, тот же
портал. Дополнительно `pointerEvents: 'none'` уже стоит — оставить.

Оба компонента остаются с прежним API (`side` — это *предпочтительная*
сторона), так что 30+ мест использования по проекту править не нужно.

## Edge Cases

| Ситуация | Поведение |
|---|---|
| Кнопка у правого края экрана | Поповер прижимается к краю с отступом 8px, стрелка съезжает к якорю |
| Кнопка в самом низу страницы | `side` переворачивается на `top`; если не влезает и туда — выбирается сторона с бо́льшим запасом |
| Экран уже 300px | Ширина `calc(100vw - 16px)`, текст переносится, поповер не режется |
| Скролл страницы при открытом поповере | Пересчёт координат; сдвиг > 40px — поповер закрывается |
| Поповер внутри карточки с `hover`-подъёмом | Портал: карточка может двигаться, поповер стоит на месте и остаётся сверху |
| Открыт второй `InfoHint` | Первый закрывается по `mousedown` мимо (логика уже есть) |
| Ключ формулы отсутствует в i18n | i18next вернёт ключ строкой — недопустимо: в PR обязателен прогон по всем `formulaKey` из кода против обоих json |
| Язык `en` | Проверка «в файле нет `Σ`, `count(`, `status=`» — часть DoD |

## Definition of Done

- Ни в ru, ни в en `reports.json` не осталось SQL-лексики; каждая подсказка
  читается человеком без опыта работы с БД.
- Поповер виден целиком при клике по любому `InfoHint` страницы Отчётов, в том
  числе у крайнего правого KPI и в правом нижнем углу.
- У каждого информационного блока вкладок Клиенты, Команда, Продажи есть
  подсказка; ни одна не переиспользует чужой `formulaKey` «за неимением своего».
- `npm run build && npm run lint` зелёные.
