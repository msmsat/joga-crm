# Velora CRM — Roadmap REPORTS: переработка раздела «Отчёты»

Архитектура «Actionable Insights»: 5 вкладок (Обзор · Продажи · Клиенты ·
Команда · Расписание), у каждого графика — заголовок, короткая расшифровка,
InfoHint с формулой, клик в детализацию, блок «Что делать» с готовыми
действиями.

**Действуют общие решения V3 (на всё приложение):** TanStack Query как кэш
данных (без F5) и единый UI-кит (`front/src/components/ui/`).

## Точка отсчёта (честно, на 2026-07-22)

- **Фронт — витрина на моках.** `useReportData.ts` умножает базовые числа на
  коэффициент периода (`day: 0.037, week: 0.19…`) — все цифры выдуманы.
  6 вкладок русскими строками в типах (`Tab = 'Основные' | …`), ноль i18n,
  валюта `₽` хардкодом, данных из Query нет.
- **Бэк — фундамент есть.** `back/routers/analytics/reports.py`:
  `/analytics/summary` (KPI + тренды к прошлому периоду), `/series`
  (revenue/expenses/bookings/new_clients по day/week/month), `/trainers`,
  `/services`, `/activity` — реальные агрегации, owner-only, скоуп
  `ctx.studio_id`. Их использует Дашборд — **не ломать**.
- **Данные для всех метрик уже в БД:** `Operation` (type in/out, category,
  method, client_id, trainer_id, product_id), `Lesson` (teacher_id, hall_id,
  total_spots, price, duration_min, status cancelled), `Reservation`
  (active/cancelled/attended, **rating**, booking_channel, cancelled_at),
  `Client` (registration_date, last_visit_date), `ClientSubscription`
  (package_id, created_at, expires_at, used/total) — ничего нового в схеме БД
  не нужно, только агрегации.
- Готовое к переиспользованию: `/loyalty/retention` (продления, V5-4),
  SQL-сегменты `loyalty/segments.py` (риск/VIP/истекающие), кампании по
  сегментам (V5-4), `useExport` (клиентский CSV), `useStudioCurrency()` +
  `getCurrencySymbol()`.

## Общие решения роадмапа

1. **Один саммари-эндпоинт на вкладку** (KPI + срезы + insights одним
   ответом, один Query-ключ) + отдельные `/series`-эндпоинты только там, где
   есть переключатель метрик или свой grouping. Меньше водопадов запросов.
2. **Фильтры и сравнение — на бэке, едино для всех вкладок.** Общий Depends
   `ReportFilters`: `date_from`, `date_to` (обязательные) + `branch_id`,
   `hall_id`, `trainer_id`, `service_id` (опциональные). Прошлый период той
   же длины бэк считает всегда и отдаёт `*_pct` (паттерн `_pct`/`prev` из
   `reports.py` — обобщается, не переписывается).
3. **«Что делать» = детерминированные SQL-правила на бэке, не AI.** Ответ —
   `{key, severity, params, action, action_params}`; текст собирает фронт из
   i18n (`reports.json → insights.<key>` с интерполяцией `params`); кнопка
   действия — навигация с предзаполненными фильтрами (`useInsightAction`).
   Правило без данных → не отдаётся (никаких пустых сигналов-заглушек).
4. **Словарь метрик** (ниже) — единственный источник формул: он же текст
   InfoHint (ключи `formulas.<metric>` в i18n), он же спека SQL.
5. **Экспорт = клиентский CSV текущего среза** через существующий
   `useExport` — реальные данные из Query-кэша, реальный файл. Каждая вкладка
   определяет свой `toCsvRows()`. PDF/серверный экспорт — в BACKLOG.
6. **Реактивность:** ключи `['reports', <tab>, <paramsHash>]` в
   `queryKeys.ts`, `staleTime` 30 с, refetch on window focus. Страница
   read-only (мутаций нет) — этого достаточно для no-F5.
7. **Деньги — рубли целыми** (решение V5 №3), символ — из
   `useStudioCurrency()`. Общий хелпер `fmtMoney` уезжает в
   `front/src/lib/format.ts` (сейчас ДВЕ копии `formatMoney` — в Overview и
   Clients; третью не плодим, старые копии мигрируют по мере касания).
8. **Heatmap — CSS grid**, без новой библиотеки. Графики — Recharts (стоит).
9. **Вкладки — ключами** (`'overview' | 'sales' | 'clients' | 'team' |
   'schedule'`), подписи через i18n. Вкладка «События» удаляется (лента
   событий живёт на Дашборде); «По услугам» вливается в Продажи.
10. **Клик в детализацию — переиспользуем существующие list-эндпоинты**
    (операции — `/finances/operations` с date-фильтром, занятия — journal
    API, клиенты — segment-эндпоинты). Новый drilldown-эндпоинт заводится
    только там, где существующего среза нет (слот heatmap, клиенты недели).
11. Старые эндпоинты `/analytics/summary|series|activity` остаются
    (Дашборд); `/analytics/trainers|/services` после R2/R4 больше никем не
    используются — удаляются в финальной зачистке R5.

## Словарь метрик (формулы InfoHint)

| Метрика | Формула |
|---|---|
| Выручка | Σ `Operation.amount`, type='in', по `op_date` периода |
| Прибыль | Выручка − Σ расходов (type='out') |
| Посещения | count `Reservation` status='attended' (join Lesson по периоду) |
| Записи | count `Reservation` status≠'cancelled' |
| No-show | `Reservation` status='active' И занятие уже закончилось (`start_time + duration_min < now`) |
| Активные клиенты | distinct `client_id` с посещением за период |
| Заполняемость | занятые места (active+attended) / Σ `total_spots` по неотменённым занятиям периода |
| Средний чек | Выручка / число доходных операций |
| Retention | доля клиентов прошлого периода, посетивших в текущем (`_retention` из reports.py) |
| Новые клиенты | `registration_date` в периоде |
| Вернувшиеся | посещение в периоде И посещали до периода |
| Потерянные | посещали в прошлом периоде, нет посещений в текущем |
| Ценность клиента | Выручка с `client_id` / distinct таких клиентов |
| Доля повторных покупок | клиенты с ≥2 доходными операциями / клиенты с ≥1 |
| % продлений | из `/loyalty/retention`: купил следующий абонемент после окончания |
| Выручка на час (тренер) | Σ in-операций с `trainer_id` / Σ часов его занятий |
| Возвращаемость (тренер) | клиенты с ≥2 посещениями у тренера / клиенты с ≥1 |
| Упущенная выручка | Σ (свободные места × `Lesson.price`) по прошедшим неотменённым занятиям |

## Порядок и зависимости

**R0 строго первым** — все эпики строятся на его тулбаре, `ReportFilters`,
общих компонентах и i18n-неймспейсе. Дальше R1 → R2 → R3 → R4 → R5
(по убыванию ценности для владельца; технически R1–R5 независимы друг от
друга и могут идти в любом порядке).

Переходный период: R0 сразу ставит новые 5 вкладок; ещё не переделанные
вкладки временно показывают старые компоненты (sales ← TabProdazhi,
team ← TabTrenery, overview ← TabOsnovnye, clients/schedule — честный
empty-state «в разработке»). Старые компоненты удаляются в эпике своей
вкладки, финальная ревизия — R5.

Внешние зависимости: % продлений (R2) и сегменты (R3) переиспользуют
`/loyalty/retention` и логику `loyalty/segments.py` — уже в проде (V5-4).
«Запустить кампанию» ведёт в готовые кампании Лояльности, свой механизм
рассылки НЕ строим.

## Definition of Done (каждый эпик)

- `npm run build` и `npm run lint` зелёные.
- Ни одного мокового числа: каждая цифра на экране — из ответа API.
- Все строки через `t('reports:…')` (ru + en), деньги через `fmtMoney` с
  символом валюты студии.
- Данные через TanStack Query: смена периода/фильтра/фокус окна обновляют
  всё без F5.
- Кнопка экспорта скачивает CSV с теми данными, что на экране.
- У каждого графика и KPI: заголовок, расшифровка, InfoHint с формулой из
  словаря, рабочий клик в детализацию.
