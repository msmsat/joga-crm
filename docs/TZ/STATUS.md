# Velora CRM — Текущий статус реализации

Этот документ отвечает на вопрос «что уже сделано на самом деле» — в отличие от `docs/TZ/FUNCTIONAL.md`, который описывает, что продукт должен уметь в итоге. Точка отсчёта после выполнения ROADMAP_V1 (эпики 1–5): фундамент безопасности, Журнал, Финансы/Дашборд, каналы роста и биллинг Fondy — реализованы и работают с базой. Обновлено: 2026-07-24.

Общая картина одной строкой: **из 14 страниц CRM реально работают с сервером 13; на моках остались только Настройки (кроме страницы Уведомлений)** — Velora AI теперь тоже реальный (см. ниже, не хватает только самой LLM-модели) — это территория эпиков 6–7 (`docs/BACKLOG/`) плюс несколько хвостов V1, перечисленных в конце.

## Работает по-настоящему (сохраняется в базе)

### Авторизация и роли
Регистрация с подтверждением email (письмо реально уходит через SMTP; без SMTP-кредов в `.env` код печатается в консоль — dev-режим), вход по паролю, восстановление пароля, онбординг, Google-вход. Входить могут все три роли (владелец, администратор, тренер) — владелец выдаёт сотруднику временный пароль при создании.

**Главная дыра прошлой версии закрыта:** сервер проверяет роль и принадлежность к студии на каждом разделе (`require_role` + `get_studio_context` во всех рабочих роутерах). Каталог, Сотрудники, Финансы, Аналитика, Лояльность, Онлайн-запись (настройки), Уведомления, Биллинг — только владелец; Клиенты — владелец и администратор; Журнал — тренер видит и меняет только свои занятия. Чужая студия везде отвечает «не найдено».

### Каркас (меню, верхняя панель, пейволл)
Скрытие пунктов меню по роли; бейдж числа клиентов в меню — реальный. Глобальный гейт подписки: без активной подписки сервер отвечает 402, фронт пускает только на «Тариф и оплата» и Профиль; баннер «подписка заканчивается / обновится» за 3 дня. AI-строка в шапке и панель AI-чата — та же реальная сессия, что и на странице Velora AI (см. ниже), без потери контекста при переключении между поверхностями.

### Каталог
Полный CRUD с сервера: филиалы (создание, правка, удаление), залы внутри филиала (создание, правка, удаление, онлайн-залы), услуги (список, создание, правка, удаление).

### Клиенты
Список пагинированный («Загрузить ещё», поиск с дебаунсом, категории со счётчиками), карточка со всеми действиями: теги, заметки, статусы, заморозка/разморозка. Кнопки карточки больше не имитация: «Записать» реально записывает на занятие, «Бонус» начисляет баллы лояльности, «Позвонить»/«Сообщение» открывают tel:/wa.me. Продажа абонемента из карточки создаёт доходную операцию и абонемент с остатком занятий. Лимиты тарифа проверяются при создании клиента.

### Сотрудники
Список, профиль, создание (с выдачей доступа и временным паролем), правка, удаление, расписание, отмена занятия. «Добавить событие» реально создаёт занятие сотруднику; «Написать»/«Позвонить» — mailto:/tel:. Лимиты тарифа проверяются при создании.

### Журнал
Полностью живой: занятия грузятся только за видимый диапазон (день/неделя/месяц), тренеры — из Сотрудников, залы — из Каталога. Создание, перенос перетаскиванием, растягивание, отмена — прямое сохранение на сервер (черновик остался чисто клиентским режимом). Запись клиента с проверкой мест и запретом дублей, отмена записи, отметка посещения (обновляет дату визита клиента и списывает занятие с абонемента). Сводка дня — из реальных занятий. После F5 всё на месте.

### Финансы (7 вкладок из 9)
Счета (балансы, изменение за день на лету), Операции (пагинированный список с фильтрами; создание двигает баланс счёта в одной транзакции), Контрагенты (баланс и сделки на лету), Документы, Цели (авто-прогресс из операций / ручной режим), Зарплаты (расчёт из занятий тренера, выплата создаёт расходную операцию). Остались на моках: «Онлайн-платежи», «Способы оплаты» (по плану — после появления клиентских оплат) и вкладка «Отчёты» внутри Финансов (не путать с отдельным разделом «Отчёты» — тот на реальных данных, см. ниже).

### Дашборд (Обзор)
Всё с реальных агрегатов: 4 метрики с трендами к прошлому периоду (`/analytics/summary`), график (`/analytics/series`), лента событий студии (наполняется из операций, клиентов, публичных броней), задачи на сегодня (CRUD), сводки по тренерам и услугам. Для админа/тренера owner-эндпоинты корректно обрабатывают 403.

### Отчёты (5 вкладок)
Полностью на реальных данных: Основные (`/analytics/overview` — выручка/прибыль/посещения/активные клиенты/заполняемость, структура выручки, динамика клиентов), По продажам (`/analytics/sales` — категории, способы оплаты, типы покупателей, топ-продукты), По клиентам (`/analytics/clients-report` — новые/вернувшиеся/потерянные, сегменты риска и лояльные из тех же матчеров, что и Лояльность), По тренерам (`/analytics/team` — нагрузка, выручка/час, возвращаемость, рейтинг, детальный дровер тренера), По расписанию (`/analytics/utilization` — тепловая карта день×час с drilldown в занятия слота, самые прибыльные/заполненные занятия, хроническая низкая загрузка, простой залов по вечерам). Единые фильтры периода/филиала/зала/тренера/услуги, инсайты с переходом в нужный раздел, экспорт CSV. Дашборд по-прежнему держит собственные `/analytics/trainers` и `/analytics/services` для виджетов «Топ услуги»/«Нагрузка тренеров» — это не дублирование Отчётов, а отдельный лёгкий срез для главной страницы.

### Velora AI
Три поверхности (страница, дровер, AI-строка в шапке) делят одну и ту же сессию: чаты, история, сообщения — реальные, из базы (`/ai/sessions`, `/ai/messages`), без потери контекста при F5 или переключении между поверхностями. Первый заход — скелетон вместо мигания пустоты; обрыв сети — тост «Не удалось загрузить» + кнопка «Повторить». Настройки ассистента (язык ответов) и агентов (Telegram/Instagram: тон, макс. длина ответа 50–4000, офчасы, системный промпт до 2000 символов) сохраняются на сервер с клиентской валидацией диапазонов. Telegram-бот подключается реальным токеном (формат проверяется на клиенте, сам токен — через API BotFather), Instagram — через OAuth (подключение/отключение, автообновление токена). Интерфейс страницы, дровера и AI-строки — полностью на `i18next` (ru/en), даты и «N мин назад» — через `Intl` по языку интерфейса.

**Не хватает только самой модели:** `generate_reply()` без `LLM_BASE_URL` в `.env` отвечает честной локализованной заглушкой («Velora AI 3.5 подключается в ближайшем обновлении…»); с заданным `LLM_BASE_URL` уходит на любой OpenAI-совместимый сервер (Ollama и т.п.) без изменений в роутере. В селекте моделей заведена одна `velora-3.5` — остальные не подключены.

### Лояльность
Конфиги всех 5 программ сохраняются на сервер (get-or-create дефолтов). Уровни, карты клиентов, сводка — из базы. Баллы: ручной бонус из карточки и автоначисление от доходных операций по курсу программы (не пускает баланс в минус). Сертификаты: выпуск с уникальным кодом, погашение (повторное — 409). Пакеты абонементов: CRUD + продажа клиенту, списание занятий при посещении, триггеры «осталось 1–2 занятия».

### Уведомления
Три живых канала: **Telegram** (бот студии, подключение токеном от BotFather), **Email** (адрес отправителя → код на почту → верификация), **WhatsApp** (Embedded Signup, предупреждение о платности сообщения). Instagram/SMS/Push — bool-колонки в БД остаются, но UI и `notify()` их игнорируют (см. ROADMAP_NOTIFICATIONS/ROADMAP.md, архитектурное решение). Диспетчер `notify(db, studio_id, role, event_id, context)` — единая точка отправки, шаблоны локализуются по `Studio.language`, суммы — по `Studio.currency`. Матрица «роль × событие × канал» боевая: галочки реально управляют отправкой, не витрина.

Живые триггеры (см. `back/services/notifier.py` — 29 шаблонов): запись клиента (c1), отмена (c3/a2), перенос занятия (c11), оплата ≥ порога (c4/a4/o3), абонемент «осталось 1–2 занятия» (c5), абонемент истекает (c6), день рождения клиента раз в день (c7/t8, `back/services/daily_notify.py`), выплата зарплаты тренеру (t6, `POST /finances/salaries/{id}/pay`), отчёт за день/неделю владельцу и админу (a8/o1, тот же daily-цикл), лояльность/сертификаты/рефералы (c8/c9/o5..o8), публичная бронь (a1/a3/a6/a10).

**В BACKLOG** (галочки в матрице сохраняются, шаблона у `notify()` нет — честно показано как «пока не отправляется», без имитации): a7 «Конфликт расписания», a9 «Попытка входа», o9 «Экспорт данных», t1/t2/t5/t7 (трансляция клиентских событий записи/отмены/переноса/отзыва на сторону тренера) — детали см. `docs/BACKLOG/README.md`.

Фронт: React Query + Zustand (только UI-состояние — активная роль, черновик матрицы), инвалидация ключей вместо F5, `refetchOnWindowFocus`. Ошибки — только `useToast`/`ErrorBoundary`, ноль локальных error-стейтов на странице.

### Онлайн-запись
Правила записи, брендинг и каналы сохраняются на сервер (Telegram-токен — в конфиг канала). Публичный API без авторизации работает: каталог услуг, свободные слоты (уважают правила записи), бронирование по имени и телефону (find-or-create клиента, `booking_channel=web`, письмо-подтверждение, событие в ленте). Rate-limit: 5 броней/мин, 30 чтений/мин с IP. UI мини-приложения нет — только API под него.

### Тариф и оплата (биллинг Fondy)
Серверный каталог тарифов — источник цен (фронт-моки снесены); free_trial на 14 дней после онбординга (до онбординга — план `none`); checkout через редирект на Fondy; вебхук с проверкой подписи и идемпотентностью по order_id; возвраты с откатом подписки; продление в один клик по сохранённой карте (rectoken); лимиты тарифа (Старт 3 сотрудника/100 клиентов, Pro 15/1000, Business безлимит) с модалкой апгрейда в UI. История счетов и карта •••• — с сервера; форма ручного ввода карты удалена (PCI). Валюта — из настроек студии (RUB исключён; UAH/EUR/USD/GBP/PLN/CZK).

**Проверено автотестами** (`back/tests/`): весь чек-лист вебхука (approved/declined/reversed, повторная доставка — no-op, неверная подпись — 403, лимиты) — `test_billing_webhook.py`; начисление баллов — `test_loyalty_points.py`. Плюс self-check'и `python -m services.fondy` и `python -m routers.billing.webhook`.

**НЕ проверено:** ручной прогон на тестовом мерчанте Fondy (клик по реальной checkout-странице, приём боевых колбэков через туннель, сквозной путь «оплатил → CRM открылась»). Боевые ключи — при деплое (эпик 7). До этого биллинг нельзя считать проверенным end-to-end.

### Профиль
Реальные данные вошедшего пользователя (имя, email, телефон), сохранение правок, смена пароля. «Связанные аккаунты» и «Сессии» — по-прежнему мок (план: эпик 6).

## Осталось на моках

### Настройки
Все вкладки, кроме личных настроек, — вёрстка: серверные обработчики общих настроек, команды, безопасности (сессии, ключи), интеграций (WhatsApp/Telegram/Google Calendar/1С), данных (бэкапы, экспорт) — заглушки `router = APIRouter()`. Логотип показывает превью без загрузки. Территория эпика 6.

### Клиентское мини-приложение
UI не существует (вторая очередь после MVP CRM), но публичный API записи под него уже работает.

## Хвосты ROADMAP_V1 — кандидаты в новый roadmap

1. **Ручной прогон биллинга на тестовом мерчанте Fondy** (эпик 5, задача 13, ручная часть) + боевые ключи при деплое.
2. **«Тренер видит только своих клиентов»** — не реализовано; сейчас тренеру раздел Клиенты закрыт целиком (403). Связь клиент→тренер через занятия уже есть в данных.
3. **Google-вход для сотрудников** — Google-логин по-прежнему ищет только владельца; существующий админ/тренер через Google не войдёт (по email/паролю — входит).
4. **Вкладки Финансов «Онлайн-платежи» и «Способы оплаты»** — ждут контура клиентских оплат (post-MVP по плану эпика 3).
5. **Автосписание подписки по расписанию** — нет планировщика; продление кнопкой/при заходе. Post-MVP.
6. **Журнал не читает query-параметры** — переходы из инсайтов Отчётов (`open_journal`/`add_lesson`: `lesson_id`, `teacher_id`, `hall_id`, `weekday`/`hour` и т.д.) корректно формируют URL, но страница Журнала их игнорирует — попадание всегда на текущий день без подсветки/префилла. Кнопки «Перенести»/«Отменить» в срезе «Хроническая низкая загрузка» (Отчёты → Расписание) страдают тем же.
7. Из «вне рамок» V1: мультистудийность (переключение студий), реальная отправка в Telegram/WhatsApp/SMS/Push, серверные черновики Журнала.

Впереди (бэклоги — `docs/BACKLOG/`): **эпик 6** (подключение настоящей LLM-модели к Velora AI + добивка Настроек, тема, i18n остальных страниц) и **эпик 7** (деплой, домен, SSL, юридика для верификации Fondy, бэкапы, мониторинг — без него продавать нельзя).

## Стек и общее техническое состояние

Бэкенд: FastAPI, SQLAlchemy (asyncpg/PostgreSQL), Alembic, Pydantic, JWT; slowapi (rate-limit), aiosmtplib (почта), aiohttp (Fondy). Фронтенд: React 19, TypeScript, Vite, React Router.

Все серверные разделы, кроме четырёх вкладок `/settings`, — рабочие: auth, studio (каталог+услуги+медиа), clients (+бонусы, +абонементы), staff, schedule, finances (6 роутеров), analytics, loyalty (4 роутера), booking (настройки+публичный), billing (каталог, checkout, вебхук, возвраты), settings/notifications, ai (сессии, сообщения, настройки, агенты Telegram/Instagram — сама генерация ответа за `LLM_BASE_URL`, см. Velora AI выше). Списки клиентов, сотрудников и операций — пагинированы (контракт `Page`: items/total/offset/limit). Сборка фронтенда `npm run build` проходит чисто (проверено 2026-07-24).
