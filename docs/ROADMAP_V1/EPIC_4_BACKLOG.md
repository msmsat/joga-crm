- **Схемы — заглушки:** `back/schemas/loyalty/*`, `back/schemas/settings/booking.py`; схем уведомлений нет вообще.

- **UI на 100% моках:** `pages/dashboard/Booking/` (каналы захардкожены, тумблеры uncontrolled, превью StudioMockup недостижимо), `Loyalty/` (5 программ, дровер-формы ничего не сохраняют), `Notifications/` (матрица в локальном стейте, события с id `c1/t1/...` в constants.ts). - **Отправки нет никакой:** email-код регистрации — `print()` в `back/routers/auth/register.py`; aiogram числится в requirements, но не используется; SMTP-кода нет.
- **Паттерн публичного эндпоинта** (без oauth2_scheme) — `back/routers/auth/register.py`; готовая логика брони с лимитом мест и запретом дублей — `book_lesson` в роутере клиентов.
- **Из Эпика 1:** `get_studio_context` и `require_role`. **Из Эпика 2:** занятия (без них публичной записи не на что записывать). **Из Эпика 3:** `POST /finances/operations` — точка триггера начисления баллов.

## Жёсткие правила эпика (MVP)

1. **Единственный реальный канал доставки — Email (SMTP).** У клиентов в базе уже есть email, а `tg_id` пуст (Telegram-бота нет — клиенту неоткуда было написать боту). Один модуль `back/services/mailer.py` — им же заменяем `print()` в регистрации. Telegram/WhatsApp/Instagram/SMS/Push — только сохранение конфигурации без отправки.
2. **Публичная запись — только API** (`/public/booking/*`, без JWT, studio_id в пути): каталог занятий → бронь по имени и телефону (find-or-create Client, `booking_channel="web"`). UI мини-приложения — post-MVP (раздел 17 ТЗ).
3. **Триггеры лояльности и уведомлений** — обычные вызовы хелперов из существующих эндпоинтов. Никаких очередей и фоновых воркеров; отправка письма — await с try/except: ошибка отправки логируется и не валит запрос.
4. **Диспетчер уведомлений** — одна функция `notify(studio_id, role, event_id, context)`: читает матрицу; включён email — шлёт; остальные каналы молча пропускает. Канонические event_id — строки из фронтовых `NOTIF_EVENTS` (`c1`, `t1`, ...).
5. **Доступ:** все `/loyalty/*`, `/booking/*` (настройки), `/settings/notifications/*` — `require_role("owner")`; `/public/booking/*` — без авторизации.
6. **Контракт-фиксы в задаче схем:** `LoyaltyConfig.expiry_period` — бэкенд диктует строку (фронт-тип `number|null` поправить); короткие ключи NotificationSettings (`telegram`, `email`, ...) ↔ колонки `*_notifications` — маппинг alias-ами в Pydantic-схеме.

**Обозначения сложности:** 🟢 простая · 🟡 средняя · 🔴 сложная, делать внимательно.

### 0a. Завести SMTP-аккаунт и получить креды · 🟢 · 0:30
**Простыми словами:** чтобы письма (код регистрации, подтверждение записи) реально уходили, нужен почтовый аккаунт с включённым SMTP-доступом.
**Зачем:** задача 7 пишет код отправки, но без реальных кредов он останется в dev-режиме (print в консоль).
**Что конкретно сделать:**
1. Выбрать вариант: обычный ящик с паролем приложения (быстрый старт) или транзакционный сервис (лучше доставляемость, есть бесплатные тарифы).
2. Включить SMTP, получить host/port/user/password.
3. Положить в `back/.env`: `SMTP_HOST`, `SMTP_PORT`, `SMTP_USER`, `SMTP_PASS`, `SMTP_FROM` (не коммитить!).
**⚠️ Подводные камни:** личный ящик для продакшена быстро упрётся в лимиты и спам-фильтры — для запуска хватит, но SPF/DKIM на своём домене (Эпик 7) обязателен до продаж.
**Готово, когда:** креды лежат в .env; тестовое письмо через них уходит (проверится задачей 7).


**Простыми словами:** замкнуть цикл абонемента: владелец продаёт клиенту пакет занятий → у клиента абонемент с остатком → отметка посещения списывает занятие → на остатке 1–2 клиент получает уведомление.
**Зачем:** без этого пакеты из задачи 5 — просто витрина: продать нельзя, остаток не уменьшается, а события «осталось 1–2 занятия» и «абонемент истекает» из матрицы уведомлений никогда не сработают. Карточка клиента уже показывает абонемент и заморозку — но купить его негде.
**Где:** `back/routers/clients/*` (продажа из карточки), `back/routers/schedule/reservations.py` (attend — Эпик 2, задача 18), `back/services/notifier.py`; фронт — кнопка «Продать абонемент» в карточке клиента.
**Контракт:**
```

Ответ 201: абонемент клиента (тип, остаток, срок) + создана доходная операция категории «Абонементы»
```
**Что конкретно сделать:**
1. Продажа: создать/обновить абонемент клиента по пакету (остаток = занятий в пакете, срок = длительность пакета); создать доходную операцию (Эпик 3) — она же триггерит баллы (задача 4).

3. Триггеры: остаток стал 1 или 2 → notify «осталось N занятий»; остаток 0 или срок истёк → notify «абонемент закончился».
**⚠️ Подводные камни:** замороженный абонемент не списывается и не истекает (заморозка уже работает — не сломать); attend идемпотентен — повторная отметка не должна списать дважды.
**Готово, когда:** продал пакет из карточки — появились операция и абонемент с остатком; отметка посещения уменьшила остаток; на остатке 2 пришло письмо (при включённой галочке).

## Порядок работы (что за чем)

Фаза 0 — в любой момент до задач 7 и 9. Дальше строго по фазам **1 → 2 → 3 → 4**; внутри фаз линейно.
- Связки между фазами: задача 8 (диспетчер) нужна задаче 11 (публичная бронь шлёт подтверждение); задача 4 нужна задаче 16 (кнопка «Бонус»); задача 5b требует attend из Эпика 2 (задача 18) и операции Эпика 3, а её уведомления шлются через задачу 8.
- Задачи 10–11 требуют работающего Эпика 2 (занятия); задача 4 — Эпика 3 (операции).

## Что специально НЕ входит в Эпик 4

- **Реальная отправка в Telegram/WhatsApp/Instagram/SMS/Push** — только сохранение конфигурации; живой канал один — Email.
- **UI клиентского мини-приложения/виджета** — post-MVP (раздел 17 ТЗ); публичный API готовится под него уже сейчас; StudioMockup остаётся мок-превью.
- **Модель лога уведомлений** — её нет в БД; проверка сквозного потока — по факту получения письма; лог — post-MVP.
- **Скидки и сертификаты в кассовом потоке** (применение скидки к операции, оплата сертификатом) — конфиги сохраняются; интеграция с оплатами — Эпик 5.
