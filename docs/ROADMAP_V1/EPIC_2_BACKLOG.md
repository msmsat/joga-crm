# Эпик 2 — Журнал (расписание занятий)

> Оживление Журнала: превратить красивый макет с моками в реально работающий центр записи на занятия.
> Источник: [`docs/ROADMAP.md`](ROADMAP.md), Эпик 2.

## Что уже есть в коде (точка старта)

- **Роутеры `back/routers/schedule/*` пустые** (заглушки), схемы `back/schemas/schedule/*` — пустые заготовки. НО расписание НЕ с нуля: в `back/routers/staff/schedule.py` уже есть **рабочая референс-реализация** — чтение занятий по `teacher_id`+`studio_id` (`get_today_schedule`/`get_month_schedule`), вычисление `booked_count` и `cancel_lesson` (отмена занятия + каскад резерваций + teacher-скоуп). Задачи 3, 6, 9 переиспользуют этот код, а не пишут заново.
- **Модели готовы** (`back/models/schedule.py`): `Lesson` (есть `teacher_id` → это даёт ролевой скоуп, `start_time`, `duration_min`, `total_spots`, `status`) и `Reservation`. **Миграции не нужны.**
- **Контракт фронта уже задан:** `front/src/api/schedule/schedule.api.ts` описывает точные адреса эндпоинтов — бэкенд пишем ровно под него.
- **Переиспользуем:** рабочий `POST /clients/{id}/booking` из Эпика 1 (проверка мест + запрет дублей) — та же логика в записи на занятие.
- **Из Эпика 1:** `get_studio_context` и `require_role` в `back/dependencies.py`.
- **Главная сложность фронта:** сетка Журнала (`front/src/pages/dashboard/Journal/`) работает на своём мок-типе `Booking` (индексы времени 0–14; тренер как индекс; зал как строка-название), а бэкенд — на `Lesson` (дата-время; id). Нужен переходник Lesson↔Booking. Тренеры и залы захардкожены в `constants.ts`.

## Три жёстких правила эпика

1. **Никаких серверных черновиков.** `isDraftMode` остаётся локальным на клиенте. Каждое действие (перенос / растяжка / создание / удаление) → сразу запрос → сохранение в БД.
2. **Грузим только видимый диапазон дат** (день или неделя), а не всю базу занятий.
3. **Ролевой скоуп:** тренер меняет только свои занятия (по `teacher_id`); владелец и админ — любые.

## Фаза 2 — Бэкенд: чтение

### 3. Список занятий за диапазон дат · 🔴 · 1:30
**Простыми словами:** отдавать занятия только за видимые дни (день или неделя), не всю базу.
**Зачем:** скорость — грузим окно дат, а не тысячи занятий.
**Где:** `back/routers/schedule/lessons.py`.
**Контракт:**
```
GET /schedule/lessons?date_from=2026-07-13&date_to=2026-07-19&hall_id=<опц.>
Ответ 200: [LessonRead { id, name, teacher_id, teacher_name, hall_id,
            start_time, duration_min, total_spots, booked_count, status, price, level }]
Ошибки: 401 без токена · 422 нет date_from/date_to
```
**Что конкретно сделать:**
1. Фильтр по `studio_id` через `get_studio_context`; фильтр `start_time` между date_from и date_to; опциональный `hall_id`.
2. При роли `teacher` принудительно фильтровать по `teacher_id` текущего пользователя — тренер видит только свои занятия (ТЗ 2.3).
3. В ответ класть `booked_count` (одним запросом с GROUP BY, не N+1).
**⚠️ Подводные камни:** `booked_count` через отдельный запрос на каждое занятие = N+1 и тормоза; считать одним JOIN/подзапросом. Референс чтения — `get_month_schedule` в `staff/schedule.py`.
**Готово, когда:** GET за неделю возвращает занятия только этих 7 дней; у каждого верное число записанных; тренер получает только свои занятия.

## Фаза 3 — Бэкенд: запись

### 6. Правило «тренер меняет только своё» · 🟡 · 1:00
**Простыми словами:** общая проверка: тренер трогает только свои занятия; владелец и админ — любые.
**Зачем:** ролевой скоуп; иначе тренер поменяет чужое расписание.
**Где:** `back/routers/schedule/lessons.py`, `back/dependencies.py`.
**Что конкретно сделать:** хелпер: берёт занятие, сверяет `teacher_id` с текущим юзером; для owner/admin пропускает всё; иначе 403.
**⚠️ Подводные камни:** образец проверки `teacher_id` уже есть в `staff/schedule.py` (`cancel_lesson`) — переиспользовать, не писать заново.
**Готово, когда:** тренер получает 403 при правке чужого занятия; своё — ок; владелец правит любое.

### 7. Создать занятие · 🟡 · 1:00
**Простыми словами:** сохранять новое занятие в базу — кнопка «Создать занятие» должна реально работать.
**Где:** `back/routers/schedule/lessons.py`.
**Контракт:**
```
POST /schedule/lessons
Тело: { name, teacher_id, hall_id, start_time, duration_min, total_spots, price, level, equipment }
Ответ 201: LessonRead (booked_count = 0)
Ошибки: 400 конец раньше начала · 400 total_spots вне 1–50 · 404 чужой зал/тренер · 403 тренер создаёт не себе
```
**Что конкретно сделать:**
1. Проставить `studio_id` из контекста; проверить, что зал и тренер принадлежат студии.
2. Заполнить `teacher_name` по `teacher_id` (обязательное денормализованное поле).
3. Валидации: лимит группы 1–50; конец позже начала.
**⚠️ Подводные камни:** забыть `teacher_name` — карточка занятия в сетке останется без имени тренера.
**Готово, когда:** созданное занятие видно после перезагрузки страницы; имя тренера на карточке верное.

### 8. Изменить / перенести / растянуть занятие · 🔴 · 1:30
**Простыми словами:** одно действие сохраняет перенос по времени, смену зала и тренера, новую длительность — прямое сохранение drag-n-drop.
**Где:** `back/routers/schedule/lessons.py`.
**Контракт:**
```
PATCH /schedule/lessons/{id}
Тело (все поля опциональны): { start_time, duration_min, hall_id, teacher_id, name, total_spots, price }
Ответ 200: LessonRead
Ошибки: 403 тренер правит чужое · 404 чужая студия · 400 total_spots < уже записанных
```
**Что конкретно сделать:**
1. Менять только присланные поля; при смене `teacher_id` обновлять `teacher_name`.
2. Те же валидации, что в создании + скоуп тренера (задача 6).
3. Запретить `total_spots` меньше текущего `booked_count`.
**⚠️ Подводные камни:** уменьшение мест ниже числа записанных молча «переполнит» занятие — проверять явно.
**Готово, когда:** перенос карточки сохраняется; после F5 занятие на новом месте; после смены тренера на карточке его имя.

- Всего **18 задач**. Строго линейно по фазам **1 → 2 → 3 → 4**. Внутри фазы 3 задача 6 (скоуп) идёт до 7–9; задача 18 (отметка посещения) — после 10–11. Задачи 16–17 (фронт-долги из Эпика 1) делаются после 12–15, когда контур Журнала уже работает.
- **Не с нуля:** `back/routers/staff/schedule.py` уже содержит рабочий образец чтения занятий (teacher-скоуп, `booked_count`) и `cancel_lesson` (отмена + каскад резерваций). Задачи 3, 6, 9 переиспользуют его.
- **Про черновик:** задачи 13–14 сохраняют КАЖДОЕ действие сразу (прямое сохранение). Кнопка «Опубликовать» / `isDraftMode` остаётся чисто визуальной на клиенте — серверную публикацию батчем в MVP НЕ делаем.
- **Тренер-скоуп клиентов из Эпика 1** («тренер видит только своих клиентов») можно закрыть попутно после задачи 18: теперь есть связь `reservations → lesson → teacher_id`. Если время позволит — сделать сразу; иначе отдельным мелким тикетом.
