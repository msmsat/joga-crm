# Эпик 1 — Фундамент (безопасность + добивка Каталога и Профиля)

Цель эпика простыми словами: сделать так, чтобы **сервер сам проверял, кому что можно** (сейчас он почти не проверяет — это дыра), навести порядок в том, **как приходят списки** (чтобы приложение не тормозило на больших данных), и **дописать то, что на экране есть, а на сервере — нет** (редактирование филиалов, залы, услуги, страница Профиля).

**Важно понимать про старт:**
- Базу данных менять не нужно — все нужные таблицы уже есть.
- Часть работы уже сделана: сервер уже умеет отдавать данные вошедшего пользователя (проблема Профиля — только на фронтенде, там вшиты фейковые имя и почта).
- Главная дыра: почти нигде на сервере не проверяется роль (владелец / админ / тренер).
- Про «Написать/Позвонить»: на сервере уже есть **заглушки-эндпоинты** для сотрудников (`back/routers/staff/actions.py`) и клиентов (log_call/send_message в `clients`) — они проверяют данные и отвечают «ок», но **ничего реально не отправляют**. Реальная отправка сообщений — это Эпик 4 (нужны каналы связи). Для «Позвонить» серверная логика и не нужна: звонок — это `tel:`-ссылка на фронте, эндпоинт лишь отдаёт номер.

**Обозначения сложности:** 🟢 простая · 🟡 средняя · 🔴 сложная, делать внимательно.

## Фаза 0 — Подготовка

### 1. Проверить, что в базе всё на месте · 🟢 · 0:45
**Простыми словами:** прежде чем писать код, убедиться, что в базе данных уже есть все нужные поля и таблицы, чтобы по ходу не выяснилось, что чего-то не хватает.
**Зачем:** сэкономить время — если чего-то нет, лучше узнать это сейчас, а не на середине эпика.
**Где смотреть:** `back/models/*.py` (только читаем, ничего не меняем).
**Что конкретно сделать:**

3. Проверить, что у зала есть поле «онлайн-зал» (`is_online`) и есть таблица услуг (`Service`).
**Готово, когда:** записал вывод «всё на месте, миграции не нужны». Если чего-то нет — завёл отдельную мелкую задачу на добавление поля.

## Фаза 1 — Фундамент бэкенда (это блокирует всё остальное)

### 2. Единый формат списков — обёртка `Page` · 🟡 · 1:00
**Простыми словами:** сейчас, когда сервер отдаёт список (например клиентов), он присылает просто «пачку» записей и не говорит, сколько их всего в базе. Из-за этого фронт не может сделать кнопку «Загрузить ещё». Делаем стандартную «коробку» вокруг любого списка, в которой лежат и сами записи, и число «а всего их столько-то».
**Зачем:** без этого приложение либо тянет ВСЕ данные разом (тормоза на 1000+ записей), либо не знает, есть ли ещё.
**Где:** `back/schemas/common.py`.
**Что конкретно сделать:**
1. Создать переиспользуемую модель `Page` с полями: `items` (сами записи), `total` (сколько всего), `offset` (сколько пропустили), `limit` (сколько взяли за раз).
2. Существующий в этом же файле неиспользуемый `PaginatedResponse` (page/per_page) удалить или заменить новой моделью — в проекте должен остаться ровно один контракт пагинации.
**Готово, когда:** эту «коробку» можно подключить к любому списку, и в ответе сервера появляются `items` и `total`; старый `PaginatedResponse` нигде не используется.

### 3. Общая функция «какая студия у пользователя» · 🔴 · 2:00
**Простыми словами:** сейчас в разных файлах сервера ОДИН И ТОТ ЖЕ кусок кода повторяется — «достань из токена, к какой студии относится вошедший человек». Копипаста = источник ошибок. Пишем это один раз в общем месте, заодно достаём его роль (владелец/админ/тренер).
**Зачем:** это основа всей проверки прав. От неё зависят почти все следующие задачи.
**Где:** `back/dependencies.py`.
**Что конкретно сделать:**
1. Написать функцию `get_studio_context`, которая возвращает: кто вошёл, его студию и его роль.
2. ⚠️ **Важная ловушка:** сейчас токен содержит студию и роль только если у человека ОДНА студия. Если несколько — в токене их нет. Функция обязана это учесть (иначе такие пользователи получат ошибку «нет доступа» вообще везде).
**Готово, когда:** любой файл сервера может одной строкой получить студию и роль вошедшего.

### 4. Охранник по ролям — `require_role` · 🔴 · 1:30
**Простыми словами:** «охранник на входе». Вешаешь его на действие или раздел — и сервер сам проверяет: у человека нужная роль? Нет — отвечает «403, нет доступа» и не пускает. Сейчас такой охраны нет вообще: тренер, зная адрес запроса, теоретически может достать чужие данные напрямую.
**Зачем:** это и есть закрытие главной дыры в безопасности.
**Где:** `back/dependencies.py`.
**Что конкретно сделать:**
1. Написать `require_role("owner")` / `require_role("owner", "admin")` — которую можно повесить на любой эндпоинт.
2. Если роль не подходит — сервер отдаёт ошибку 403.
**Готово, когда:** на тестовом действии тренер получает «403 нет доступа», а владелец проходит.

### 4b. Вход для админа и тренера — доступ сотрудника · 🟡 · 1:30
**Простыми словами:** сейчас войти в CRM может ТОЛЬКО владелец — вход остальных ролей заблокирован на сервере (см. docs/TZ/STATUS.md). Все проверки ролей этого эпика невозможно даже протестировать, пока админ и тренер физически не могут войти. Открываем вход и даём владельцу способ выдать сотруднику доступ.
**Зачем:** без этого продукт «для команды студии» работает только для владельца, а чек-лист Verification этого эпика (прогон под тремя ролями) неисполним.
**Где:** `back/routers/auth/login.py` (owner-only фильтр), `back/routers/staff/*` (создание сотрудника), фронт — форма создания сотрудника.
**Что конкретно сделать:**
1. Убрать в логине ограничение «вход только владельцу»: пускать любого участника студии (`studio_members`), роль — в контекст токена.
2. При создании сотрудника владелец задаёт временный пароль (простейший MVP-способ; красивое приглашение по email — улучшение Эпика 4, когда появится mailer).
3. Смена пароля сотрудником — через уже существующий flow смены пароля.
**⚠️ Подводные камни:** онбординг 5 шагов — только для владельца новой студии; сотрудник после входа должен попадать сразу в CRM своей студии, а не в онбординг.
**Готово, когда:** созданный владельцем тренер входит со своим email/паролем, видит только свои разделы; чек-лист Verification эпика реально прогоняется под тремя ролями.

### 5. Переключить старый код на новую функцию · 🟡 · 2:00
**Простыми словами:** скучная, но обязательная уборка. Пройтись по уже написанным файлам и заменить старый повторяющийся код на новую общую функцию из задачи 3. Ничего нового не появляется — просто наводим порядок.
**Зачем:** чтобы дальше не тащить копипасту и не чинить одну и ту же ошибку в пяти местах.
**Где (все 5 файлов, где сейчас есть эта копипаста):** `back/routers/clients/profiles.py`, `back/routers/staff/profiles.py`, `back/routers/staff/schedule.py`, `back/routers/staff/actions.py`, `back/routers/studio/router.py`.
**Что конкретно сделать:**
1. Удалить из этих файлов повторяющийся кусок про «достань студию из токена».
2. Подключить вместо него общую функцию из задачи 3.
**Готово, когда:** помощник `_studio_id_from_token` удалён из всех 5 файлов, а всё, что работало раньше, работает так же.
**⚠️ Осторожно:** в файле клиентов ~20 таких мест. Если займёт больше 2ч — разбить на подзадачи по файлам.

## Фаза 6 — Фронтенд (оживляем экраны)

### 17. Правильно реагировать на «нет доступа» (403) · 🟡 · 1:00
**Простыми словами:** после закрытия разделов по ролям сервер начнёт иногда отвечать «403, нет доступа». Сейчас фронт этого не ожидает и может сломаться или выкинуть из аккаунта. Учим его показывать понятное сообщение.
**Где:** `front/src/api/client.ts`.
**Готово, когда:** при ответе «403» появляется сообщение «Нет доступа», а страница не ломается и человека не разлогинивает.

### 18. Дописать вызовы сервера для Каталога · 🟢 · 0:45
**Простыми словами:** на фронте есть «переводчик» между кнопками и сервером (api-файл). Дописываем в него новые вызовы: изменить филиал, удалить филиал, создать/изменить/удалить зал.
**Где:** `front/src/api/studio/studio.api.ts`, `front/src/api/studio/studio.types.ts`.
**⚠️ Осторожно (частая причина «белого экрана»):** описания данных на фронте (TypeScript) должны точно совпадать с тем, что реально шлёт сервер. Если поле на сервере может быть пустым (`null`) — на фронте оно тоже должно быть помечено как необязательное. Иначе экран падает.
**Готово, когда:** новые вызовы обращаются к правильным адресам сервера, типы совпадают с ответами.

### 19. Оживить формы филиала и зала · 🟡 · 1:30

**Где:** `front/src/pages/dashboard/Catalog/components/modals/EditStudio.tsx`, `Catalog/components/StudioSection.tsx`, `Catalog/hooks/useCatalogList.ts`.
**Готово, когда:** редактирование/удаление филиала и зала реально меняет данные и остаётся после перезагрузки страницы.
**ℹ️ Опционально (решить по ТЗ):** редактирование часов работы филиала (`BranchWorkingHours`) в задачах 11–12 пока не заложено — форма создания филиала (`BranchCreate`) их не содержит. Если форма филиала в MVP должна редактировать часы работы — добавить их в схемы `BranchCreate/BranchUpdate` и в задачи 11 (бэк) и 19 (фронт). Иначе — явно отложить в v2.

## Порядок работы (что за чем)

Сначала — фундамент, он блокирует всё: **1 → 2 → 3 → 4 → 4b → 5**.
После этого работа идёт двумя параллельными дорожками:
- **Бэкенд:** 6, 7, 8 → 9, 10 → 11–16
- **Фронтенд:** 17 → 18, 19, 20 → 21, 22

Дорожки синхронизируются по «формам данных» (какие поля шлёт сервер) — поэтому бэкенд по каждому разделу лучше делать чуть раньше, чем фронт его подключает.

## Что специально НЕ входит в Эпик 1

- **Тренер видит только своих клиентов** — переезжает в Эпик 2: для этого нужна связь через занятия, а занятий пока нет.
- **Проверка ролей в Журнале / Финансах / Настройках / Отчётах** — там на сервере пока вообще нет действий, охранять нечего. Охранник (задача 4) будет готов и повесится, когда эти разделы начнут писать (Эпики 2–3).
- **Переключение между несколькими студиями** — отложено за пределы MVP. Разделение данных по студиям при этом работает уже сейчас, но UI-переключателя не делаем.
- **«Связанные аккаунты» и переключение аккаунтов в Профиле** (ТЗ 2.15) — остаются моком до v2; реальные сессии устройства появятся в Эпике 6 (Безопасность).

## Как проверить (Verification)

Финальный чек-лист «кто чего НЕ может» — прогнать под тремя аккаунтами (владелец / админ / тренер) через Swagger (`http://localhost:8000/docs`):

- **Каталог:** админ и тренер получают 403 на список филиалов/залов/услуг и на любые правки; владелец — полный доступ; чужой филиал/зал/услуга — «не найдено».
- **Клиенты:** тренер получает 403 на список и карточку; админ и владелец видят только клиентов своей студии.
- **Сотрудники:** админ и тренер получают 403 на список и профили; владелец — полный доступ.
- **Списки:** `/clients` и `/staff` отдают `items` + `total`; порция и `total` сходятся с числом записей в базе.
- **Профиль:** у двух разных аккаунтов — разные данные; «Сохранить» реально меняет имя и телефон в базе.
- **Фронт:** `cd front && npm run dev` — при 403 показывается «Нет доступа» без падения страницы и разлогина; `npm run build && npm run lint` — чисто.
