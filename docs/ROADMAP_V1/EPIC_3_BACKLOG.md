## Что уже есть в коде (точка старта)

Деньги хранятся как Integer.

- **Экспорт уже работает механически:** `Reports/hooks/useExport.ts` — CSV через Blob + BOM, PDF = `window.print()`. Нужны только реальные данные.
- **Эталон для копирования паттернов** — `back/routers/clients/profiles.py`.
- **Из Эпика 1:** обёртка `Page` (`back/schemas/common.py`), `get_studio_context` и `require_role` (`back/dependencies.py`). **Из Эпика 2:** занятия и записи (нужны для зарплат, посещаемости, «записей на сегодня»). Если они не готовы — сначала закрыть их.

## Жёсткие правила эпика (MVP)

1. **Никаких OLAP и снапшотов.** Все агрегаты — живые SQLAlchemy-запросы (SUM/COUNT/GROUP BY/date_trunc) по `operations`, `lessons`, `reservations`, `clients`. `DailyMetricSnapshot` не трогаем — это оптимизация после MVP.
2. **Операции — единственный источник денежных фактов.** Выручка и расходы везде считаются из таблицы `operations`; зарплатная выплата сама создаёт расходную операцию.
3. **Дашборд не имеет своих эндпоинтов** — питается из `/analytics/summary`, `/series`, `/trainers`, `/services`, `/activity`, `/tasks`. Ноль дублирования логики.
4. **Список операций — строго `Page[T]`** (контракт Эпика 1: items/total/offset/limit) с серверными фильтрами.
5. **Производные поля не храним:** `Account.daily_change`, `Counterparty.balance/deals_count`, `FinancialGoal.current_amount` (auto-режим) считаются на лету из операций.
6. **Доступ:** всё `/finances/*` — `require_role("owner")`; `/analytics/*` — все роли (Дашборд по ТЗ доступен всем), страница Отчётов закрыта owner-ом на фронте.

**Обозначения сложности:** 🟢 простая · 🟡 средняя · 🔴 сложная, делать внимательно.

## Фаза 1 — Базовые CRUD Финансов

### 1. Схемы Финансов и Аналитики · 🟡 · 1:30
**Простыми словами:** описать, в каком виде счета, операции, контрагенты, цели, зарплаты, задачи и события ходят между сервером и фронтом.
**Зачем:** контракт данных — фундамент всех задач эпика; фронт-типы уже существуют и диктуют поля.
**Где:** `back/schemas/finances/accounts.py`, `operations.py`, `salary.py`, новый `goals.py`; `back/schemas/analytics/reports.py`, `tasks.py`; `back/schemas/finances/__init__.py` и `analytics/__init__.py`.
**Что конкретно сделать:** заполнить заготовки классами `AccountRead/Create/Update`, `OperationRead/Create`, `CounterpartyRead/Create/Update`, `FinDocumentRead/Create`, `GoalRead/Create/Update`, `SalaryRead`, `StudioTaskRead/Create/Update`, `ActivityLogRead` — наследовать `BaseSchema` из `back/schemas/_base.py`.
**⚠️ Подводные камни:** поля строго 1:1 с `front/src/api/finances/finances.types.ts` и `analytics/analytics.types.ts`; суммы — Integer (копейки), как в моделях.
**Готово, когда:** все схемы импортируются; поля совпадают с фронт-типами.

### 2. Счета: список, создание, правка · 🟡 · 1:00
**Простыми словами:** сервер отдаёт счета студии (касса, расчётный, эквайринг) и позволяет добавить свой счёт.
**Зачем:** вкладка «Счета и кассы» сейчас рисует выдуманные балансы.
**Где:** `back/routers/finances/accounts.py`.
**Контракт:**
```
GET  /finances/accounts               → 200: [AccountRead { id, name, type, balance, daily_change }]
POST /finances/accounts   { name, type, balance? }        → 201: AccountRead
PATCH /finances/accounts/{id}  { name?, type? }           → 200: AccountRead
DELETE /finances/accounts/{id}                            → 204
Все: require_role(owner); чужой studio_id → 404
```
**⚠️ Подводные камни:** `daily_change` НЕ хранить — считать на лету как сумму операций счёта за сегодня.
**Готово, когда:** через Swagger счёт создаётся; список отдаёт балансы; `daily_change` меняется после добавления операции за сегодня.

### 3. Операции: пагинированный список с фильтрами · 🔴 · 1:30
**Простыми словами:** отдавать доходы и расходы порциями с фильтрами по типу, категории, счёту, датам и поиском.
**Зачем:** главная таблица финансов; на тысячах операций без пагинации фронт умрёт.
**Где:** `back/routers/finances/operations.py`.
**Контракт:**
```
GET /finances/operations?type=&category=&account_id=&client_id=&date_from=&date_to=&search=&offset=0&limit=20
Ответ 200: Page[OperationRead] { items, total, offset, limit }   — сортировка op_date DESC
```
**⚠️ Подводные камни:** `total` считать тем же набором фильтров, что и `items`, иначе кнопка «Загрузить ещё» будет врать.
**Готово, когда:** ответ содержит items и total; фильтр по типу и датам сужает выборку; порция 20 штук при total = реальному числу в базе.

### 4. Операции: создание и баланс счёта · 🟡 · 1:00
**Простыми словами:** создание дохода/расхода сразу двигает баланс привязанного счёта.
**Зачем:** единый источник денежных фактов; на нём стоит вся аналитика эпика.
**Где:** `back/routers/finances/operations.py`.
**Контракт:**
```
POST /finances/operations
Тело: { title, type: in|out, category, amount, account_id, op_date, client_id?, counterparty_id?, payment_method? }
Ответ 201: OperationRead
DELETE /finances/operations/{id} → 204 (с откатом баланса)
Ошибки: 404 чужой account_id/client_id/counterparty_id
```
**Что конкретно сделать:** проставить `studio_id`; проверить принадлежность связанных сущностей студии; при `type=in` прибавить amount к `Account.balance`, при `type=out` вычесть; DELETE — откатить.
**⚠️ Подводные камни:** обновление баланса и создание операции — в одной транзакции, иначе при падении посередине баланс разъедется с историей.
**Готово, когда:** создал доход 1000 — баланс счёта вырос на 1000; удалил — вернулся; операция с чужим account_id даёт 404.

### 5. Контрагенты и Документы: CRUD · 🟡 · 1:30
**Простыми словами:** сервер хранит юрлиц/ИП/физлиц и документы (акты, счета, договоры) со статусами.
**Зачем:** две вкладки Финансов сейчас полностью нарисованные.
**Где:** новые `back/routers/finances/counterparties.py` и `documents.py`; подключить в `back/routers/finances/router.py`.
**Контракт:**
```
GET/POST/PATCH/DELETE /finances/counterparties[/{id}]
  CounterpartyRead { id, name, type: legal|ip|person, category, requisites, balance, deals_count }
GET/POST/PATCH/DELETE /finances/documents[/{id}]
  FinDocumentRead { id, title, doc_type, status: draft|pending|signed, counterparty_id, created_at }
```
**⚠️ Подводные камни:** `balance` и `deals_count` контрагента считать на лету из операций GROUP BY counterparty_id (правило 5). Файл документа в MVP опционален — паттерн загрузки есть в `back/routers/studio/media.py`.
**Готово, когда:** контрагент создаётся; его deals_count растёт после привязанной операции; документ меняет статус и переживает перезапуск.

### 6. Цели: CRUD с авто-прогрессом · 🟡 · 1:00
**Простыми словами:** финансовые цели студии; прогресс либо вводится руками, либо сам считается из операций.
**Зачем:** вкладка «Цели» по ТЗ имеет авто/ручной режим (`tracking_mode` уже в модели).
**Где:** новый `back/routers/finances/goals.py`, `back/schemas/finances/goals.py`; подключить в `finances/router.py`.
**Контракт:**
```
GET/POST/PATCH/DELETE /finances/goals[/{id}]
  GoalRead { id, title, target_amount, current_amount, tracking_mode: auto|manual, category, deadline }
```
**Что конкретно сделать:** при `tracking_mode=auto` `current_amount` на лету = SUM доходных операций категории цели от created_at до deadline; при `manual` — брать из PATCH.
**Готово, когда:** авто-цель категории «Абонементы» растёт после дохода этой категории; ручная цель меняется только через PATCH.

### 7. Зарплаты: расчёт и выплата · 🔴 · 2:00
**Простыми словами:** посчитать зарплату тренера из его занятий и ставки; выплата помечает расчёт и создаёт расходную операцию.
**Зачем:** вкладка «Зарплаты» плюс замыкание денег: выплата сама попадает в расходы и отчёты.
**Где:** `back/routers/finances/salary.py`.
**Контракт:**
```
GET /finances/salaries?period_start=&period_end=
Ответ 200: [SalaryRead { user_id, name, sessions_count, rate, rate_type, amount, status }]
POST /finances/salaries/{user_id}/pay  { period_start, period_end }
Ответ 201: SalaryPayment + создана расходная Operation категории «Зарплата»
Ошибки: 409 повторная выплата за тот же период
```
**Что конкретно сделать:** по каждому тренеру `sessions_count` = COUNT занятий (`Lesson` по `teacher_id` за период; status != cancelled), `amount` по `User.rate`/`rate_type`; при выплате — `SalaryPayment` со снапшотом ставки + расходная операция.
**⚠️ Подводные камни:** ставка в момент выплаты снапшотится в `SalaryPayment` — если завтра ставку поменяют, старые выплаты не должны «пересчитаться».
**Готово, когда:** расчёт совпадает с числом занятий тренера в Журнале; после выплаты появились SalaryPayment со status=paid и расходная операция на ту же сумму.

## Фаза 2 — Агрегация Аналитики

### 8. Сводка периода с трендами · 🔴 · 2:30
**Простыми словами:** один эндпоинт отдаёт выручку, расходы, прибыль, средний чек, активных клиентов, записи и удержание за период плюс сравнение с прошлым периодом.
**Зачем:** сердце эпика — питает 4 метрики Дашборда, Отчёты и вкладку «Отчёты» Финансов.
**Где:** `back/routers/analytics/reports.py`.
**Контракт:**
```
GET /analytics/summary?date_from=&date_to=
Ответ 200: {
  revenue, expenses, profit, avg_check,
  active_clients, bookings, retention, attendance,
  trends: { revenue_pct, expenses_pct, active_clients_pct, bookings_pct, retention_pct }
}
```
**Что конкретно сделать:**
1. SUM операций in/out; `avg_check` = выручка / число доходных операций; `bookings` = COUNT активных Reservation за период; `active_clients` = COUNT клиентов с визитом.
2. `retention` = доля клиентов прошлого периода, вернувшихся в текущем; посещаемость — по `Reservation.status=attended` (отметка посещения — Эпик 2, задача 18).
3. Блок `trends` — те же цифры за предыдущий период той же длины и % изменения.
**⚠️ Подводные камни:** только SQLAlchemy SUM/COUNT/GROUP BY — никаких снапшотов; деление на ноль в трендах (прошлый период пустой) — отдавать null, не 500.
**Готово, когда:** Swagger отдаёт сводку за месяц; цифры сходятся с ручным подсчётом по таблицам; trends показывает знак изменения.

### 9. Ряды для графиков · 🟡 · 1:00
**Простыми словами:** отдать точки график-ряда (выручка/записи/клиенты по дням-неделям-месяцам).
**Зачем:** график Дашборда, графики Отчётов и линия доход/расход Финансов — один эндпоинт.
**Где:** `back/routers/analytics/reports.py`.
**Контракт:**
```
GET /analytics/series?metric=revenue|expenses|bookings|new_clients&group=day|week|month&date_from=&date_to=
Ответ 200: [{ period: "2026-07-01", value: 15000 }]
```
**Что конкретно сделать:** `func.date_trunc` + GROUP BY.
**Готово, когда:** за месяц с group=day приходит ~30 точек; сумма точек revenue равна выручке из /summary за тот же период.

### 10. Срезы по тренерам и услугам · 🟡 · 1:00
**Простыми словами:** таблицы «по тренерам» (занятия, выручка) и «по услугам» (выручка, доля) за период.
**Зачем:** вкладки Отчётов «По тренерам» и «По услугам», сводные виджеты Дашборда; этот же эндпоинт питает показатели в профиле сотрудника (записи, выручка, загрузка) — их фронт-подключение сделать вместе с задачами 15–16.

**Контракт:**

```
GET /analytics/services?date_from=&date_to=  → [{ service, revenue, share_pct }]
```
**Готово, когда:** оба эндпоинта отдают реальные строки; сумма выручки по тренерам не превышает общую из /summary.

### 11. Задачи на сегодня и лента событий · 🟡 · 1:30
**Простыми словами:** простой CRUD задач владельца плюс чтение ленты «что произошло в студии» с автозаписью событий.
**Зачем:** два виджета Дашборда; лента по ROADMAP наполняется каждым эпиком.
**Где:** новый `back/routers/analytics/tasks.py`, `back/routers/analytics/reports.py`; подключить в `analytics/router.py`; точки записи в `back/routers/finances/operations.py` и `back/routers/clients/profiles.py`.
**Контракт:**
```

GET /analytics/activity?limit=20  → [ActivityLogRead], DESC по времени
```
**Что конкретно сделать:** хелпер `log_activity()` и вызовы из создания операции, создания клиента, записи на занятие.
**Готово, когда:** задача создаётся/выполняется/удаляется через Swagger; после создания клиента и операции в /activity появились две записи.

## Порядок работы (что за чем)

Строго по фазам: **1 → 2 → 3 → 4**.
- Внутри фазы 1: 1 → 2 → 3 → 4, затем 5–7 в любом порядке.
- Фаза 3 (фронт Финансов) стартует после задач 2–7; фаза 4 (Дашборд и Отчёты) — после 8–11; задача 17 (бейджи) — в любой момент фазы 4.
- Задача 7 (зарплаты) и части задач 8/10 (посещаемость, записи) требуют работающего Эпика 2 — без занятий эти цифры будут нулевыми, но эндпоинты не сломаются.

## Что специально НЕ входит в Эпик 3

- **Вкладки «Онлайн-платежи» и «Методы оплаты»** — остаются на моках: каналы приёма — Эпик 4, реальные платежи — Эпик 5.
- **`DailyMetricSnapshot`** — не используется; агрегаты живые. Снапшоты вернутся как оптимизация после MVP, если живые запросы начнут тормозить.
- **Recharts** — не внедряем; кастомные графики уже работают (зависимость висит в package.json, но это долг на потом).
- **Отзывы (StudioReview) и цели тренеров (TrainerSalesGoal)** — модели есть, UI-потребителя в этом эпике нет.
- **Автосоздание операций из оплат** — в Журнале MVP нет приёма денег, поэтому «события оплаты формируют операции» из ROADMAP в этом эпике сводится к ручным операциям и зарплатным выплатам; автоматические операции появятся с онлайн-оплатами в Эпике 5. Факт «визита» закрывает отметка посещения (Эпик 2, задача 18).

## Как проверить (Verification)

1. **Бэкенд:** Swagger (`http://localhost:8000/docs`) — создать счёт → доход 5000 (категория «Абонементы») → расход 2000 → `/analytics/summary` показывает выручку 5000, расходы 2000, прибыль 3000; `/finances/operations` отдаёт Page с total=2.
2. **Зарплата:** у тренера с занятиями (Эпик 2) `/finances/salaries` считает по ставке; `pay` создаёт расходную операцию — расходы в `/summary` выросли.
3. **Фронт:** `cd front && npm run dev` — создать операцию в UI, **обновить страницу (F5)** — всё на месте; Дашборд показывает те же цифры; добавить/выполнить задачу в виджете; выгрузить CSV из Отчётов и открыть файл. `npm run build && npm run lint` — чисто.
