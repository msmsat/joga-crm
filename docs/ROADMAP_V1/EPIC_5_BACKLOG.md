- **Из Эпика 1:** `require_role`, `get_studio_context`.

## Жёсткие правила эпика (MVP)

1. **Один провайдер, ноль абстракций.** Никаких Strategy/Factory и per-studio настроек шлюза: это подписка студии на **саму CRM** (деньги получает Velora), мерчант Fondy один на всё приложение, креды в `.env`, вебхук один. Прямой модуль `back/services/fondy.py` без интерфейсов «на будущее».
2. **Вебхук — единственный источник истины о платеже.** Статус меняется только из server callback Fondy, никогда — из redirect'а пользователя.
3. **Подпись проверяется на каждом вебхуке** (SHA1-схема Fondy: секрет + отсортированные непустые параметры через вертикальную черту). Неверная подпись → 403 + лог, тело не обрабатывается — фейковое «Оплачено» не пройдёт.
4. **Идемпотентность через `order_id`:** уникальный на каждый инвойс; повторный вебхук по уже обработанному инвойсу → 200 no-op. Банк может прислать статус 3–4 раза — тариф начислится один.
5. **PCI: карточные данные не касаются нашего сервера.** Оплата — редирект на Fondy Checkout; «привязанная карта» — токен `rectoken` из вебхука + masked card для UI. Форма ввода номера карты на фронте удаляется.
6. **Суммы в копейках (Integer)**, сумма к оплате считается ТОЛЬКО на сервере по каталогу — фронту не доверяем.
7. **⚠️ Валюта:** Fondy не обрабатывает RUB — валюта списания задаётся `FONDY_CURRENCY` в .env (UAH/USD/EUR); ценники «₽» на витрине остаются по ТЗ. Решение — задача 0b.
8. **Источник истины о подписке — база данных, НЕ JWT.** Статус оплаты в токен не зашиваем: JWT выдаётся при входе и живёт до перелогина — юзер оплатил посреди сессии (или подписка истекла), а токен всё ещё говорит старое; данным на клиенте доверять нельзя. Сервер проверяет подписку по БД зависимостью (задача 8b), фронт узнаёт статус из GET /billing/plan при загрузке приложения.
9. **Оплаты в онбординге нет.** Путь юзера: регистрация → план `none` (без даты) → онбординг пройден → `free_trial` на 14 дней с полным доступом → оплата, когда триал кончился или юзер сам захотел раньше. Неоплативший видит только страницу «Тариф и оплата».

**Обозначения сложности:** 🟢 простая · 🟡 средняя · 🔴 сложная, делать внимательно.

**Что конкретно сделать:**

### 0b. Решение по валюте списания · 🟢 · 0:15
**Простыми словами:** Fondy не обрабатывает RUB. Нужно решить и зафиксировать: в какой валюте реально списываем (UAH/USD/EUR) и как это соотносится с ценниками «₽» на витрине.
**Зачем:** правило 7 говорит «бизнес-вопрос, зафиксировать до боевого запуска» — вот задача, которая его фиксирует; без решения нельзя заполнить `FONDY_CURRENCY`.
**Что конкретно сделать:** выбрать валюту; решить, менять ли ценники на витрине (₽ → выбранная валюта) или держать двойное отображение; записать решение в этот файл и в `docs/ROADMAP.md`.
**Готово, когда:** `FONDY_CURRENCY` определена; решение по витрине записано.

> **✅ РЕШЕНО (2026-07-12):** Целевые рынки — Украина, ЕС, Чехия и др. (RUB исключён, Fondy его не проводит).
> - **Валюта списания = валюта студии.** Юзер выбирает валюту в онбординге/настройках; `FONDY_CURRENCY` в `.env` — лишь дефолт/фолбэк (сейчас `UAH`), а источник истины — региональные настройки студии.

> - **Витрина:** единый символ выбранной валюты (не `₽`). Символ ценников следует за валютой студии.
> - **Следствие (не входит в 0b):** заменить хардкод `₽` в форматтерах фронта (`formatCurrency` и пр.) на символ из валюты студии — это отдельная задача витрины/онбординга, здесь только зафиксировано решение.

## Фаза 2 — Fondy

### 3. Fondy-клиент и криптоподпись · 🔴 · 2:00
**Простыми словами:** модуль общения с Fondy: построить подпись, проверить подпись, создать ссылку на оплату.
**Зачем:** ядро интеграции; подпись — защита от фейкового вебхука «Оплачено».
**Где:** новый `back/services/fondy.py`; `back/.env.example`.
**Что конкретно сделать:**
1. `build_signature(params)`: SHA1 от секрета + отсортированные по ключу непустые параметры через вертикальную черту (схема Fondy).
2. `verify_signature(payload)` для вебхуков.
3. `create_checkout(order_id, amount, description, server_callback_url, response_url, required_rectoken=Y)` через aiohttp → POST api.fondy.eu/api/checkout/url → checkout_url.
4. Креды `FONDY_MERCHANT_ID`/`FONDY_PASSWORD`/`FONDY_CURRENCY` из .env по паттерну `back/security.py`; для дева — тестовый мерчант (задача 0a).
**⚠️ Подводные камни:** в подпись входят только НЕПУСТЫЕ параметры — включение пустых строк даёт неверную подпись, которую сложно отлаживать.
**Готово, когда:** юнит-проверка — подпись совпадает с примером из документации Fondy; неверная подпись не проходит verify; checkout_url открывается в тестовом режиме.

### 4. Создание платежа (checkout) · 🟡 · 1:00
**Простыми словами:** выбор тарифа и периода создаёт счёт в статусе pending и возвращает ссылку на страницу оплаты Fondy.
**Зачем:** начало платёжного цикла; сумма считается на сервере — фронту не доверяем.
**Где:** новый `back/routers/billing/checkout.py`; подключить в `billing/router.py`.
**Контракт:**
```
POST /billing/checkout
Тело: { plan: start|pro|business, period_months: 1|6|12|24 }
Ответ 200: { checkout_url }
Ошибки: 422 неизвестный план/период
```
**Что конкретно сделать:** валидировать по каталогу задачи 2; посчитать сумму со скидкой на сервере; создать BillingInvoice status=pending с уникальным order_id (`velora-{invoice_id}-{uuid4}`); вызвать fondy.create_checkout.
**⚠️ Подводные камни:** повторный checkout по тому же плану — новый инвойс (старые pending протухают у Fondy сами), не пытаться переиспользовать order_id.
**Готово, когда:** POST возвращает рабочую ссылку; в базе pending-инвойс с order_id; сумма за 12 мес Pro = 2490×12×0.7 в копейках.

### 5. Вебхук: подпись, идемпотентность, активация · 🔴 · 2:00
**Простыми словами:** публичный приёмник колбэков от банка: проверяет подпись, не даёт начислить тариф дважды, активирует подписку. Самая критичная задача эпика.
**Где:** новый `back/routers/billing/webhook.py`; подключить в `billing/router.py`.
**Контракт:**
```
POST /billing/webhook/fondy   — без JWT (паттерн auth/register.py)
Ответ: всегда 200 после валидной обработки; 403 при неверной подписи
```
**Что конкретно сделать — шаги строго по порядку:**
1. `verify_signature` — неверная → 403 и лог.
2. Найти инвойс по order_id — нет → 200 и лог (не 404: банку нельзя отвечать ошибкой на мусор).
3. Идемпотентность: инвойс уже paid и пришёл approved → 200 no-op.
4. approved → status=paid, paid_at=now; активировать/продлить StudioBillingPlan (plan_name, expires_at += period_months, status=active); сохранить rectoken + masked_card в PaymentCard (upsert по user_id).
5. declined/expired → status=failed.
**⚠️ Подводные камни:** банк может прислать один и тот же статус 3–4 раза — тариф должен начислиться ровно один раз; ошибочный 4xx/5xx в ответ на валидный вебхук заставит Fondy ретраить и может задвоить обработку при гонке.
**Готово, когда:** тестовая оплата активирует подписку; повторная отправка того же вебхука ничего не меняет; вебхук с подделанной подписью → 403 и никаких изменений в базе.

### 6. Возвраты (refund) · 🔴 · 1:30
**Простыми словами:** владелец может вернуть платёж; возврат откатывает подписку.
**Зачем:** обязательная часть платёжки по ROADMAP; без него споры решаются вручную через саппорт банка.
**Где:** `back/routers/billing/checkout.py` (или новый `refunds.py`), `back/services/fondy.py`, `back/routers/billing/webhook.py`.
**Контракт:**
```
POST /billing/invoices/{id}/refund   — require_role(owner)
Ответ 200; Ошибки: 409 не-paid инвойс или повторный refund
```
**Что конкретно сделать:** вызвать Fondy reverse (order_id, amount, currency + подпись); в вебхуке обработать `order_status=reversed` → инвойс status=refunded + откат подписки (expires_at −= period_months; если в прошлом — status=expired).
**Готово, когда:** возврат тестового платежа помечает инвойс refunded; подписка откатилась; второй refund того же инвойса — 409.

### 7. Продление по сохранённой карте (rectoken) · 🟡 · 1:00
**Простыми словами:** кнопка «Продлить» списывает оплату по сохранённой карте без редиректа на форму.
**Зачем:** удержание — продление в один клик; фундамент будущего автосписания.
**Где:** `back/routers/billing/checkout.py`, `back/services/fondy.py`.
**Контракт:**
```
POST /billing/renew
Ответ 200: { invoice_id }; Ошибки: 400 нет сохранённой карты
```
**Что конкретно сделать:** взять rectoken из PaymentCard; создать pending-инвойс с новым order_id; вызвать Fondy recurring (rectoken + подпись); итоговый статус придёт в тот же вебхук задачи 5.
**⚠️ Подводные камни:** автосписание по крону НЕ делаем (нет планировщика) — `auto_renewal` остаётся флагом для UI.
**Готово, когда:** продление без редиректа создаёт paid-инвойс через вебхук; expires_at сдвинулся на период; без сохранённой карты — 400.

## Фаза 3 — Лимиты тарифа

### 8. Применение лимитов тарифа · 🟡 · 1:00
**Простыми словами:** тариф Старт не даёт завести больше 3 сотрудников и 100 клиентов; Pro — свои лимиты.
**Зачем:** монетизация без лимитов не работает — незачем платить за Pro.
**Где:** новый `back/services/plan_limits.py`; вызовы в POST создания сотрудника (`back/routers/staff`) и клиента (`back/routers/clients/profiles.py`).
**Что конкретно сделать:** `check_plan_limit(db, studio_id, entity)`: взять план студии (`none` — доступ уже закрыт гейтом 8b; `free_trial` — лимиты Pro); сравнить COUNT сотрудников/клиентов с лимитом каталога задачи 2; превышение → 403 с detail-кодом `limit_exceeded` и понятным текстом; истёкшая подписка (expires_at < now) блокирует создание так же.
**Готово, когда:** на Старте 4-й сотрудник и 101-й клиент получают 403 limit_exceeded; после апгрейда на Pro — создаются; Business не ограничен.

### 8b. Жизненный цикл подписки и глобальный гейт · 🔴 · 2:00
**Простыми словами:** система всегда знает, оплачено у студии или нет, и не пускает неоплативших дальше страницы оплаты. Путь юзера: регистрация → план `none` (без даты) → онбординг пройден → `free_trial` на 14 дней с полным доступом → оплата → платный план.
**Зачем:** без этого задача 8 блокирует только создание записей — неоплативший может пользоваться CRM вечно. Это главная дыра монетизации.
**⚠️ Почему НЕ JWT:** хранить «оплачено до» в токене — плохой способ. Токен выдаётся при входе и не обновляется до перелогина: юзер оплатил посреди сессии — токен всё ещё говорит «не оплачено» (и наоборот — подписка истекла, а токен «оплачен»); плюс данным на клиенте доверять нельзя в принципе (как и с ролями в Эпике 1). Правильно: истина в БД (StudioBillingPlan), сервер проверяет зависимостью, фронт спрашивает GET /billing/plan при загрузке.
**Где:** `back/dependencies.py`, `back/routers/auth/onboarding.py` (завершение онбординга), `back/routers/billing/plans.py`.
**Что конкретно сделать:**
1. При регистрации строка StudioBillingPlan НЕ создаётся; GET /billing/plan для такой студии отдаёт `plan_name="none"`, `expires_at=null` — UI показывает прочерк, никакой даты.
2. На завершении онбординга создать план `free_trial`: `expires_at = now + 14 дней`, `status=trial`; лимиты триала = лимиты Pro (15 сотрудников / 1000 клиентов).
3. Зависимость `require_active_subscription`: план none или `expires_at < now` → **402** с detail-кодом `subscription_expired`. Повесить на все разделы данных (clients, staff, studio, schedule, finances, analytics, loyalty, настройки booking, settings, ai). **НЕ вешать** на `/auth/*`, `/billing/*`, `/public/*` и вебхук — иначе юзер не сможет ни войти, ни оплатить.
4. Автопродление без крона: если при запросе подписка истекла, но `auto_renewal=true` и есть rectoken — инициировать recurring-списание (задача 7) один раз (пометка «попытка была», чтобы не дёргать банк на каждый запрос); доступ открывается только после approved-вебхука. Опция: здесь же триггерное письмо «заканчивается через 3 дня» при заходе (если ≤3 дней и ещё не слали — notify() из Эпика 4 + пометка даты отправки).
**Готово, когда:** новый аккаунт до онбординга видит план none без даты; после онбординга — free_trial с датой +14 дней; юзер с истёкшим триалом получает 402 на /clients, но /billing/plan и /billing/checkout работают.

## Порядок работы (что за чем)

**Задачу 0a подать во время Эпика 3** (верификация идёт параллельно). Дальше строго по фазам **0b → 1 → 2 → 3 → 4 → 5**. Внутри фазы 2 порядок жёсткий: **3 → 4 → 5 → 6 → 7** — вебхук (5) нужен и возвратам (6), и продлению (7). Задачи 8 и 8b независимы и могут идти сразу после 2 (8b — ядро гейта: без него монетизация дырявая, делать не откладывая). Фронт (9–12c) — после 2–7; 12c — после 12b (использует его контекст подписки). Для локальной отладки вебхука нужен туннель (ngrok/localtunnel) — Fondy должен достучаться до localhost:8000.

## Что специально НЕ входит в Эпик 5

- **Мульти-провайдерная архитектура** (Strategy/Factory, per-studio шлюзы) — провайдер один, жёстко. Второй провайдер — это второй эпик, а не «ещё один класс».
- **Автосписание по расписанию** — нет планировщика; продление кнопкой по rectoken. Post-MVP.
- **Онлайн-оплаты клиентов студии** (абонементы/занятия через Fondy) — другой контур денег (мерчанты студий, а не Velora); post-MVP.
- **PDF-квитанции инвойсов** (pdf_url в модели) — существующая клиентская печать остаётся.
- **Переименование карточки «Яндекс.Касса (ЮKassa)»** в Настройки → Интеграции (`front/src/pages/dashboard/Settings/components/tabs/IntegrationsTab.tsx:30`) — территория Эпика 6; упоминания ЮKassa в `docs/TZ/FUNCTIONAL.md:1461` и `CLAUDE.md:132` обновляет владелец продукта.
