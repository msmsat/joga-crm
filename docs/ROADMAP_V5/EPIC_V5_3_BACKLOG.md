# Эпик V5-3 — Сводка-результат + программы Промокоды, Депозит, UI сертификатов

Цель эпика простыми словами: верх Лояльности должен отвечать на вопрос «что
мне дала лояльность в деньгах», а не «сколько тумблеров включено». Плюс два
пробела: сертификаты имеют полный бэк (выпуск с кодом, погашение), но выпустить
сертификат из UI **негде** — программа-пустышка; а опция рефералки «На депозит»
ссылается на депозит, которого в продукте не существует. Эпик добавляет
сводку-результат, программы «Промокоды и акции» и «Депозит», и UI сертификатов.

Источник: аудит 5 в `docs/ROADMAP_V5/ROADMAP.md`, пп. 3 (мёртвый UI
сертификатов), 7 (обсуждено 2026-07-16).

**Важно понимать про старт:**
- Идёт после V5-2: единицы денег уже рубли, stats уже со счётчиками,
  хуки уже на Query, страница уже на i18n — новые блоки строить сразу по этим
  правилам, не по старым.
- `getCertificates` / `createCertificate` / `redeemCertificate` на фронте уже
  написаны (`loyalty.api.ts:63-70`) и никем не вызываются — задача 5 их
  оживляет, API не трогаем.
- Карточек программ становится 6: баллы и уровни, скидки, сертификаты,
  реферальная, промокоды, депозит (абонементы ушли в Каталог — V3-3, фаза 8).
  Прогресс «N из M» — из длины списка, не хардкодом.

**Обозначения сложности:** 🟢 простая · 🟡 средняя · 🔴 сложная, делать внимательно.

## Фаза 2 — Новые программы

### 3. Промокоды и акции · 🔴 · 2:00
**Простыми словами:** владелец создаёт код `ЛЕТО25` со скидкой 25 %, сроком и
лимитом использований; при продаже/оплате код можно применить.
**Где:** `back/models/loyalty.py` (+ `StudioPromoCode`), новый
`back/routers/loyalty/promocodes.py`, дровер новой карточки в
`Loyalty/components/drawer/configs/`.
**Что конкретно сделать:**
1. Модель: code (unique per studio), discount_type (percent/amount), value,
   valid_until, usage_limit, used_count, is_active. Конфиг-программы отдельной
   таблицей не заводить — карточка «включена», если есть активные промокоды
   (лениво и честно).
2. CRUD `/loyalty/promocodes` (owner); `POST /loyalty/promocodes/check` —
   валидация кода (существует, не истёк, лимит не выбран) → скидка.
3. Применение MVP: поле «Промокод» при продаже абонемента
   (`POST /clients/{id}/subscription`) — цена операции уменьшается,
   `used_count += 1`. Остальные точки оплаты — по мере их появления.
4. Дровер: список кодов + создание (код, скидка, срок, лимит) + выключение;
   дубль кода → 409 с detail «Такой промокод уже есть» — тост показывает
   как есть.
**⚠️ Подводные камни:** код нормализовать (trim + upper) на бэке, иначе
«лето25» и «ЛЕТО25 » станут разными кодами.

### 4. Депозит / клубный баланс · 🔴 · 1:30
**Простыми словами:** у клиента появляется денежный баланс студии: пополнить,
списать в оплату, начислить как бонус рефералки.
**Где:** `back/models/loyalty.py` (`ClientLoyaltyCard.deposit_balance` — без
новой таблицы, депозит живёт на карте клиента), `back/routers/clients/loyalty.py`
(+ пополнение/списание по образцу баллов), карточка клиента (блок баланса),
дровер новой карточки программы.
**Что конкретно сделать:**
1. `deposit_balance: int = 0` (+ миграция) и `DepositTransaction` по образцу
   `LoyaltyPointTransaction` (история обязательна — это деньги).
2. `POST /clients/{id}/deposit` (сумма ±, описание; в минус не пускать —
   400). Пополнение через кассу создаёт доходную операцию (образец —
   продажа абонемента).
3. Дровер программы: тумблер + сводка (суммарный баланс, топ клиентов).
4. Рефералка: `bonus_type='deposit'` начисляет на депозит — опция «На
   депозит» перестаёт быть пустышкой (аудит п.3).
**⚠️ Подводные камни:** единицы — рубли целыми (правило V5-2).
