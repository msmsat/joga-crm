# Эпик V5-4 — Умные сценарии, сегменты, кампании, аналитика продлений

Цель эпика простыми словами: превратить Лояльность из набора настроек в
автопилот удержания — и это главный козырь против Altegio. Владелец собирает
правило человеческим языком («не был 21 день → подарить 200 баллов → email»),
система ежедневно сама находит таких клиентов и действует. Рядом — живые
сегменты («на грани ухода», «VIP без визита 14 дней») с одной конкретной
кнопкой, и аналитика продлений абонементов.

Источник: аудит 5 в `docs/ROADMAP_V5/ROADMAP.md`, п. 7 (обсуждено 2026-07-16).

**Важно понимать про старт:**
- Идёт последним: наградам нужны пакеты с честным сроком (V3-3 фаза 8, Каталог), депозит и
  промокоды (V5-3), честные единицы денег (V5-2).
- Канал доставки MVP — **email**: `back/services/notifier.py` уже умеет
  матрицу событий и отправку; telegram/whatsapp/sms там честно пропускаются
  до появления интеграций. Селектор канала в конструкторе показывает
  недоступные каналы задизейбленными с подписью «после подключения
  интеграции» — не врать, что отправим в Telegram.
- Сегменты **не** хранятся в БД — это SQL-запросы на лету. Данные для всех
  пяти уже есть: `Client.last_visit`, `status`, `ClientSubscription`
  (остаток/срок), `Reservation.attended`.

**Обозначения сложности:** 🟢 простая · 🟡 средняя · 🔴 сложная, делать внимательно.

## Фаза 1 — Сценарии (бэк)

### 1. Модель + CRUD + шаблоны · 🟡 · 1:15
**Где:** `back/models/loyalty.py` (+ `LoyaltyScenario`), новый
`back/routers/loyalty/scenarios.py`.
**Что конкретно сделать:**
1. Модель: `trigger_type` (inactive_days / low_subscription / birthday /
   nth_visit / referral), `trigger_params` JSON (пороги: дней, занятий, N),
   `action_type` (points / gift_classes / certificate / renewal_offer),
   `action_params` JSON (сколько баллов/занятий, номинал, package_id,
   скидка), `channel` (`email` — пока единственный живой), `is_enabled`,
   `fired_count`, `last_run_at`.
2. CRUD `/loyalty/scenarios` (owner). POST без тела полей — из шаблона:
   `template` = один из 5 (дефолтные параметры из аудита: 21 день, ≤2
   занятия, ДР, 5-й визит, привёл друга).
3. Валидация параметров: дни ≥ 1, баллы/занятия ≥ 1, скидка 0–100.

### 2. Исполнитель · 🔴 · 2:00
**Простыми словами:** раз в сутки бэк сам проходит включённые сценарии,
находит подходящих клиентов и выполняет действие — и никогда не бьёт одного
клиента одним сценарием дважды подряд.
**Где:** новый `back/services/scenario_runner.py`, `back/main.py` (lifespan).
**Что конкретно сделать:**
1. Фоновый asyncio-таск в lifespan: раз в сутки (и прогон при старте, если
   `last_run_at` старше суток) — по образцу ничего нет, пишем первый; без
   APScheduler, обычный `asyncio.create_task` + `sleep`.
2. Матчинг по trigger_type — те же SQL, что сегменты (задача 5): выделить
   общие запросы в модуль, **не** копипастить.
3. **Идемпотентность:** таблица `ScenarioFire` (scenario_id, client_id,
   fired_at) — клиент не попадает под тот же сценарий повторно, пока не
   выполнит «выход из условия» (для inactive_days — новый визит; для
   birthday — раз в год; для nth_visit — по номеру визита). Это сердце
   задачи, продумать на каждый триггер.
4. Ошибка одного клиента не валит прогон (try/except + лог, как в notifier);
   `fired_count += n`, `last_run_at` — после прогона.
**⚠️ Подводные камни:** несколько воркеров uvicorn = несколько тасков —
на MVP фиксируем один воркер (как сейчас и есть), пометить ponytail-комментом
(апгрейд — advisory lock в PG).

### 3. Действия · 🟡 · 1:30
**Где:** `scenario_runner.py`, переиспользуя готовое.
**Что конкретно сделать:**
1. `points` → `_apply_points` из `clients/loyalty.py` (готов).
2. `gift_classes` → **абонемент как награда**: +N к `total_classes` активного
   абонемента клиента; нет активного — создать бесплатный `ClientSubscription`
   на N занятий со сроком 30 дней (продажу и операцию не создавать — это
   подарок).
3. `certificate` → `create_certificate`-логика с номиналом из params.
4. `renewal_offer` → письмо через notifier с пакетом (название, цена,
   скидка из params) — новый event_id в `_render`.
5. Каждое действие дублируется письмом клиенту («Вам подарок: …»), если
   у клиента есть email; нет email — действие выполняется, письмо
   пропускается молча (лог).

## Фаза 2 — UI

### 4. Конструктор и карточки сценариев · 🔴 · 2:00
**Простыми словами:** сценарий читается как предложение: «Если клиент [не был
21 день] → [подарить 200 баллов] → [Email]», не как автоматизация для
маркетолога.
**Где:** новая секция `Loyalty/components/sections/ScenariosBoard.tsx` +
дровер/модалка на ките.
**Что конкретно сделать:**
1. Карточка сценария: фраза-предложение, тумблер, счётчик «сработал N раз»,
   последний прогон. Выключенные — приглушённые.
2. «+ Сценарий» → выбор из 5 шаблонов (человеческие названия и описания) →
   параметры инлайн в фразе (число дней/баллов/занятий — инпуты прямо в
   предложении, селекты из кита).
3. Каналы: email активен; telegram/whatsapp — задизейблены с подписью
   «после подключения интеграции».
4. Query-ключ `['loyalty','scenarios']`; мутации с тостами и валидацией
   (правила V5-2). Всё через `t()`.

## Фаза 3 — Сегменты

### 5. `/loyalty/segments` + блок · 🟡 · 1:30
**Где:** `back/routers/loyalty/` (+ `segments.py`), новая секция
`SegmentsBoard.tsx`.
**Что конкретно сделать:**
1. GET `/loyalty/segments` → 5 сегментов с count и превью клиентов (имя +
   дней с визита): at_risk (21+ день без визита, активные), vip_idle (VIP,
   14+ дней), expiring_subscription (≤2 занятия или ≤7 дней срока),
   lost_newcomers (1 визит, 14+ дней, без второго), upsell_candidates
   (8+ визитов за 30 дней, без активного абонемента).
2. SQL — общий модуль со сценариями (задача 2), один источник правды.
3. UI: карточка сегмента = название, счётчик, 2–3 клиента превью,
   **одна кнопка** («Выдать бонус» или «Запустить кампанию» — по смыслу
   сегмента). Пустой сегмент — «Здесь пусто, и это хорошо».

### 6. Кампании · 🟡 · 1:15
**Где:** `segments.py`, `SegmentsBoard.tsx`.
**Что конкретно сделать:**
1. `POST /loyalty/segments/{key}/campaign` — body: `{action: 'points'|'email',
   points?, message?}`: bulk-начисление и/или письмо каждому в сегменте
   (notifier, пропуская клиентов без email), ответ — «обработано N,
   писем M».
2. UI: кнопка → ConfirmModal («Начислить 200 баллов 14 клиентам сегмента
   „На грани ухода"?») → прогресс → success-тост с итогом.
3. Инвалидация `['loyalty','stats']` и `['loyalty','segments']`.

## Фаза 4 — Аналитика

### 7. Продления и удержание · 🟡 · 1:15
**Где:** `cards.py` или `segments.py` (+ `/loyalty/retention`),
`StatsBoard.tsx` (график).
**Что конкретно сделать:**
1. GET `/loyalty/retention`: % продливших (клиенты, купившие следующий
   `ClientSubscription` в течение 30 дней после окончания предыдущего),
   среднее число пакетов на клиента, ряд по месяцам (6 мес) — продано /
   продлено.
2. График в сводке (Recharts, как везде) + два числа-KPI; empty-state,
   пока продаж абонементов нет.
**⚠️ Подводные камни:** `ClientSubscription` не хранит `package_id` и дату
покупки хранит только через `expires_at` — добавить `package_id` (nullable)
и `created_at` (+ миграция), иначе продления не посчитать честно. Продажа
(`clients/subscriptions.py`) начинает их заполнять; старые строки — null,
аналитика честно считает с момента миграции.
