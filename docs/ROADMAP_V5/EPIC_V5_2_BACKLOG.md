# Эпик V5-2 — Честная Лояльность: деньги, ключи-значения, i18n, Query, ошибки, уровни

Цель эпика простыми словами: страница Лояльности должна перестать врать и
замолкать. Сегодня карточки программ показывают выдуманные числа («89
клиентов» — хардкод), баллы начисляются в 100 раз меньше положенного (бэк
делит рубли как копейки), на свежем конфиге сертификатов не подсвечен ни один
тип (русские значения фронта не совпадают с англ. дефолтами БД), ошибки
глотаются молча, i18n — ноль, а уровни лояльности нельзя редактировать. Эпик
чинит всё это и переводит страницу на общий фундамент (Query + UI-кит).

Источник: аудит 5 в `docs/ROADMAP_V5/ROADMAP.md`, пп. 1–5, 8 (обсуждено 2026-07-16).

**Важно понимать про старт:**
- Баг денег подтверждён кодом: Финансы пишут `amount` рублями
  (`OperationsTab.tsx:108` — `parseInt` без ×100), а `accrue_points`
  (`back/routers/clients/loyalty.py:58`) считает `(amount // 100) // rate` —
  копейками. Оплата 1500 ₽ при курсе 100 даёт 0 баллов вместо 15.
- Второе место с копейками — `back/services/notifier.py` (шаблон c4:
  `amount / 100`). Больше делений на 100 в проекте нет (проверено grep'ом),
  фикс — два места + тест.
- Эндпоинты страницы все реальные (таблица в аудите) — эпик не строит новых,
  кроме `PUT /loyalty/levels` и счётчиков в `/loyalty/stats`.
- Этот эпик первый в очереди аудита 5: на кривых единицах и русских
  значениях-в-БД вся новая аналитика V5-3/V5-4 будет врать.

**Обозначения сложности:** 🟢 простая · 🟡 средняя · 🔴 сложная, делать внимательно.

## Фаза 1 — Деньги

### 1. Рубли целыми по всему проекту · 🟡 · 1:00
**Простыми словами:** оплата 1500 ₽ при курсе «100 ₽ = 1 балл» должна давать
15 баллов, а письмо клиенту — писать «1500 ₽», не «15 ₽».
**Где:** `back/routers/clients/loyalty.py:58` (`accrue_points`),
`back/services/notifier.py` (шаблон c4), `back/models/loyalty.py`
(`referrer_bonus` default 50000 → 500), `back/tests/test_loyalty_points.py`.
**Что конкретно сделать:**
1. `accrue_points`: `points = amount // cfg.points_exchange_rate` — убрать
   `// 100`, поправить докстринг («сумма в рублях»).
2. `notifier.py` c4: `f"{amount or 0:.0f} ₽"` — убрать `/ 100`.
3. Дефолт `referrer_bonus` → 500 (+ миграция значения по умолчанию; уже
   сохранённые 50000 поправить в той же миграции, если строка не трогалась).
4. Обновить `test_loyalty_points.py` под рубли; прогнать.
**⚠️ Подводные камни:** биллинг тарифов (Fondy) живёт в копейках — это его
родная единица, **не трогать**; правило «рубли» — про `Operation.amount`,
балансы счетов и цены пакетов.

## Фаза 2 — Ключи

### 2. Значения конфигов — ключи · 🟡 · 1:15
**Простыми словами:** в БД должны лежать языконезависимые ключи, а не «Именной»
и «После первого визита» — иначе en-интерфейс невозможен, а свежие конфиги
уже сегодня не совпадают с кнопками UI.
**Где:** `CertificatesConfig.tsx` (`TYPES`), `ReferralConfig.tsx`
(`CONDITIONS`, `BONUS_TYPES`), `LoyaltyConfig.tsx` (`EXPIRY_OPTIONS`);
дефолты моделей уже ключами — `gift`, `first_visit`, `points`, `never`.
**Что конкретно сделать:**
1. Фронт: опции — пары `{key, label: t(...)}`; в БД уходит только key:
   `gift/named/service`, `first_visit/first_payment/registration`,
   `points/deposit/discount`, `3m/6m/1y/never`.
2. Alembic data-миграция: маппинг старых русских значений на ключи
   (по образцу миграции категорий услуг из V3).
3. Проверить: свежая студия → GET конфига → в дровере подсвечен дефолтный
   тип (сейчас — не подсвечен ни один, это баг из аудита п.1).

### 5. Ошибки видимы + валидация · 🟡 · 1:15
**Где:** `useLoyalty.ts` (все мутации), три catch-заглушки
(`useLoyalty.ts:52`, `StatsBoard.tsx:41`, `SubscriptionsConfig.tsx:27` —
последняя уедет с V5-1), формы дроверов.
**Что конкретно сделать:**
1. Мутации: onError → error-тост из кита (`detail` бэка как есть, статусы —
   переводом); `toggleProgram` при ошибке возвращает тумблер назад.
2. Ошибки загрузки → тост + состояние «не удалось загрузить, повторить».
3. Валидация до отправки: курс ≥ 1, `expiry_days` ≥ 1, номинал > 0, скидка
   0–100 %, бонусы ≥ 0, название программы непустое — подсветка + сообщение.
4. Успешное сохранение → success-тост («Программа активирована»).

### 7. Уровни редактируемые · 🔴 · 1:30
**Простыми словами:** владелец должен сам решать, с какой суммы начинается
«Золото», и уметь добавить четвёртый уровень.
**Где:** `back/routers/loyalty/cards.py` (+ `PUT /loyalty/levels`),
`back/schemas/loyalty.py`, `LoyaltyConfig.tsx` (статичный блок «Уровни»).
**Что конкретно сделать:**
1. `PUT /loyalty/levels` — bulk-замена: список {id?, name, color,
   min_threshold, max_threshold, sort_order}. Валидация: ≥ 1 уровня, первый
   с 0, последний с max=null, пороги непрерывны без дыр/перекрытий → 400 с
   человеческим detail.
2. Удалённые уровни: карты перепривязываются на лету (`_level_for` уже
   пересчитывает по порогам при каждом GET `/cards` — ничего не мигрировать).
