# Эпик V5-1 — Абонементы в Каталоге: вкладка, CRUD, срок и услуги пакета

Цель эпика простыми словами: абонемент — это товар студии, как услуга или зал,
и жить он должен в Каталоге. Сегодня пакеты занятий показываются read-only в
дровере Лояльности (создать/изменить из UI нельзя, хотя CRUD на бэке готов),
длительность продажи захардкожена 90 дней, а «на какие занятия» пакет — вообще
не существует как понятие. Эпик заводит в Каталоге третью вкладку «Абонементы»
с полным CRUD и убирает абонементы из Лояльности.

Источник: аудит 5 в `docs/ROADMAP_V5/ROADMAP.md`, п. 6 (обсуждено 2026-07-16).

**Важно понимать про старт:**
- Бэк-CRUD уже написан (`back/routers/loyalty/packages.py`): list/create/patch
  + мягкое удаление (`is_active=false`). Фронт-методы тоже есть
  (`loyalty.api.ts:72-82`) — они просто никем не вызываются. Эпик в основном
  про UI и два новых поля.
- Продажа клиенту (`back/routers/clients/subscriptions.py`) уже работает и
  помечена ponytail-комментарием: срок — константа `_SUBSCRIPTION_VALID_DAYS = 90`.
  Это место забирает `duration_days` из пакета.
- Модалку строить на модальном ките V3-2 (задача 5 того эпика). Если кит ещё
  не готов — правило прежнее: кто первый, тот и создаёт по общей спеке.

**Обозначения сложности:** 🟢 простая · 🟡 средняя · 🔴 сложная, делать внимательно.

## Фаза 1 — Бэк

### 1. `duration_days` + `service_ids` в пакете · 🟡 · 1:00
**Простыми словами:** пакет должен сам знать, сколько он действует и на какие
занятия распространяется.
**Где:** `back/models/loyalty.py` (`SubscriptionPackage`),
`back/schemas/loyalty.py`, `back/routers/clients/subscriptions.py:23`.
**Что конкретно сделать:**
1. Модель: `duration_days: int = 90`, `service_ids: JSON | None`
   (null = все занятия). Alembic-миграция.
2. Схемы Create/Update/Read — те же поля.
3. `sell_subscription`: `expires_at = today + timedelta(days=package.duration_days)`,
   константу `_SUBSCRIPTION_VALID_DAYS` удалить вместе с ponytail-комментарием.
**⚠️ Подводные камни:** `service_ids` пока только хранится и показывается
(фильтрация записи по услугам — вместе с клиентским мини-приложением);
не городить проверок при записи на занятие в этом эпике.

### 2. Пути переезжают в `/catalog/subscriptions*` · 🟢 · 0:30
**Простыми словами:** раздел в Каталоге — значит и API там же, эндпоинты
должны быть прозрачными.
**Где:** `back/main.py` / `back/routers/loyalty/router.py`,
`back/routers/studio/router.py`, `front/src/api/loyalty/loyalty.api.ts`.
**Что конкретно сделать:**
1. `packages.py` и config-эндпоинты `/loyalty/subscriptions-config` включить
   под каталожный роутер: `/catalog/subscriptions`, `/catalog/subscriptions-config`.
2. Фронт: методы пакетов и конфига абонементов перенести в
   `front/src/api/catalog/` (или обновить пути в `loyalty.api.ts` — по месту),
   типы `SubscriptionPackage`, `SubscriptionProgramConfig` — за ними.
**⚠️ Подводные камни:** продажа клиенту остаётся на своём месте
(`POST /clients/{id}/subscription`) — её не трогать.

## Фаза 2 — Каталог

### 3. Вкладка «Абонементы»: список пакетов · 🟡 · 1:30
**Простыми словами:** в Каталоге рядом со «Студии» и «Услуги» — третья вкладка
с карточками пакетов.
**Где:** `front/src/pages/dashboard/Catalog/Catalog.tsx` (`CatalogTab`),
новый `components/SubscriptionSection.tsx`, `locales/*/catalog.json`.
**Что конкретно сделать:**
1. Карточка пакета: название, число занятий, цена (валюта из настроек студии),
   срок (`duration_days`), услуги (chips или «Все занятия»), бейдж
   «Снят с продажи» для `is_active=false`.
2. Блок «Настройки программы» (заморозка / перенос / автопродление) — те же
   три тумблера, что жили в дровере Лояльности, PATCH конфига.
3. Пустое состояние: «Пакетов пока нет» + CTA «Создать пакет».
4. Все строки — через `t('catalog:subscriptions.…')` (ru + en) сразу, не
   вторым заходом.
**⚠️ Подводные камни:** вкладка доступна только владельцу — как весь Каталог,
дополнительных проверок не нужно.

### 4. Модалка пакета на ките + валидация · 🔴 · 1:30
**Простыми словами:** создать и изменить пакет — в единой премиальной модалке,
а не в третьей копии самодельного shell'а.
**Где:** `components/modals/` в Catalog, модальный кит из
`front/src/components/ui/modal/` (V3-2).
**Что конкретно сделать:**
1. Поля: название, число занятий, цена, срок действия (дней), услуги
   (мультивыбор из реальных `/studio/services`; пусто = все), активность.
2. Валидация до отправки: название непустое, занятий ≥ 1, цена ≥ 0,
   срок ≥ 1 — подсветка поля + сообщение (паттерн V3-3).
3. Ошибки сервера → единый error-тост; успех → success-тост + инвалидация
   `['catalog','packages']`.
**⚠️ Подводные камни:** `per_visit_price` не вводить руками — считать на лету
`price / class_count` и показывать подсказкой (одно поле лжи меньше).

### 5. Деактивация пакета через ConfirmModal · 🟢 · 0:30
**Простыми словами:** «удалить» пакет = снять с продажи (на бэке уже мягкое
удаление), с человеческим подтверждением.
**Где:** `SubscriptionSection.tsx`, `ConfirmModal` из кита.
**Что конкретно сделать:**
1. Кнопка «Снять с продажи» → ConfirmModal («Проданные абонементы продолжат
   действовать») → `DELETE /catalog/subscriptions/{id}` → инвалидация.
2. Для снятого пакета — кнопка «Вернуть в продажу» (`PATCH is_active=true`).

### 7. Сквозной прогон · 🟢 · 0:45
Создать пакет (8 занятий, 60 дней, услуга «Йога») → продать клиенту из
карточки клиента → проверить: остаток 8, срок = сегодня + 60 (не 90!),

пакета отдаёт 400. Всё — на двух языках интерфейса.
