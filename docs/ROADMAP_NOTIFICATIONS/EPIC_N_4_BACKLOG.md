# Эпик N-4 — Боевые триггеры: зарплата, дни рождения, отчёты, врезки в процессы

Цель простыми словами: матрица перестаёт быть декорацией — каждое событие
каталога реально стреляет из рабочего процесса. Кнопка «Выплатить» шлёт
тренеру t6, дни рождения поздравляют клиента (c7) и напоминают тренерам (t8),
в конце дня админ получает «Отчёт за день» (a8), владелец — сводку (o1),
остальные события админа/владельца врезаны в свои эндпоинты.

**Важно понимать про старт:**
- Ядро готово (N-1): `notify(db, studio_id, role, event_id, context)` сама
  решает каналы/язык/валюту — каждая врезка это **одна строка** после
  успешного commit. `notify` не роняет запрос (try/except внутри).
- Уже подключены (клиент): c1 (`booking/public.py:251`,
  `reservations.py:79`), c3/c11 (`lessons.py:468/385`), c4
  (`operations.py:284`), c5/c6 (`reservations.py:217/220`) — не дублировать.
- Фоновый цикл-образец: `back/services/scenario_runner.py` (`_loop` +
  `start_scenario_loop` из `main.py` lifespan, своя сессия на итерацию).
- `Client.birth_date` есть (`back/models/client.py:19`); часовой пояс —
  `Studio.timezone` (nullable).
- Роли `trainer`/`admin`/`owner` в матрице до сих пор не имели ни одного
  реального вызова — этот эпик закрывает всё.

**Обозначения сложности:** 🟢 простая · 🟡 средняя · 🔴 сложная, делать внимательно.

## Фаза 3 — Фоновый цикл

### 6. daily_notify.py: дни рождения + отчёты дня/недели · 🔴 · 2:00
**Где:** новый `back/services/daily_notify.py`, `back/main.py` (lifespan).
**Что конкретно сделать:**
1. Каркас — копия паттерна `scenario_runner`: `_loop` со сном 30 минут,
   `start_daily_notify_loop(session_maker) -> asyncio.Task`; в `main.py`
   lifespan — второй таск рядом с `start_scenario_loop` (строка 38), отмена
   в `finally`.
2. Состояние «что уже слали сегодня» — без новой таблицы, в
   `StudioIntegration(integration_type="notify_state")`.config:
   `{"bday": "2026-07-22", "report": "...", "weekly": "...", "billing": "..."}`
   (дата студийного дня; переприсваивать config целиком — JSON-колонка).
3. На каждом тике, по каждой студии (select студий, у которых есть хоть один
   включённый toggle — не гонять пустые):
   - `tz = ZoneInfo(studio.timezone or "UTC")`, `today = now(tz).date()`;
   - **дни рождения** (первый тик дня, `state["bday"] != today`): клиенты с
     `birth_date` месяц+день == today →
     `notify(..., "client", "c7", {"client_id": c.id, "client_name": ...})`
     каждому; если именинники есть —
     `notify(..., "trainer", "t8", {"names": ", ".join(...)})` один раз;
   - **отчёт за день** (`now(tz).hour >= 20` и `state["report"] != today`):
     собрать `revenue` (sum `Operation.amount`, `type == "in"`, дата ==
     today), `lessons` (count `Lesson` на today), `new_clients` (count
     `Client.created_at` == today) → `notify(..., "admin", "a8", ctx)` и
     `notify(..., "owner", "o1", ctx)`; тут же **o4**: если
     `revenue < 0.5 * avg(выручка за 7 предыдущих дней)` и avg > 0 →
     `notify(..., "owner", "o4", ...)`
     (`# ponytail: наивная эвристика аномалии; статистика — после MVP`);
   - **еженедельный отчёт** (понедельник, первый тик дня): суммы за
     прошлую неделю → `notify(..., "owner", "o2", ...)`;
   - **тариф** (раз в день): `StudioBillingPlan.expires_at` и
     `notify_before_days` → осталось ≤ n дней → `notify(..., "owner", "o6",
     {"days_left": n})`.
4. Каждая студия — в своей сессии/try-except (образец —
   `run_due_scenarios`): падение одной не глушит остальных.
**⚠️ Подводные камни:** сначала обновить `state` в БД, потом слать? Нет —
наоборот: слать, потом фиксировать дату; при падении отправки на рестарте
попробует снова. Двойная отправка возможна только при падении между send и
commit — принято для MVP.
