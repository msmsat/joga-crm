# Epic B4 — Флоу оплаты: модалка IBAN/Карта + безопасный токен-метод

**Цель:** заменить прямой редирект на Fondy на **глобальную модалку выбора способа
оплаты** (IBAN vs Карта, §4), а физическую отрисовку карты — на безопасный
токен-бейдж (§5). Живые autopay-тумблеры с гейтом «только карта».

**Зависит от:** B0, B2 · **Блокирует:** —

**Файлы:** новый `components/modals/PaymentMethodModal.tsx`; правки
`PlansTab.tsx` (кнопка «Оплатить»), `PaymentMethodTab.tsx` (карта→токен, тумблеры).

## Frontend UI & Components

### 1. Модалка выбора способа оплаты (§4)

Триггер — кнопка «Оплатить» (`PlansTab.tsx:309`) и «Продолжить план» (B6).
Сейчас `startCheckout` (`useBillingCalculator.ts:52`) сразу редиректит — **меняем**:
кнопка открывает модалку, редирект/показ IBAN происходит внутри.

**Строго на ките** (`ModalShell` + `ModalHeader/Body/Footer` из `components/ui/index`,
§5 — свои модалки запрещены). Premium 2-колоночный грид (§5).

Шаг 1 — выбор ветки:
```
┌─────────── Способ оплаты ───────────┐
│  ┌───────────────┐  ┌─────────────┐ │
│  │  🏦  IBAN     │  │  💳 Карта   │ │   ← 2 tile-кнопки
│  │  Счёт/инвойс  │  │  Apple/GPay │ │
│  └───────────────┘  └─────────────┘ │
└──────────────────────────────────────┘
```

**Ветка IBAN** (в этом же окне, §4): вызвать `billingApi.checkoutIban(plan, period)`,
показать сгенерированный инвойс:
```
Инвойс INV-2026-000123
Получатель:   Velora CRM LLC
IBAN:         DE.. .... .... .. 1000 ...      [ Копировать ]
Назначение:   velora-123-<uuid>               [ Копировать ]
Сумма:        formatMoney(amount/100, currency)
```
- Кнопки «Копировать» → `navigator.clipboard.writeText` + `useToast().success('Скопировано')`.
- Внизу: «Счёт создан, оплатите по реквизитам — статус обновится после поступления».
- Никаких полей ввода карты в этой ветке.

**Ветка Карта** (§4): три действия
- `[ Apple Pay ]` и `[ Google Pay ]` — кнопки (в тестовом контуре ведут на тот же
  Fondy `checkout_url`, где доступны кошельки; фичи-детект `window.ApplePaySession`
  для показа Apple Pay). Не эмулируем платёж сами.
- `[ Оплатить картой ]` → `billingApi.checkout(plan, period)` → редирект на защищённую
  страницу Fondy (реквизиты вводятся **там**, не у нас — PCI, §5). Форму «ручного ввода
  реквизитов» рендерим как **заглушку-превью полей**, но submit уводит на Fondy —
  реальные PAN/CVV в наш фронт не попадают.

> Компонент модалки — **общий** для страницы, лежит в `Billing/components/modals/`.
> Не дублировать логику checkout в двух местах — `startCheckout` теперь только открывает модалку.

### 2. Безопасный токен-метод вместо отрисовки карты (§5)

**Удалить** `PaymentMethodTab.tsx:36-77` — премиальную визуальную карту с
`•••• {card_last4}`, `card_expiry`, держателем. Хранение/показ PAN недопустим.

**Заменить** на **токен-бейдж** (дизайн-решение, §5) — «сейф-чип», а не карта:

```
┌──────────────────────────────────────────────┐
│  🛡  Способ оплаты защищён                     │
│                                                │
│   ╭────────────────────────────────────╮      │
│   │  ●  Visa · tok_••••4242            │      │  ← бренд + токен-хвост, НЕ номер карты
│   │     Токен провайдера · истекает 12/29 │   │
│   ╰────────────────────────────────────╯      │
│                                                │
│   Реальные данные карты хранятся у платёжного  │
│   провайдера (PCI DSS L1). Мы держим только    │
│   одноразовый токен для продлений.             │
│                                                │
│         [ Продлить по этому методу ]           │
└──────────────────────────────────────────────┘
```

Ключевые отличия от старой карты:
- **Нет** ряда `•••• •••• •••• last4`, нет держателя, нет чипа-имитации PAN.
- Показываем только: `card_brand`, псевдо-токен `tok_••••{card_last4}` (last4 — это
  маскированный хвост, не PAN; допустимо), `card_expiry` как справку.
- Явная плашка «данные у провайдера, у нас — токен» → снимает вопрос PCI визуально.
- IBAN-метод (`method_type==='iban'`): бейдж «Оплата по счёту (IBAN)», **без** токена и
  **без** кнопки автопродления (см. §3).
- Пустой стейт (`!card`) — оставить текущий текст `PaymentMethodTab.tsx:95-100` (уже корректный, увести в i18n B0).

Кнопка «Продлить» — `Button` кита, вызывает существующий `billingApi.renew()`.

### 3. Живые autopay-тумблеры + гейт «только карта» (§4)

`PaymentMethodTab.tsx:122-139` — сейчас тумблеры **статичные** (`active:true` захардкожен,
ничего не сохраняют). Переделать:

- Значения — из `plan` (B1/B2): `auto_renewal`, `email_receipt_enabled`,
  `notify_before_autocharge`, `sms_notification_enabled`.
- Компонент тумблера — **`Switch` из `components/ui/index`** (§5), не инлайн-div.
- `onChange` → `billingApi.updateAutopay({ [key]: value })` → `setPlan(res)` (без F5),
  `useToast` на успех/ошибку.
- **Новый тумблер** (аудит §5) «Напоминание перед автосписанием» = `notify_before_autocharge`.
- **Гейт:** если метод — IBAN или карты нет, `auto_renewal` и зависящие от него тумблеры
  **disabled** + подпись «Доступно при оплате картой». Бэк дублирует запрет (B2), фронт
  не обходит его — только UX-подсказка.

## Frontend API & State

- `useBillingCalculator`: состояние модалки `showPayModal`, `payBranch: 'choose'|'iban'|'card'`,
  `ibanData: IbanCheckout | null`, `payBusy`. `startCheckout` → `setShowPayModal(true)`.
- `PaymentMethodTab`: локальный стейт тумблеров синхронизируется с `plan`; оптимистично
  флипаем, на ошибке откатываем + `useToast().error`.

## Проверка


Ручной чек: «Оплатить» → модалка на `ModalShell`, две ветки; IBAN показывает инвойс+IBAN
и копирование; карта показывает Apple/GPay+«оплатить картой»→Fondy. Вкладка «Способ оплаты»:
нет номера карты, есть токен-бейдж; тумблеры сохраняются (сетевой запрос в DevTools);
без карты — автосписание disabled.

> `[модалка IBAN/карта + токен-бейдж + живые Switch]` → skipped: настоящая интеграция Apple/Google Pay через Payment Request API — тестовый контур ведёт на Fondy-кошельки. Add when будет production-мерчант.
