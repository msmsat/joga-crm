# Эпик 5 — Биллинг и монетизация (Fondy)

> Принимать реальные деньги за подписку на CRM: тарифы Старт/Pro/Business, оплата через Fondy Checkout, вебхуки с проверкой подписи, возвраты, идемпотентность, лимиты тарифов.
> Источник: [`docs/ROADMAP.md`](ROADMAP.md), Эпик 5. Провайдер — **Fondy** (вместо ранее заявленной ЮKassa).

## Что уже есть в коде (точка старта)

- **Модели почти готовы, но впервые за все эпики нужна миграция:** `back/models/settings.py` — `StudioBillingPlan` (1:1 со студией: plan_name, billing_cycle, status, expires_at, max_staff, auto_renewal), `PaymentCard` (card_last4, brand, expiry — **нет `rectoken`**), `BillingInvoice` (plan_name, amount, status, paid_at — **нет `order_id`**, без которого не работают ни вебхук-матчинг, ни идемпотентность).
- **Каталога тарифов на сервере нет:** цены (990/2490/5990 ₽), скидки периодов (1/6/12/24 мес = 0/−20/−30/−40 %) и лимиты («До 3 сотрудников», «До 100 клиентов») живут только в `front/src/pages/dashboard/Billing/constants.ts`.
- **Роутера нет:** префикс `/billing` не зарегистрирован в `main.py`; `back/schemas/settings/billing.py` — заглушка.
- **Фронтовый контракт задан:** `front/src/api/billing/billing.api.ts` (getPlan/getInvoices/getPaymentCards/addPaymentCard/upgradePlan/cancelPlan) — но он писался под «мгновенный апгрейд» и ручной ввод карты; под Fondy-поток контракт меняется (задача 9).
- **Страница `front/src/pages/dashboard/Billing/`** полностью свёрстана (калькулятор периодов, история счетов с графиком, вкладка карты с формой ввода номера) — 100 % моки.
- **Инфраструктура:** `aiohttp` уже в requirements — вызовы Fondy без новых зависимостей; env-паттерн `load_dotenv()` + `os.getenv` (как `back/security.py`); паттерн публичного эндпоинта без JWT — `back/routers/auth/register.py`.
- **Из Эпика 1:** `require_role`, `get_studio_context`.

## Жёсткие правила эпика (MVP)

1. **Один провайдер, ноль абстракций.** Никаких Strategy/Factory и per-studio настроек шлюза: это подписка студии на **саму CRM** (деньги получает Velora), мерчант Fondy один на всё приложение, креды в `.env`, вебхук один. Прямой модуль `back/services/fondy.py` без интерфейсов «на будущее».
2. **Вебхук — единственный источник истины о платеже.** Статус меняется только из server callback Fondy, никогда — из redirect'а пользователя.
3. **Подпись проверяется на каждом вебхуке** (SHA1-схема Fondy: секрет + отсортированные непустые параметры через вертикальную черту). Неверная подпись → 403 + лог, тело не обрабатывается — фейковое «Оплачено» не пройдёт.
4. **Идемпотентность через `order_id`:** уникальный на каждый инвойс; повторный вебхук по уже обработанному инвойсу → 200 no-op. Банк может прислать статус 3–4 раза — тариф начислится один.
5. **PCI: карточные данные не касаются нашего сервера.** Оплата — редирект на Fondy Checkout; «привязанная карта» — токен `rectoken` из вебхука + masked card для UI. Форма ввода номера карты на фронте удаляется.
6. **Суммы в копейках (Integer)**, сумма к оплате считается ТОЛЬКО на сервере по каталогу — фронту не доверяем.
7. **⚠️ Валюта:** Fondy не обрабатывает RUB — валюта списания задаётся `FONDY_CURRENCY` в .env (UAH/USD/EUR); ценники «₽» на витрине остаются по ТЗ. Решение о реальной валюте продаж — бизнес-вопрос, зафиксировать до боевого запуска.

**Обозначения сложности:** 🟢 простая · 🟡 средняя · 🔴 сложная, делать внимательно.

## Порядок работы (что за чем)

Строго по фазам **1 → 2 → 3 → 4 → 5**. Внутри фазы 2 порядок жёсткий: **3 → 4 → 5 → 6 → 7** — вебхук (5) нужен и возвратам (6), и продлению (7). Задача 8 независима и может идти сразу после 2. Фронт (9–12) — после 2–7. Для локальной отладки вебхука нужен туннель (ngrok/localtunnel) — Fondy должен достучаться до localhost:8000.

## Что специально НЕ входит в Эпик 5

- **Мульти-провайдерная архитектура** (Strategy/Factory; per-studio шлюзы) — провайдер один, жёстко. Второй провайдер — это второй эпик, а не «ещё один класс».
- **Автосписание по расписанию** — нет планировщика; продление кнопкой по rectoken. Post-MVP.
- **Онлайн-оплаты клиентов студии** (абонементы/занятия через Fondy) — другой контур денег (мерчанты студий, а не Velora); post-MVP.
- **PDF-квитанции инвойсов** (pdf_url в модели) — существующая клиентская печать остаётся.
- **Смена валюты витрины** — ценники в ₽ как в ТЗ; реальная валюта списания — FONDY_CURRENCY (Fondy не обрабатывает RUB); бизнес-решение зафиксировать до боевого запуска.
- **Переименование карточки «Яндекс.Касса (ЮKassa)»** в Настройки → Интеграции (`front/src/pages/dashboard/Settings/components/tabs/IntegrationsTab.tsx:30`) — territория Эпика 6; упоминания ЮKassa в `docs/TZ.md:1461` и `CLAUDE.md:132` обновляет владелец продукта.

## Как проверить (Verification)

1. **Подпись:** юнит-прогон build_signature на примере из документации Fondy; вебхук с искажённой подписью через curl → 403, база не изменилась.
2. **Идемпотентность:** прислать один и тот же approved-вебхук 4 раза подряд → подписка продлена ровно один раз, инвойс один.
3. **Полный цикл:** тестовый мерчант + ngrok → оплата Pro на 12 мес тестовой картой → redirect назад → подписка активна, инвойс paid, карта •••• сохранена → refund → инвойс refunded, подписка откатилась.
4. **Лимиты:** на Старте 4-й сотрудник → 403 + модалка апгрейда; после оплаты Pro — создаётся.
5. **Фронт:** `cd front && npm run build && npm run lint` — чисто; F5 на странице тарифов — всё с сервера.
