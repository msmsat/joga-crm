# Эпик V2-6 — Мультистудийность: несколько студий, переключение

Цель эпика простыми словами: по ТЗ (§2.4, §14.10) пользователь может состоять в нескольких студиях — владеть двумя, или быть владельцем одной и тренером в другой — и переключаться между ними. V1 осознанно вырезал это как «дорогой multi-tenancy» и оставил UI одной студии на аккаунт. Изоляция данных по `studio_id` при этом сделана давно (эпик 1) — фундамент готов, осталась надстройка: активная студия, переключатель, создание второй CRM.

**Важно понимать про старт:**
- БД **уже готова**: `studio_members` — это m2m (пользователь ↔ студия ↔ роль), один человек может иметь несколько членств с разными ролями. Миграции не нужны.
- Ловушка известна с эпика 1 (задача 3): токен несёт студию/роль, только когда студия одна; `get_studio_context` писался с оглядкой на это. Теперь «несколько студий» становится штатным случаем.
- Переключатель UI уже свёрстан: `Settings/components/modals/WorkspaceSelector.tsx` (сейчас мок).
- Это последний эпик V2 — делать после стабилизации остального: он трогает контекст авторизации, то есть вообще всё.

**Обозначения сложности:** 🟢 простая · 🟡 средняя · 🔴 сложная, делать внимательно.

## Фаза 1 — Сервер

### 2. Активная студия в контексте запроса · 🔴 · 2:30
**Простыми словами:** сервер должен знать, «в какой студии» пользователь работает прямо сейчас. Решение: клиент шлёт заголовок `X-Studio-Id` с каждым запросом; `get_studio_context` проверяет членство в этой студии и берёт роль **этого** членства. Без заголовка — как раньше: единственная студия пользователя (обратная совместимость на время перехода).
**Зачем именно заголовок, а не токен:** переключение студии не требует перевыпуска токена; токен остаётся про личность, контекст — про рабочее место.
**Где:** `back/dependencies.py` (`get_studio_context`), фронтовый api-клиент (добавить заголовок из сохранённого выбора).
**Что конкретно сделать:**
1. `get_studio_context`: если пришёл `X-Studio-Id` — проверить членство (нет членства → 403), роль — из членства; если не пришёл — прежняя логика.
2. Все существующие гейты (`require_role`) автоматически получают правильную роль — проверить, что нигде роль не читается из токена напрямую в обход контекста.
3. Пейволл: проверка подписки — подписки активной студии.
**⚠️ Подводные камни:** владелец студии А, тренер в Б — в Б он ДОЛЖЕН получать 403 на owner-разделы. Это главный тест эпика.
**Готово, когда:** один и тот же пользователь в студии А — владелец (всё видит), в Б — тренер (только своё), переключение — сменой заголовка.

### 3. «Мои студии» + смена активной · 🟡 · 1:00
**Простыми словами:** эндпоинт списка студий пользователя — для переключателя: название, логотип, моя роль в каждой.
**Где:** `back/routers/auth/` или `studio/` — по месту (не новый роутер).
**Что конкретно сделать:**
1. `GET /me/studios`: id, название, логотип, роль членства; сортировка — сначала владение.
2. Выбор активной хранится на клиенте (localStorage) — серверного состояния «активная студия» не заводим (контекст и так в каждом запросе). ponytail: серверное поле «последняя студия» — только если попросят кросс-девайс.
**Готово, когда:** список приходит с ролями; фронту есть из чего строить переключатель.

### 4. Создание второй студии (онбординг повторно) · 🟡 · 1:30
**Простыми словами:** владелец создаёт ещё одну CRM: проходит тот же онбординг из 5 шагов, но уже будучи залогиненным. Новая студия = новое членство owner + свой триал/подписка.
**Где:** `back/routers/auth/onboarding` (или studio) — открыть повторный проход для существующего пользователя.
**Что конкретно сделать:**
1. Эндпоинт создания студии для авторизованного: те же 5 шагов данных; создаёт студию + членство owner + `free_trial` 14 дней (как при первой регистрации).
2. Убрать предположение «онбординг = новый пользователь» по списку из задачи 1.
3. Лимит здравого смысла: не больше N студий на владельца (например 10) — от злоупотребления триалами.
**Готово, когда:** пользователь с одной студией создаёт вторую и оказывается в ней владельцем с триалом; первая не затронута.

## Фаза 2 — Фронт

### 5. Переключатель студий (оживить WorkspaceSelector) · 🟡 · 1:30
**Простыми словами:** свёрстанный `WorkspaceSelector` подключить к `GET /me/studios`: список студий с ролями, галочка на активной, пункт «+ Создать студию». Точка входа — Настройки и/или клик по названию студии в сайдбаре (по ТЗ §1.1 логотип+название — в шапке меню).
**Где:** `front/src/pages/dashboard/Settings/components/modals/WorkspaceSelector.tsx`, сайдбар.
**Что конкретно сделать:**
1. Реальный список вместо мока; роль — бейджем («Владелец» / «Администратор» / «Тренер» через `t()`).
2. Выбор → сохранить id в localStorage → api-клиент шлёт `X-Studio-Id` (задача 2) → задача 6 перезагружает мир.
3. У пользователя с одной студией переключатель не мозолит глаза (виден, но скромно, или скрыт до второй студии — решить по месту).
**Готово, когда:** переключение меняет студию без повторного логина.

### 6. Смена студии = смена мира: сброс кэшей и роли · 🔴 · 2:00
**Простыми словами:** после переключения ничто из старой студии не должно остаться на экране: списки, счётчики, бейдж клиентов, роль (меню перестраивается — тренер видит меньше владельца), выбранные фильтры и «запомненные» id (например `staff_active_id` в localStorage указывает на сотрудника чужой студии).
**Зачем это отдельная сложная задача:** утечка данных между студиями на экране — это баг доверия, хуже падения.
**Где:** auth-контекст фронта, все localStorage-ключи с id (найти поиском), роутинг-гейты по роли.
**Что конкретно сделать:**
1. Простейший надёжный механизм: после смены — полная перезагрузка приложения (`location.assign('/dashboard')`). ponytail: изящная инвалидация кэшей без перезагрузки — только если перезагрузка окажется заметно неприятной.
2. localStorage-ключи с чужими id — чистить при смене (или скоупить ключи по studio_id).
3. Роль в новой студии другая → меню и страницы перестраиваются (владелец→тренер: Финансов больше нет).
**Готово, когда:** после переключения ни один элемент экрана не показывает данные прошлой студии; «назад» в браузере не воскрешает чужое.

### 7. «Создать ещё одну CRM» из Настроек · 🟡 · 1:00
**Простыми словами:** кнопка в переключателе/Настройках → онбординг-мастер (тот же 5-шаговый, он уже свёрстан) в режиме «вторая студия» → по завершении переключение в неё.
**Где:** онбординг-флоу фронта + WorkspaceSelector.
**Что конкретно сделать:** прокинуть режим «existing user»; после создания — активировать новую студию (задачи 5–6).
**Готово, когда:** владелец создаёт вторую студию, не выходя из аккаунта, и сразу в ней работает.

## Фаза 3 — Проверка

### 8. Прогон «две студии, две роли» + STATUS · 🟡 · 1:30
**Что конкретно сделать:**
1. Сценарий-максимум: пользователь — владелец студии А и тренер в студии Б (создан туда владельцем Б). В А видит всё; переключился в Б — только Журнал/Клиенты со своим скоупом; owner-эндпоинты Б по прямому запросу с `X-Studio-Id: Б` → 403.
2. Данные: клиенты/операции/занятия А не видны из Б ни в UI, ни по API.
3. Подписки независимы: триал Б не влияет на оплаченную А; пейволл срабатывает по активной.
4. `npm run build && npm run lint`; обновить `docs/TZ/STATUS.md» («мультистудийность» из «вне рамок» — в «работает») и ROADMAP.md.
**Готово, когда:** сценарий-максимум проходит; изоляция подтверждена прямыми запросами.

## Definition of Done эпика

- Пользователь состоит в нескольких студиях с разной ролью в каждой и переключается без перелогина.
- Роль и доступ всегда соответствуют активной студии — и в UI, и на сервере.
- Ни одной утечки данных между студиями (проверено прямыми запросами).
- Создание второй CRM — самообслуживание из Настроек.
- STATUS.md обновлён; сборка чистая.
