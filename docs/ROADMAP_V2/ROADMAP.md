V1 (эпики 1–5) выполнен. V2 строится из **аудитов страниц**: проходим раздел за разделом,
фиксируем что лжёт/не работает/дублируется, принимаем решения и превращаем их в задачи.
Отложенные эпики 6–7 лежат рядом (`EPIC_6_BACKLOG.md`, `EPIC_7_BACKLOG.md`) — вернёмся после аудитов.

При 6 продуктивных часах в день — **~17 рабочих дней (3,5 недели)** соло, без крупных находок аудитов (они оцениваются отдельно по факту). Оценки для разработчика, уверенно знающего стек.

**Порядок и логика:**
1. **V2-1** первым: чинит дыру в модели ролей и рождает общий Toast — фундамент для всего дальнейшего.
2. **V2-2** сразу за ним: раскатка UI-кита, пока паттерн свежий; дальше все эпики строят на общих компонентах.
3. **V2-3** — закрывает обещания V1 (Отчёты — последний «данные-раздел» на выдумке).
4. **V2-4** — серия аудитов; можно вести параллельно с V2-2/V2-3 по одной странице за сессию.
5. **V2-5** — мини-приложение: вторая половина продукта, стартует на стабилизированной CRM; отзывы из него разблокируют рейтинг/NPS в Отчётах.
6. **V2-6** — последним: трогает контекст авторизации, то есть вообще всё.

**Не пересекаемся с отложенными эпиками 6–7:** Velora AI (настоящий LLM), серверные вкладки Настроек, тема/полный i18n-добив, деплой/домен/бэкапы — не трогаем; аудит V2-4 эти зоны обходит.

**Вне рамок V2 (кандидаты дальше):** реальная отправка в Telegram/WhatsApp/SMS/Push; автосписание подписки по расписанию; серверные черновики Журнала; мероприятия/ивенты как сущность (вкладка «События» Отчётов); вкладки Финансов «Онлайн-платежи» и «Способы оплаты» (ждут контура клиентских оплат); ручной прогон Fondy на тестовом мерчанте (план готовится в V2-4, исполнение — с деплоем в эпике 7).

## Аудит 1: Сотрудники (Staff) — обсуждено 2026-07-13

### 1. Языки: страница целиком на выбранном языке

**Сейчас:** страница почти вся через `t()`, но карточки-метрики профиля («Выручка», «Рейтинг»,
«Записей», «Загрузка»…) — русский хардкод в `label` внутри
`front/src/pages/dashboard/Staff/role-capabilities.json`. При смене языка не переводятся.

**Решение:**
- `label` в JSON заменить на перевод по `id`: `t('staff:cards.<id>')` — секция `cards` в
  `staff.json` уже существует, докинуть недостающие ключи (ru + en).
- Пройти страницу на en и добить остальные пропуски, если найдутся (DeleteConfirmModal, тосты).

**Смежное (валюта):** `₽` захардкожен в `Staff.tsx` (`resolveCardValue`, строки ~186, 193) и в
модалках зарплаты («₽/ч»). Это противоречит решению V1 от 2026-07-12 (символ валюты студии, не `₽`).
Брать символ из настроек студии.

### 2. Залы: только правда

**Сейчас:** легенда под сеткой недели рисуется из `HALLS_CONFIG`
(`Staff/constants.ts:42` — «main / personal / massage», вымышленные названия и цвета).
При этом **реальные залы уже приходят с сервера**: `profile.halls` (id, name, color из БД —
залы, где сотрудник вёл занятия за 90 дней, `back/routers/staff/profiles.py:163`) и уже
используются в модалке «Добавить занятие». Ложь — только в легенде.

**Решение:**
- Легенду строить из `profile.halls` (реальные имена и цвета из Каталога).
- `HALLS_CONFIG` удалить.
- Сетка недели показывает рабочие часы — подпись легенды честная: «рабочее время / свободно»;
  реальные залы — отдельными чипами «Залы сотрудника».

### 3. Онлайн-статус: «я на месте»

**Сейчас:** `is_online` есть в БД (`users`) и показывается (точка в списке, статус в hero,
счётчик online в сводке) — но **никто её не ставит**: логин поле не трогает, тумблер в модалке
редактирования на сервер не отправляется. Всегда false → инфа ложная.

**Решение (обсудили): онлайн = «сегодня заходил в CRM».**
- В `users` добавить `last_online_at` (дата).
- В auth-зависимости (`get_studio_context`): если дата у пользователя не сегодняшняя — обновить.
  Одна запись в день на пользователя, никакого крона: в полночь статус «сам» устаревает.
- `is_online` в ответах API вычисляется на лету (`last_online_at == сегодня`), формат ответа
  для фронта не меняется; колонку `is_online` удалить миграцией.
- Тумблер «онлайн» из модалки редактирования убрать — статус теперь не назначается вручную.
- ponytail: сравнение по серверной дате; часовой пояс студии — после MVP.

### 4. Модалка редактирования: ромб в иллюстрации

**Сейчас:** в левой панели `front/src/components/modals/EditStaffModal.tsx:226–227` поверх
карточки-иллюстрации нарисован «бейдж редактирования»: `<path d="M197 41 L200 36 L205 41 L200 46 Z">` —
буквально ромб, выглядит как артефакт.

**Решение:** заменить на нормальную мини-иконку карандаша (или убрать бейдж совсем).

### 5. Роли и профессии: решение — только две

**Сейчас — хаос, который и чувствовался:**
- Три разных списка профессий: `Staff/constants.ts` (12: master_trainer, yoga, mfr…),
  `EditStaffModal.tsx:96` (8, включая **barber / stylist / cosmetologist / nail** — остатки
  шаблона салона красоты), свой список в `AddEmployeeModal`; в `staff.json` — 18 ключей ролей.
- Профессия отправляется в `StudioMember.role` — **то же поле, по которому `require_role`
  проверяет доступ**. Гейты знают только `owner` / `admin`; в двух местах бэка проверяется
  несуществующее `"teacher"` (`back/dependencies.py:96`, `back/routers/schedule/lessons.py:76`).
  Сотрудник с ролью «yoga» не проходит ни один гейт.
- Приглашение в Настройках шлёт роль русской строкой («Владелец», «Сотрудник») —
  `Settings/hooks/useTeam.ts:24`.

**Решено: две профессии = две роли — `trainer` и `admin`.** Профессия и роль доступа — одно
и то же. `owner` существует только как создатель студии, в выборе его нет. Новые профессии —
легко добавить потом, для MVP студии больше не надо.

**Задачи:**
- Фронт: в Add/Edit модалках — две карточки выбора (Тренер / Администратор); свободный ввод
  должности убрать; `PRESET_ROLES`, `ROLE_TO_DEPT_KEY`, `departments`, `presetServices` — выпилить;
  `staff.json` roles → trainer / admin / owner; группировка списка сотрудников — по роли.
- `role-capabilities.json` → ключи trainer / admin / owner / default (+ label через i18n, см. п.1).
- Бэк: `"teacher"` → `"trainer"` (2 места); миграция данных: `StudioMember.role` не из
  {owner, admin} → `trainer`; валидация role в схемах (Literal).
- Настройки: приглашение шлёт `admin` / `trainer`, не русские строки.
- `User.department` из обращения выводим (не шлём, ни на чём не завязано), колонку в БД не трогаем.

### 6–7. Услуги сотрудника: из каталога, не из выдумки

**Сейчас:** «услуги» в модалках — пилюли из локали (`presetServices` в `staff.json` по профессии)
плюс свободный текстовый ввод. На сервер **не отправляются вообще** — чистая косметика.
При этом связь в БД **уже есть**: `user_services` m2m (`back/models/user.py:49`) — просто не
подключена к API.

**Решено (и да — это правильный ход):**
- Услуги — выбор из реального каталога студии (`GET /services`), чекбоксы-пилюли.
- Для `admin` блок услуг скрыт — услуги только у тренера.
- Кнопка «Добавить услугу» ведёт в Каталог → Услуги. Второй CRUD услуг внутри модалки
  сотрудника не строим — единый источник правды, дубли услуг исключены.

**Задачи:**
- Бэк: `service_ids: list[int]` в `StaffCreate` / `StaffUpdate`; `services` в ответе профиля.
- Фронт: модалки грузят каталог; выбранные — по `services` профиля; `presetServices` удалить.

### 8. Ошибки: видимы всегда, тост один на всё приложение

**Сейчас:**
- **Четыре** реализации тостов: `Finances/components/ui/Toast.tsx`, `Settings/components/ui/Toast.tsx`,
  `Profile/components/ui/Toast.tsx` + inline-тост в `Staff.tsx:866`.
- Бэк на дубль email при создании честно отдаёт 409 с текстом (`profiles.py:267`) — фронт
  **молча глотает** (`Staff.tsx:813`: catch → закрыть модалку).
- `update_staff` вообще не проверяет дубль email → IntegrityError → **500** (`profiles.py:296`).
- Сохранение изменений на фронте без catch; валидация email — `includes("@")`.

**Решено:**
- Один глобальный тост: `front/src/components/ui/Toast.tsx` + `ToastProvider` / `useToast`
  (context), монтируется в DashboardLayout. Варианты success / error. Стиль — текущий премиум.
- Все ошибки API (409, 403, 422, 500…) → error-тост с `detail` от сервера; успехи → success-тост.
- Клиентская валидация до отправки: формат email, обязательные поля — подсветка + сообщение.
- Бэк: в `update_staff` проверка дубля email → 409 (как в create).
- Внедряем **сейчас только в Сотрудниках**; дальше — поэтапно:

**Очередь унификации UI (следующие итерации V2):**
1. Миграция Finances / Settings / Profile на общий тост (удалить 3 дубля).
2. Единый ConfirmModal (удалить/подтвердить) на всё приложение.
3. Единый выпадающий список (Select) — язык, время и прочая мелочь: один стиль везде.
