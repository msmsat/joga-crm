# Эпик V2-5 — Клиентское мини-приложение (ТЗ §17)

Цель эпика простыми словами: у CRM появляется **вторая половина продукта** — приложение для клиентов студии. Клиент попадает по ссылке из Telegram/Instagram/с сайта, записывается на занятие за три шага без регистрации, а в личном кабинете видит свои записи, абонемент, баллы и историю — и оставляет отзывы. Это то, ради чего владелец студии настраивал «Онлайн-запись» и брендинг.

По ТЗ это «вторая очередь — после MVP CRM»; V1 ROADMAP явно отправил его в v2. CRM-часть стабилизирована эпиками V2-1…V2-4 — теперь очередь клиентской.

**Важно понимать про старт:**
- **Публичный API уже работает** (эпик 4): каталог услуг, свободные слоты с учётом правил записи, бронирование по имени+телефону (find-or-create клиента, `booking_channel=web`, письмо-подтверждение, событие в ленте), rate-limit 5 броней/мин. UI к нему не существует — «только API под него» (STATUS).
- Брендинг (цвет, логотип, язык) и правила записи уже сохраняются в разделе «Онлайн-запись» — мини-приложение их потребляет, не изобретает.
- Новая сущность в этом эпике одна — **отзывы** (модели нет нигде); они же разблокируют «Рейтинг» и NPS в Отчётах (честные заглушки из V2-3).
- Механизм входа клиента в кабинет — ключевое решение эпика (задача 8): SMS у нас нет, есть email.

**Обозначения сложности:** 🟢 простая · 🟡 средняя · 🔴 сложная, делать внимательно.

## Фаза 0 — Решения

### 1. Архитектура: где живёт мини-приложение · 🟡 · 1:00
**Простыми словами:** решить, как мини-приложение существует технически, до первой строчки кода. Рекомендация — **публичные роуты в существующем фронте** (`/s/:studioSlug/*`): один билд, общие ui-компоненты, деплой без второго приложения. Отдельное приложение заводим только если решим иначе с владельцем.
**Зачем:** ошибка здесь = переезд всего эпика.
**Что конкретно сделать:**
1. Зафиксировать: роуты `/s/:slug` вне DashboardLayout (без сайдбара, без auth-гейта CRM), мобайл-первая вёрстка.
2. Слаг студии: проверить, есть ли у студии публичный идентификатор (slug/id в настройках Онлайн-записи) — если нет, добавить (миграция + поле в настройках).
3. Решение записать в ROADMAP.md (как решение по валюте в V1).
**Готово, когда:** решение записано; slug существует; пустой роут `/s/:slug` открывается без авторизации.

## Фаза 1 — Скелет и брендинг

### 2. Публичная страница студии по ссылке · 🟡 · 2:00
**Простыми словами:** главная страница студии для клиента: название, логотип, адрес, часы работы — и кнопка «Записаться». Всё с публичного API (данные уже отдаются каталогом/настройками — проверить, что есть публичный эндпоинт «инфо о студии», при отсутствии добавить).
**Где:** новый `front/src/pages/public/` (структура по правилам §5 CLAUDE.md), `back/routers/booking/` (публичный роутер — расширить при необходимости).
**Что конкретно сделать:**
1. Публичный `GET /public/{slug}/info`: название, логотип, адрес, часы, брендинг — без чувствительного (без email владельца и т.п.).
2. Страница: шапка со студией, кнопка «Записаться» → поток записи, ссылка «Мои записи» → кабинет.
3. Несуществующий slug → человеческая страница «Студия не найдена».
4. Rate-limit чтения (30/мин с IP уже есть — распространить на новый эндпоинт).
**Готово, когда:** страница студии открывается по ссылке в режиме инкогнито и показывает реальные данные студии.

### 3. Брендинг: цвет, логотип, язык студии · 🟡 · 1:30
**Простыми словами:** владелец в «Онлайн-записи» выбрал фирменный цвет и язык — мини-приложение обязано их применить: акцентный цвет кнопок, логотип, весь интерфейс на языке студии. Превью в CRM наконец встречается с реальностью.
**Где:** тема через CSS-переменные в корне публичного лейаута; язык — i18n с namespace `public` (ru + en).
**Что конкретно сделать:**
1. Брендинг из `/info` → CSS-переменные (акцент, его hover/soft-варианты — вычислять, не хранить три цвета).
2. Язык интерфейса = язык из настроек мини-приложения (не язык браузера — воля студии первична; фолбэк ru).
3. Все строки мини-приложения — через `t()` с первого дня (урок аудита 1).
**Готово, когда:** смена цвета/языка в CRM меняет мини-приложение после обновления страницы; превью в «Онлайн-записи» совпадает с реальностью.

## Фаза 2 — Быстрая запись без входа (ТЗ §17.1)

### 4. Шаг 1: выбор услуги · 🟡 · 1:30
**Простыми словами:** список услуг студии: название, длительность, цена, свободные места — по клику переход к выбору времени. Источник — публичный каталог услуг (уже работает).
**Что конкретно сделать:**
1. Карточки услуг с публичного API; цена — в валюте студии.
2. Пустой каталог → «Студия ещё не добавила услуги» (честно, не вечный скелетон).
3. Прогресс-индикатор шагов (1 → 2 → 3) — поток из превью «Онлайн-записи».
**Готово, когда:** услуги реальные; выбор ведёт на шаг 2 с выбранной услугой.

### 5. Шаг 2: дата и время из слотов · 🟡 · 2:00
**Простыми словами:** календарная полоса дней (в пределах горизонта записи из правил студии) и сетка свободных слотов по выбранной услуге. Слоты — с публичного API, который уже уважает правила записи.
**Что конкретно сделать:**
1. Лента дней: сегодня + N дней (N — из правил); дни без слотов видны, но приглушены.
2. Слоты по дню: время + свободные места; занятые не показываем.
3. Обновление при возврате на шаг (слот могли занять); день без слотов → «На этот день записи нет».
**Готово, когда:** слоты совпадают с Журналом CRM; правило «запись закрывается за X часов» реально прячет ближние слоты.

### 6. Шаг 3: контакты + «Запись подтверждена» · 🟡 · 1:30
**Простыми словами:** имя + телефон (+ email по желанию — на него придёт подтверждение), кнопка «Записаться» → экран «Запись подтверждена» со сводкой: услуга, дата/время, стоимость. Бэкенд-бронирование уже работает (find-or-create клиента, письмо, событие в ленте).
**Что конкретно сделать:**
1. Форма с валидацией до отправки (телефон — маска, email — regexp; уроки V2-1 задачи 9).
2. Экран подтверждения по ТЗ; кнопка «Записаться ещё».
3. Ошибки API видимы: слот заняли, rate-limit, «запись закрыта» → человеческий текст, не тишина (общий Toast из V2-1 переиспользовать в публичном лейауте).
**Готово, когда:** сквозной путь «ссылка → услуга → слот → подтверждение» проходит с телефона; бронь видна в Журнале CRM; клиент создан без дублей.

### 7. Границы записи: правила студии в UI · 🟡 · 1:30
**Простыми словами:** правила («за сколько часов закрывается запись», «на сколько дней вперёд», «до какого времени можно отменить») сервер уже применяет — но клиент должен их **видеть**, а не натыкаться: ближние слоты не показываются, у записи в кабинете подписано «отменить можно до 20:00».
**Что конкретно сделать:**
1. Правила приходят в `/info`; UI их отображает (подписи, приглушённые состояния).
2. Прогнать граничные случаи против сервера: слот ровно на границе часа закрытия, последний день горизонта, отмена за минуту до дедлайна — UI и сервер согласны.
3. Ошибка «правило не пустило» от сервера — понятный текст на языке студии.
**Готово, когда:** клиент не может даже попытаться сделать то, что правила запрещают, и понимает почему.

## Фаза 3 — Личный кабинет клиента (ТЗ §17.2)

### 8. Вход клиента: код на email · 🔴 · 2:30
**Простыми словами:** кабинет требует узнать клиента. SMS-канала у нас нет; у клиентов есть email (собирается при записи и в CRM). Решение: вход по одноразовому коду на email — тот же механизм, что подтверждение регистрации владельца (mailer готов). Клиент вводит email → получает 6-значный код → сессия клиента.
**Зачем это отдельная сложная задача:** это новый контур авторизации — **клиентский**, полностью отдельный от JWT сотрудников CRM: свой токен, свои права (только свои данные своей студии), без доступа к CRM-эндпоинтам.
**Где:** `back/routers/booking/` (публичный контур) — новые эндпоинты `POST /public/{slug}/auth/code` и `/auth/verify`; хранение кода — как у email-верификации.
**Что конкретно сделать:**
1. Код: 6 цифр, TTL 10 минут, не больше 3 отправок в час на email (rate-limit).
2. Верификация → клиентский JWT с `client_id` + `studio_id`; scope проверяется на каждом клиентском эндпоинте.
3. Email, которого нет среди клиентов студии → «отправили код, если вы записывались» (не раскрывать базу).
4. ponytail: без паролей, без refresh-токенов — короткоживущий токен (7 дней), протухла — новый код.
**Готово, когда:** клиент входит по коду и получает только свои данные; токен клиента на CRM-эндпоинтах — 401; чужая студия — 404.

### 9. Мои записи + отмена по правилам · 🟡 · 2:00
**Простыми словами:** список записей клиента: предстоящие и прошедшие; предстоящую можно отменить, пока правило студии позволяет — с подписью, до какого времени.
**Где:** бэк — клиентские эндпоинты списка/отмены (отмена переиспользует логику отмены брони + правило дедлайна); фронт — вкладка кабинета.
**Что конкретно сделать:**
1. `GET` записей клиента (предстоящие/прошедшие, по дате); отмена — с проверкой дедлайна на сервере (UI-подпись — из задачи 7).
2. Отмена возвращает занятие в свободные места, событие в ленту студии, письмо-подтверждение отмены (через матрицу Уведомлений).
3. Запись «из кабинета» (кнопка «Записаться ещё») — тот же поток фазы 2, но имя/телефон уже подставлены.
**Готово, когда:** отмена клиентом видна в Журнале CRM мгновенно; после дедлайна кнопки отмены нет, и сервер тоже не даёт.

### 10. Абонемент, баллы, история визитов · 🟡 · 2:00
**Простыми словами:** три информационные секции кабинета: мой абонемент (тип, остаток занятий, срок), карта лояльности (баллы, уровень), история посещений. Все данные уже живут в CRM (абонементы, баллы, визиты) — отдать их клиенту про себя.
**Где:** бэк — клиентский `GET /me` (профиль + абонемент + баллы + уровень), `GET /me/visits`; фронт — секции кабинета.
**Что конкретно сделать:**
1. Абонемент: остаток/срок; заморозка (если заморожен — показать честно); нет абонемента → блок-приглашение «доступны в студии» (продажа — в CRM, не в мини-приложении).
2. Баллы и уровень — из программ Лояльности (если программа карт выключена студией — блок скрыт).
3. История визитов — постранично, по дате; будущие записи сюда не попадают (они в «Моих записях»).
**Готово, когда:** цифры кабинета совпадают с карточкой клиента в CRM.

## Фаза 4 — Отзывы (новая сущность)

### 11. Модель отзывов + API · 🟡 · 2:00
**Простыми словами:** отзывов в продукте нет нигде — заводим: клиент оценивает посещённое занятие (1–5 звёзд + текст). Именно отзывы разблокируют «Рейтинг тренера» и NPS-виджеты Отчётов, которые в V2-3 остались честными заглушками.
**Где:** модель `Review` — в существующий файл моделей (client или events — по месту, **не новый файл**: клиент, занятие, оценка, текст, дата; уникальность — один отзыв на посещение); клиентский `POST`, CRM-шный `GET`.
**Что конкретно сделать:**
1. Миграция: `reviews (id, studio_id, client_id, lesson_id, rating 1–5, text nullable, created_at)`, unique (client_id, lesson_id).
2. `POST` — только за занятие со статусом «посетил» у этого клиента (иначе 403); правка своего отзыва — разрешена (upsert).
3. `GET` для CRM: по тренеру и по студии, с агрегатом (средняя, количество) — под Отчёты и профиль сотрудника.
**Готово, когда:** отзыв нельзя оставить за чужое или непосещённое занятие; агрегаты считаются.

### 12. Отзывы в кабинете + рейтинг в CRM · 🟡 · 2:00
**Простыми словами:** в истории посещений у занятия появляется «Оценить» (звёзды + текст); в CRM рейтинг оживает: карточка «Рейтинг» в профиле сотрудника (`avg_rating` наконец получает источник), виджеты Отчётов «Рейтинг» и NPS переключаются с заглушек на данные.
**Где:** фронт кабинета (история визитов), `back/routers/staff/profiles.py` (`avg_rating` — считать из отзывов), Отчёты (виджеты из V2-3 задачи 6).
**Что конкретно сделать:**
1. UI оценки в истории; свой отзыв виден и правится.
2. `avg_rating` профиля сотрудника — среднее по отзывам его занятий (пересчёт на лету при чтении; кэшировать — только если станет медленно).
3. В Отчётах убрать заглушки «появится после первых отзывов» — подключить агрегат; NPS считать по стандартной формуле от оценок (5 = промоутер, 4 = нейтрал, 1–3 = критик) — зафиксировать формулу в задаче.
**Готово, когда:** отзыв клиента меняет рейтинг тренера в CRM; Отчёты показывают реальный рейтинг.

## Фаза 5 — Проверка

### 13. Сквозной прогон «клиент с нуля» + STATUS · 🟡 · 1:30
**Что конкретно сделать:**
1. С телефона (или эмуляции): ссылка → запись без входа → письмо → вход по коду → отмена → повторная запись → визит (отметка в CRM) → списание с абонемента → отзыв → рейтинг в CRM.
2. Изоляция: клиент студии А по своему токену не достаёт данные студии Б; клиентский токен не открывает CRM.
3. Rate-limit'ы: перебор кода, спам бронями.
4. `npm run build && npm run lint`; обновить `docs/TZ/STATUS.md` (раздел «Клиентское мини-приложение» переезжает в «работает») и ROADMAP.md.
**Готово, когда:** полный путь клиента проходит без участия разработчика.

## Definition of Done эпика

- Клиент записывается по публичной ссылке за 3 шага без регистрации; бронь мгновенно видна в CRM.
- Кабинет: вход по email-коду; записи/отмена по правилам, абонемент, баллы, история — совпадают с CRM.
- Отзывы работают и питают рейтинг тренера + виджеты Отчётов.
- Брендинг студии применён; всё на языке студии; изоляция данных проверена прямыми запросами.
- STATUS.md обновлён; сборка чистая.
