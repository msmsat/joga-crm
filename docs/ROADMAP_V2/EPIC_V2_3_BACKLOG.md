# Эпик V2-3 — Хвосты V1: Отчёты на правде, доступ тренера, Google-вход

Цель эпика простыми словами: закрыть три несделанных обещания V1. **Отчёты** — единственный «данные-раздел», который до сих пор показывает выдумку, хотя серверные агрегаты уже готовы и питают Дашборд. **Тренер** по ТЗ видит своих клиентов — сейчас ему раздел закрыт целиком. **Google-вход** работает только для владельца — админ и тренер через Google не войдут.

Источник: `docs/TZ/STATUS.md`, раздел «Хвосты ROADMAP_V1» (пп. 1, 3, 4) + ТЗ разделы 5, 11.

**Важно понимать про старт:**
- Бэкенд для Отчётов **в основном готов**: `/analytics/summary`, `/series`, `/trainers`, `/services` уже питают Дашборд. Работа по Отчётам — по большей части фронтовая: подключить вкладки к API вместо заготовленных чисел.
- Но не для всех виджетов ТЗ данные существуют в принципе: NPS и «Рейтинг» требуют отзывов (появятся с мини-приложением, эпик V2-5), вкладка «События» — модели мероприятий (нет нигде). Врать нельзя — для таких виджетов отдельная задача-решение.
- Связь «клиент → тренер» уже есть в данных: через занятия и записи (`lessons.teacher_id` + `reservations.client_id`). Модель менять не нужно.

**Обозначения сложности:** 🟢 простая · 🟡 средняя · 🔴 сложная, делать внимательно.

## Фаза 1 — Отчёты: подключить фронт к правде

### 2. Недостающие агрегаты на сервере · 🟡 · 2:00
**Простыми словами:** дашбордные эндпоинты покрывают не всё, что просит страница Отчётов. По карте из задачи 1 добавить недостающее — ожидаемо: разбивка выручки по категориям операций (структура доходов), разбивка по способам оплаты, средний чек по дням недели, удержание (вернулись 2-й раз / регулярные / отток), загрузка по дням за 2 недели.
**Зачем:** фронт не должен ничего досчитывать из сырых списков — агрегаты считает сервер (на больших данных это единственный вариант).
**Где:** `back/routers/analytics/` (расширить существующий роутер, не плодить новый), период — теми же параметрами, что у `/summary` (день/неделя/месяц/год + диапазон).
**Что конкретно сделать:**
1. По карте задачи 1 — добавить эндпоинты/поля к существующим ответам (меньше эндпоинтов лучше: расширять `/summary` и `/series`, где влезает по смыслу).
2. Все запросы — со `studio_id`-скоупом и `require_role("owner")`, как у соседей.
3. Индексы: проверить, что по `operations(studio_id, created_at)` и `reservations(status)` запросы не сканируют всё.
**Готово, когда:** каждый виджет с пометкой «надо добавить» имеет свой источник; ответы проверены руками на тестовой студии.

### 3. Вкладка «Основные» на реальных данных · 🟡 · 2:00
**Простыми словами:** главная вкладка Отчётов: 5 метрик с трендами, график выручки, удержание, загрузка за 2 недели, структура доходов, показатели качества — всё это сейчас заготовленные числа с множителями. Подключаем к API; переключение периода (День/Неделя/Месяц/Год) перезапрашивает данные.
**Где:** `front/src/pages/dashboard/Reports/**` (вкладка «Основные»), образец подключения — Дашборд (`Overview`), он уже ходит в те же эндпоинты.
**Что конкретно сделать:**
1. Снести мок-константы вкладки; типы — по ответам API (`types.ts` строго по Pydantic-схемам, CLAUDE.md §8).
2. Метрики + тренды — `/analytics/summary`; график — `/series`; удержание и загрузка — из задачи 2; структура доходов — из задачи 2.
3. NPS и «Рейтинг» в «Показателях качества» — см. задачу 6 (честная заглушка).
4. Состояния: загрузка (скелетоны), пусто (новая студия без данных — нули, не NaN), ошибка (тост).
**Готово, когда:** цифры вкладки совпадают с Дашбордом на одном периоде; у новой студии — нули без падений.

### 4. Вкладка «По продажам» на реальных данных · 🟡 · 1:30
**Простыми словами:** выручка, транзакции, средний чек, продажи по продуктам (абонементы/разовые/сертификаты), способы оплаты, средний чек по дням — всё из операций Финансов.
**Где:** та же страница, вкладка «По продажам»; данные — задача 2 + существующие операции.
**Что конкретно сделать:**
1. «Общая выручка» и «vs прошлый %» — из summary с датами периода.
2. «Продажи по продуктам» — разбивка доходных операций по категориям; бейджи «ТОП»/«РОСТ» — по месту (лидер по сумме / лучший прирост к прошлому периоду); раскрытие позиции: способ оплаты — из операций; «Покупатели: постоянные/новые» — если считать нечем, скрыть под-блок (не выдумывать).
3. «Средний чек по дням» Пн–Вс и «Способы оплаты» — из задачи 2.
**Готово, когда:** каждая цифра вкладки выводится из операций; продажа абонемента из карточки клиента меняет цифры после обновления.

### 5. Вкладки «По тренерам» и «По услугам» · 🟡 · 1:30
**Простыми словами:** обе вкладки уже имеют готовые серверные источники — `/analytics/trainers` и `/analytics/services` питают сводки Дашборда. Подключить карточки, детализацию и таблицу.
**Где:** вкладки «По тренерам», «По услугам»; API готов.
**Что конкретно сделать:**
1. Карточки тренеров (выручка, занятия) и сравнение по выручке — `/trainers`; «Рейтинг ★» — честная заглушка до V2-5 (задача 6); KPI-полосы — что считается из данных (заполняемость групп, отмены) показываем, «выполнение плана» без модели планов — скрыть.
2. Карточки и таблица услуг (занятий, выручка, средний чек, доля) — `/services`; тренд-бейдж — сравнение с прошлым периодом.
3. График «Загрузка по дням (7 недель)» — из занятий тренера (расширение `/trainers/{id}` если нужно — согласовать с задачей 2).
**Готово, когда:** выбор тренера показывает его реальные занятия и выручку; суммы по услугам сходятся с Финансами.

### 6. Вкладка «Все» + честные заглушки (NPS, События) · 🟡 · 1:30
**Простыми словами:** вкладка «Все» — три сводные карточки (Клиенты/Занятия/Финансы) и лента активности: всё есть в данных (лента событий студии уже реальна на Дашборде). Отдельно — принципиальное решение по виджетам без данных: NPS, «Рейтинг по отзывам», вкладка «События» (мероприятий как сущности в продукте нет).
**Зачем:** «пусть будет правда» — правило из аудита 1 распространяется на Отчёты: виджет либо показывает данные, либо честно говорит, что данных пока нет.
**Что конкретно сделать:**
1. Карточки «Все» — summary + операции; лента — тот же источник, что у Дашборда, с фильтром периода.
2. NPS / Рейтинг: состояние «Появится после первых отзывов» (отзывы — эпик V2-5), не рисовать случайное число.
3. Вкладка «События»: скрыть до появления сущности мероприятий; завести строку в «Вне рамок» ROADMAP_V2 («Мероприятия/ивенты — отдельная фича»).
**Готово, когда:** на странице Отчётов нет ни одного числа, не выводимого из базы.

**Где:** существующий экспорт-код страницы Отчётов.
**Что конкретно сделать:** источником экспорта сделать текущее состояние загруженных данных (тот же стейт, что рендерит вкладки); период и дата — в имени файла.

**Простыми словами:** у Финансов та же болезнь на одной вкладке — «Отчёты» с заготовленными сводками. Подключить к тем же агрегатам (это упрощённое зеркало «Основных»).

## Фаза 2 — Тренер видит своих клиентов (ТЗ §5, сводка ролей)

### 9. Сервер: клиенты тренера — только свои · 🔴 · 2:00
**Простыми словами:** по ТЗ тренер видит только клиентов, которые ходят к нему. Сейчас проще: тренеру раздел закрыт целиком (403). «Свой клиент» = есть запись (reservation, не отменённая) на занятие этого тренера. Открываем список и карточку с этим фильтром.
**Зачем:** ТЗ; тренеру реально нужно смотреть контакты и историю своих людей.
**Где:** `back/routers/clients/**` — гейт с `require_role("owner", "admin")` расширить до `trainer` + скоуп-фильтр; образец скоупа — `get_scoped_lesson` в `back/dependencies.py`.
**Что конкретно сделать:**
1. Список: для роли `trainer` — join через `reservations`→`lessons.teacher_id == ctx.user.id`, distinct; счётчики категорий — по той же выборке.
2. Карточка: тот же критерий; чужой клиент → 404 (не 403 — не раскрывать существование).
3. Действия тренера над клиентом — только чтение + заметки (правку статуса/тегов/абонемента оставить owner/admin — зафиксировать выбор в задаче явно).
4. ⚠️ Пагинация и поиск обязаны работать с фильтром (contract `Page` не ломать).
**Готово, когда:** тренер видит ровно тех, кто записывался к нему; прямой запрос чужого клиента по id — 404; owner/admin видят всё как раньше.

### 10. Фронт: раздел Клиенты для тренера · 🟡 · 1:30
**Простыми словами:** показать тренеру раздел Клиенты (сейчас пункт меню скрыт): та же таблица, но без кнопок, которых ему нельзя.
**Где:** сайдбар (условие скрытия пункта), страница Клиентов (роль-зависимые кнопки: добавить/удалить/сменить статус).
**Что конкретно сделать:**
1. Пункт меню «Клиенты» виден тренеру; страница работает с его выборкой (сервер уже фильтрует).
2. Скрыть тренеру: добавление, удаление, смену статуса, продажу абонемента — по решению из задачи 9.3; «Позвонить»/«Написать»/заметки — оставить.
3. Категории-счётчики — от его выборки.
**Готово, когда:** тренер полноценно работает со своими клиентами и не видит недоступных действий.

## Фаза 3 — Вход

### 11. Google-вход для админа и тренера · 🟡 · 1:00
**Простыми словами:** Google-логин сейчас ищет только владельца — сотрудник с Google-аккаунтом на том же email получает отказ, хотя по email/паролю входит. Поправить поиск: любой участник студии.
**Где:** `back/routers/auth/` (обработчик Google OAuth), логика поиска пользователя.
**Что конкретно сделать:**
1. После проверки Google-токена искать пользователя по email среди всех (как в обычном логине), а не только среди владельцев; membership и роль — в контекст как при обычном входе.
2. Сотрудник без Google-привязки, входящий по Google впервые с совпадающим email — связать аккаунты (та же логика, что у владельца).
3. ⚠️ Сотрудник не должен попадать в онбординг владельца (грабли из эпика 1, задача 4b, — проверить и здесь).
**Готово, когда:** созданный владельцем тренер входит через Google на том же email и попадает в свою студию со своей ролью.
