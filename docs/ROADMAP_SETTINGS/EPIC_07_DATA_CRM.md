**Цель:** вкладка «Данные» отдаёт настоящие выгрузки вместо анимации
загрузки; «Сменить CRM» превращается из мока в рабочее переключение
между студиями пользователя, с созданием новой через онбординг.

**Зависимости:** эпик 1, **эпик 5** (удаление аккаунта → редирект на
выбор CRM), эпик 4 (общий CSV-генератор). **Оценка: ~7:30.**

## Точка отсчёта

**Вкладка «Данные».** `DataTab.tsx` (177 строк) — целиком имитация:
`setTimeout(2000)` → тост «Резервная копия создана»,
`setTimeout(1500)` → «Файл backup_joga_live.sql скачан», четыре карточки
экспорта с захардкоженными размерами («4.2 МБ», «12.8 МБ») и
`setTimeout(1800)` вместо запроса. Цифры «2.4 ГБ / 10 ГБ» и полоска
заполнения — константы прямо в JSX.

Модель `StudioBackupSettings` под всё это существует, но никогда не
читается — **удаляется в эпике 1** вместе с блоком «Хранилище данных».
По ТЗ от раздела остаётся только экспорт.

> Почему бэкапы уезжают, а не реализуются: резервное копирование — это
> задача инфраструктуры (`pg_dump` по расписанию на стороне хостинга),
> а не кнопки в SaaS-интерфейсе. Кнопка «сделать бэкап», дергающая
> `pg_dump` из приложения, — это выдача пользователю дампа чужих данных
> при первой же ошибке в скоупинге. Бэкапы остаются в эпике деплоя
> (`docs/BACKLOG/`), в Настройках — только экспорт **своих** данных.

**«Сменить CRM».** Кнопка внизу левой навигации (`Settings.tsx:186`)
ставит `isLoggedOut = true` и открывает `WorkspaceSelector` со списком
`INITIAL_STUDIOS_LIST` — двумя выдуманными студиями («Pilates & Wellness
Studio», «Barbershop Blade & Co»). Выбор показывает тост и закрывает
модалку. Ничего не переключается.

**А на бэке нужное уже есть:**

```python
# back/routers/auth/onboarding.py:97
@router.post("/select-studio", response_model=TokenResponse)
```

Ручка проверяет членство в `StudioMember` и **выдаёт новый JWT с нужными
`studio_id`/`role`**. Это и есть переключение CRM — не хватает только
списка студий и фронта.

**Единственный настоящий блокер** — там же, строкой выше:

```python
if current_user.is_onboarded:
    raise HTTPException(400, "Онбординг уже пройден")
```

Создать **вторую** студию сейчас невозможно by design.

## Терминология (зафиксировано)

ТЗ говорит «CRM», код говорит `Studio`. В продукте «сменить CRM» = сменить
рабочее пространство (студию), «добавить CRM» = создать новую студию и
пройти по ней онбординг. **Справочника внешних CRM-систем в БД нет и не
заводится** — «список доступных CRM из БД» реализуется как список студий,
где пользователь состоит участником (`StudioMember`). Мультистудийность
числится в хвостах `ROADMAP_V1` («вне рамок V1») — этот эпик её закрывает.

## User Stories

- **Как владелец** я выгружаю список клиентов в CSV и открываю его в
  Excel — файл настоящий, с моими данными.
- **Как владелец** я вижу список своих студий и переключаюсь между ними
  одним кликом, без повторного логина.
- **Как владелец** я создаю вторую студию и прохожу по ней онбординг
  заново.
- **Как владелец**, удалив компанию, я попадаю на выбор оставшихся
  студий, а не на белый экран.

## Часть A. Раздел «Данные» — реальный экспорт

### Задача 1. Бэк: экспорт-эндпоинты (~1:45)

**Слой:** `back/routers/settings/data.py` (новый, в `settings/router.py`),
генератор — `back/services/exporter.py` **из эпика 4** (пишется там,
здесь переиспользуется; второй CSV-writer не заводим).

```
GET /settings/data/export?kind={kind}&date_from=&date_to=
→ 200 text/csv;charset=utf-8, Content-Disposition: attachment
```

| `kind` | Что выгружается | Источник |
|---|---|---|
| `clients` | ФИО, телефон, email, дата рождения, статус, теги, визиты, сумма покупок, дата регистрации | `Client` |
| `schedule` | дата, занятие, тренер, зал, мест, записано, посещено, статус | `Lesson` + `Reservation` |
| `finances` | дата, тип, категория, сумма, метод, счёт, клиент, статус | `Operation` |
| `subscriptions` | клиент, пакет, остаток, срок, статус | `ClientSubscription` |

**Обязательное:**
- `require_role("owner")` + `WHERE studio_id == ctx.studio_id` в каждом
  запросе. Экспорт — самая привлекательная ручка для утечки: она по
  определению отдаёт всё сразу.
- `kind: Literal[...]` — не подстановка строки в имя таблицы (SQL-инъекция
  через `kind` была бы учебной).
- Заголовки колонок — на языке студии (`_studio_prefs`), суммы — в
  валюте студии, символ в заголовке колонки.
- Диапазон дат опционален; без него — всё, но с `LIMIT` по здравому
  смыслу (100 000 строк) и понятной ошибкой при превышении.
- **`GET /settings/data/export/estimate?kind=…` → `{"rows": 412}`** —
  чтобы UI показывал реальное «412 записей» вместо выдуманных «4.2 МБ».

**Приватность.** В `clients` попадают персональные данные. Логируем факт
выгрузки (`ActivityLog` уже есть в моделях): кто, что, когда. При
разборе утечки это единственный способ понять, что произошло.

## Часть B. «Сменить CRM» — мультистудийность

### Задача 3. Бэк: список и создание студий (~1:30)

**Слой:** `back/routers/auth/onboarding.py` (рядом с существующим
`select-studio`).

```
GET /auth/studios → 200
[{"id": 1, "name": "Pilates & Wellness", "role": "owner",
  "logo_url": "/uploads/…", "is_current": true,
  "members_count": 4, "clients_count": 412}]
```

`POST /auth/select-studio` — **уже существует**, менять не нужно;
возвращает новый JWT с выбранной студией.

**Создание второй студии** — снять блокировку `is_onboarded`, не ломая
первичный сценарий:

```python
@router.post("/studios", response_model=TokenResponse)     # NEW
async def create_studio(request: OnboardingRequest, ...):
    """Дополнительная студия для уже онбординженного пользователя.
    Тело и валидация — те же, что у /onboarding; переиспользуем
    _create_studio_with_defaults(), выделенную из complete_onboarding."""
```

Рефакторинг: тело `complete_onboarding` (создание `Studio`,
`StudioMember(role='owner')`, `StudioWorkingHours`,
`StudioNotificationSettings`, `StudioBookingSettings`) выносится в
`_create_studio_with_defaults(user, data, db)`, обе ручки её зовут.
`/onboarding` дополнительно ставит `is_onboarded = True`.

**Edge case, который нельзя пропустить:** валидация телефона в
`complete_onboarding` (строка ~36) требует **глобальной уникальности
`User.phone`** — при создании второй студии тем же пользователем она
свалится на его собственном номере. В `/studios` эта проверка не
переносится (телефон уже принадлежит этому же пользователю).

**Лимиты.** Новая студия = новая подписка. `POST /auth/studios` создаёт
студию **без** `StudioBillingPlan` → пейволл (`_sub_gate` в `main.py`)
сам приведёт владельца на оплату. Отдельной логики не нужно — гейт уже
написан и работает.

### Задача 4. Фронт: страница выбора CRM (~1:45)

**Роут:** `/select-crm` — **вне** `DashboardLayout` (студия ещё не
выбрана — сайдбар и пейволл рисовать не на чем), но за `PrivateRoute`.

**Файлы:**
- `front/src/pages/SelectCrm.tsx` (новый)
- `front/src/App.tsx` (+ роут)
- удалить `Settings/components/modals/WorkspaceSelector.tsx`,
  `INITIAL_STUDIOS_LIST` из `constants.ts`, тип `Studio` из `types.ts`,
  стейт `isLoggedOut`/`studiosList` из `Settings.tsx`

```tsx
const { data: studios } = useQuery({ queryKey: queryKeys.workspaces,
                                     queryFn: authApi.getStudios });
const select = useMutation({
  mutationFn: authApi.selectStudio,
  onSuccess: ({ access_token }) => {
    localStorage.setItem('token', access_token);   // новый JWT со studio_id
    qc.clear();                                    // ← КРИТИЧНО, см. ниже
    navigate('/dashboard');
  },
});
```

> **`qc.clear()` обязателен.** Кэш Query набит данными предыдущей студии
> (клиенты, занятия, финансы). Без сброса пользователь после
> переключения увидит чужие данные до первого рефетча — визуально это
> неотличимо от утечки между студиями.

Карточка студии: логотип/инициалы, название, роль, счётчики,
бейдж «Текущая». Внизу — «Создать новую студию» → `/onboarding?new=1`.

**Кнопка в Настройках** («Сменить CRM», низ левой навигации) →
`navigate('/select-crm')`. Модального `WorkspaceSelector` больше нет:
смена рабочего пространства — это выход из контекста студии, ей нужен
полный экран, а не модалка поверх настроек этой же студии.

### Задача 5. Повторный онбординг для новой студии (~1:15)

**Триггер — query-параметр, не глобальный флаг.** `/onboarding?new=1`
переиспользует существующий 5-шаговый мастер, меняя только конечный
вызов:

```ts
const isNew = searchParams.get('new') === '1';
const submit = isNew ? authApi.createStudio : authApi.completeOnboarding;
// оба → TokenResponse; onSuccess: сохранить токен, qc.clear(), /dashboard
```

**Почему не «сбросить `is_onboarded = false`»**, как просит формулировка
ТЗ «программный триггер на повторное прохождение Onboarding»: сброс флага
у пользователя означает, что `PrivateRoute` (`App.tsx:55`) начнёт
редиректить его на онбординг **из любой существующей студии** — уже
настроенная студия станет недоступна, пока он не создаст ещё одну.
Флаг `is_onboarded` — это «пользователь вообще завершил старт», а не
«студия настроена». Параметр `?new=1` даёт ровно тот же UX (мастер
открывается заново) без разрушения инварианта.

**Guard:** прямой заход на `/onboarding` без `?new=1` при
`is_onboarded === true` → редирект на `/dashboard` (сейчас страница
просто отдаст 400 с бэка на сабмите — пользователь узнает об этом на
пятом шаге).

### Задача 6. Связка с удалением аккаунта (эпик 5) (~0:30)

`DELETE /settings/security/account` возвращает `{"redirect": "/select-crm"}`.
Фронт:

```ts
onSuccess: async () => {
  qc.clear();
  const studios = await authApi.getStudios();
  if (studios.length === 0) { logout(); navigate('/register'); }   // студий не осталось
  else navigate('/select-crm');
}
```

**Edge case:** удалённая студия — единственная. Тогда токен указывает на
несуществующий `studio_id`, `get_studio_context` ответит 403 «не состоит
ни в одной студии», и `/select-crm` покажет пустой список. Проверяем
длину списка **до** перехода и уводим на регистрацию нового пространства.

## Edge cases (сводно)

| Случай | Поведение |
|---|---|
| Пользователь в одной студии | кнопка «Сменить CRM» ведёт на `/select-crm` с одной карточкой и предложением создать вторую — прячем не кнопку, а бессмысленное состояние |
| Токен без `studio_id` (мультистудийный) | `get_studio_context` уже отвечает 400 «Не выбрана активная студия» → интерцептор фронта ведёт на `/select-crm` |
| Приглашение в чужую студию | список строится по `StudioMember` — приглашённый увидит и владельца, и свою; роль показывается на карточке |
| Переключение с открытой формой | предупреждение о несохранённых данных не делаем (`# ponytail: без dirty-guard`) |
| Экспорт клиентов на 100 000 строк | стриминг + жёсткий `LIMIT`; при превышении — 413 с просьбой сузить период |
| Новая студия без подписки | пейволл `_sub_gate` уводит на оплату — отдельной логики не пишем |
| Две вкладки, в одной переключили студию | вторая работает по старому токену до его истечения; принудительная синхронизация вкладок — в BACKLOG |
