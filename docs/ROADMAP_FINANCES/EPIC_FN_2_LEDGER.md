### ✅ 2.1 Дыра авто-леджера: продажа сертификата → операция · 🟡 · 1:00
**Простыми словами:** карта денежных потоков продукта: продажа абонемента ✅
(`back/routers/clients/subscriptions.py:87`), депозит ✅ (`back/routers/clients/loyalty.py:149`),
зарплата ✅ (`back/routers/finances/salary.py:141`), ручная операция ✅. Продажа
сертификата ❌ — `create_certificate` (`back/routers/loyalty/certificates.py:51`)
берёт `amount` и не создаёт ни операцию, ни движение по счёту. Погашение сертификата
операцию создавать **не должно** (деньги уже получены при продаже — двойной счёт).
**Что конкретно сделать:**
1. `back/schemas/loyalty/certificates.py` → в схему выпуска добавить опциональный
   `account_id`.
2. `back/routers/loyalty/certificates.py` → в `create_certificate` по образцу
   `sell_subscription`: проверка счёта студии → `db.add(Operation(type="in",
   title=f"Сертификат «…»", category="Сертификаты", amount, client_id, account_id))` +
   `account.balance += amount`; один commit с сертификатом.
3. `front/src/api/loyalty/loyalty.api.ts` + форма выпуска
   (`Loyalty/components/drawer/configs/certificates/IssueCertificateForm.tsx`) →
   передать выбранный счёт (селект счетов из `financesApi.getAccounts()`); инвалидация
   `finOperationsAll`, `finAccounts`.
**Готово, когда:** выпуск сертификата виден в «Операциях» категорией «Сертификаты»,
баланс счёта вырос; погашение операций не плодит.

### ✅ 2.2 `OperationRead`: отдать `counterparty_id` и `trainer_id` · 🟢 · 0:30
**Простыми словами:** в БД у операции есть `counterparty_id` и `trainer_id`
(`back/models/finances.py:32-33`), POST их принимает, но `OperationRead`
(`back/schemas/finances/operations.py:7`) и `_to_read` (`operations.py:34`) их не отдают —
фронт в принципе не может показать, к кому привязан расход. Без этого 2.3 и 2.4 слепые.
**Что конкретно сделать:**
1. `back/schemas/finances/operations.py` → `OperationRead` + `counterparty_id`,
   `trainer_id` (Optional).
2. `back/routers/finances/operations.py:34` → `_to_read` прокинуть оба поля.
3. `front/src/api/finances/finances.types.ts` → тип `Operation` дополнить; в развёрнутой
   строке `OperationsTab` показать контрагента (имя — из кэша `finCounterparties`).
**Готово, когда:** расход с контрагентом показывает его имя в развёрнутой строке.

### ✅ 2.3 `PATCH /finances/operations/{id}` с пересчётом баланса + правка в UI · 🟡 · 2:00
**Простыми словами:** операцию можно только создать и удалить — правки суммы/категории/
контрагента нет ни на бэке, ни в UI. Внимание на деньги: смена суммы или счёта обязана
пересчитать балансы затронутых счетов (образец отката — `delete_operation`,
`operations.py:161-166`).
**Что конкретно сделать:**
1. `back/schemas/finances/operations.py` → `OperationUpdate` (title, amount, op_date,
   category, method, account_id, client_id, counterparty_id — все Optional).
2. `back/routers/finances/operations.py` → `PATCH /operations/{operation_id}`:
   снять старый вклад с старого счёта (`-_balance_delta(старое)`), применить поля,
   добавить новый вклад на новый счёт; чужие id — 404 через `_own_or_404`; один commit.
3. `back/tests/` → тест: правка суммы 100→300 двигает баланс счёта ровно на +200;
   перенос операции между счетами двигает оба.
4. `front/src/api/finances/finances.api.ts` → `updateOperation(id, payload)`;
   `OperationsTab.tsx` → в развёрнутой строке кнопка «Редактировать» → inline-форма
   (по образцу правки цели `GoalsTab.tsx:236-259`); мутация с инвалидацией по матрице 1.3.
**Готово, когда:** тест зелёный; правка суммы из UI мгновенно обновляет баланс на «Счетах».

### ✅ 2.4 Контрагент в форме операции: селект + быстрое создание · 🟡 · 1:30
**Простыми словами:** бэк принимает `counterparty_id` в POST (`operations.py:110`),
но форма создания операции (`OperationsTab.tsx:106-120`) его не отправляет — связать
расход «Аренда» с арендодателем нельзя. Плюс требование: контрагента можно завести
не выходя из формы.
**Что конкретно сделать:**
1. `OperationsTab.tsx` → в форму (для типа «Расход» — обязательно показывать, для
   «Дохода» — тоже, опционально) селект контрагента из кэша `finCounterparties`
   + первый пункт «+ Новый контрагент».
2. «+ Новый контрагент» → раскрывает мини-форму (название + тип) прямо в форме операции;
   сабмит — существующий `financesApi.createCounterparty` → созданный id сразу
   подставляется в селект; инвалидация `finCounterparties`.
3. `createOperation` → передавать `counterparty_id`; после создания операции balance и
   deals_count контрагента пересчитаются сами (они считаются на лету —
   `counterparties.py:20`), нужна только инвалидация из 1.3.
**Готово, когда:** расход «Аренда» создаётся с новым контрагентом одним заходом в форму,
на вкладке «Контрагенты» у него deals_count=1 и минусовый баланс без F5.

### ✅ 2.5 Цели: категория из реальных + расходные цели · 🟡 · 2:00
**Простыми словами:** бэк уже сам считает прогресс авто-цели по категории доходов
(`goals.py:16`), но фронт предлагает вписать категорию руками текстом
(`GoalsTab.tsx:214`) — опечатался и цель мертва. И цель можно привязать только к доходам,
а аудит требует выбор «категория расходов или доходов» (например, лимит на аренду).
**Что конкретно сделать:**
1. `back/routers/finances/operations.py` → `GET /operations/categories`
   (`?type=in|out`): `SELECT DISTINCT category` по студии, без пустых. Роут объявить
   **выше** `/operations/{operation_id}`-роутов.
2. `back/models/finances.py` → `FinancialGoal.op_type: Mapped[str]` (default `"in"`);
   миграция `alembic revision --autogenerate` → `upgrade head`.
3. `back/routers/finances/goals.py` → `_auto_amount` фильтровать `Operation.type ==
   goal.op_type`; `back/schemas/finances/goals.py` → `op_type` в Create/Read
   (Literal["in","out"], default "in").
4. `front/src/api/finances/finances.api.ts` → `getOperationCategories(type)`;
   `GoalsTab.tsx` → в форме авто-цели переключатель «Доходы / Расходы» + селект категории
   из эндпоинта вместо текстового инпута; тип и категорию показать бейджем на карточке цели.
**Готово, когда:** авто-цель «Аренда (расходы)» растёт от каждого расхода категории
«Аренда»; выбрать несуществующую категорию невозможно.
