### 4.1 Бэк: часы и выручка в `SalaryRow` + история выплат · 🟡 · 1:30
**Простыми словами:** `_sessions_and_hours` (`back/routers/finances/salary.py:19`) уже
считает часы и выручку занятий сотрудника, но `list_salaries` кладёт в `SalaryRow`
только `sessions_count` и `amount` — расширенной статистике в карточке неоткуда взяться.
Истории выплат нет вовсе, хотя таблица `salary_payments` хранит всё (снапшот ставки,
период, `paid_at` — `back/models/finances.py:136`).
**Что конкретно сделать:**
1. `back/schemas/finances/salary.py` → `SalaryRow` + `hours_worked: float`,
   `lessons_revenue: int`, `paid_at: Optional[datetime]`.
2. `back/routers/finances/salary.py` → `list_salaries`: прокинуть `hours`/`revenue` из
   уже вызванного `_sessions_and_hours`; `paid_at` — из `SalaryPayment` периода
   (сейчас берётся только `paid_ids`, взять пары id→paid_at тем же запросом).
3. Там же → `GET /salaries/{user_id}/history`: `SalaryPayment` сотрудника по студии,
   `order_by(period_start.desc())`, response_model `list[SalaryRead]` (схема уже есть).
   Сотрудник не в студии → 404 (как в `pay_salary`).
**Готово, когда:** GET /salaries отдаёт часы/выручку/дату выплаты; history возвращает
прошлые периоды со снапшотами ставки.

### 4.2 Развёрнутая карточка: статистика, история, статус выплаты · 🟡 · 2:00
**Простыми словами:** сейчас в раскрытой панели (`SalariesTab.tsx:161-193`) — ставка,
число тренировок, итого. Надо: расширенная статистика по сотруднику, история выплат
и управление статусом прямо из карточки.
**Что конкретно сделать:**
1. `front/src/api/finances/finances.api.ts` → `getSalaryHistory(userId)`; тип
   `SalaryPayment` уже описан в `finances.types.ts`; ключ `finSalaryHistory(userId)`
   в `queryKeys.ts`; типы `SalaryRow` дополнить полями из 4.1.
2. `SalariesTab.tsx` → раскрытая панель в 2 колонки:
   - **Статистика периода:** тренировок, отработано часов, выручка занятий
     (`lessons_revenue`), начислено (`amount`) — всё из `SalaryRow`, без новых запросов;
   - **История выплат:** `useQuery(finSalaryHistory)` лениво при раскрытии — строки
     «период → сумма → дата выплаты» (снапшот ставки в тултипе кита `Tooltip`).
3. Статус в карточке: для «Ожидает» — та же кнопка «Выплатить» (существующая мутация
   `paySalary`; она уже автоматом пишет расход «Зарплата» — `salary.py:141`), после
   успеха показать `paid_at`; для «Выплачено» — бейдж с датой. Инвалидация — по матрице
   1.3 (`finSalaries`, `finOperationsAll`, `finAccounts`, `finSalaryHistory`).
**Готово, когда:** раскрытая карточка показывает часы/выручку/историю; выплата из
карточки мгновенно появляется в «Операциях» расходом «Зарплата».

### 4.3 Правка ставки из карточки · 🟡 · 1:30
**Простыми словами:** ставка правится только в профиле сотрудника (раздел Сотрудники),
из Зарплат — никак (только жёлтая подсказка «укажите её в профиле»,
`SalariesTab.tsx:186-190`). Эндпоинт уже существует: `PATCH /staff/{id}` принимает
`rate`/`rate_type` (`back/routers/staff/profiles.py:376`). Бэк не трогаем.
**Что конкретно сделать:**
1. `SalariesTab.tsx` → в раскрытой карточке у строки «Ставка» кнопка «Изменить» →
   inline-форма: число + `Select` кита (фикс/час/% — ключи `fixed/hourly/percent`,
   подписи из `finances.json`).
2. Мутация → существующий `staffApi` (`front/src/api/staff/…` — метод update профиля);
   инвалидация `finSalaries(...)` и `staff` (расчёт пересчитается на бэке при
   рефетче — ставка применяется к невыплаченным периодам, снапшоты старых выплат
   не меняются, это уже гарантирует `rate_snapshot`).
3. Жёлтую подсказку «ставка не задана» заменить на ту же inline-форму (призыв задать
   прямо здесь).
**Готово, когда:** смена ставки из Зарплат мгновенно пересчитывает «Итого к выплате»
невыплаченного периода, история прошлых выплат не меняется.
