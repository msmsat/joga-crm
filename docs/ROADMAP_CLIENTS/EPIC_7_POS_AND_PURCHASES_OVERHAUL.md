# Эпик CL-7 — Редизайн раздела «Покупки» и оживление кассы

Цель эпика простыми словами: касса из CL-6 **работает по бэку**, но выглядит как из
90-ых и в паре мест спотыкается на UX. Функционал уже есть — `POST /checkout/calculate`
и `POST /checkout/pay` (`back/routers/checkout/router.py`) реально тянут скидки через
`resolve_price`, проводят доход, начисляют продукт, списывают бонусы и пишут событие
атомарно. Поэтому CL-7 — **не переписывание логики, а тотальный редизайн UI + локализация
ошибок + чистка навигации + фикс одного реального бага оплаты наличными.** Бэк почти не
трогаем: там уже честный расчёт и синхронизация.

**Стартовая точка (что УЖЕ работает — не переделывать):**
- `POST /checkout/calculate` (`router.py:102`) и `_quote` (`:38`) считают цену честно:
  `resolve_price` (студийная скидка + персональный оффер + промокод, «самая выгодная без
  стека») + бонусы `min(баланс, цена)`. **Скидка НЕ захардкожена** — берётся из БД. Блок 4
  из ТЗ по бэку по сути уже сделан; остаётся только фронт-отображение.
- `POST /checkout/pay` (`router.py:127`) в одной транзакции (единый `db.commit()` на `:222`):
  `attach_subscription` → `Operation` (доход) + баланс счёта + `ClientPayment` + баллы;
  списание бонусов `apply_points_change`; пометка промокода/оффера; `log_activity("payment")`
  в ленту клиента. Синхронизация Финансов/LTV/Событий (Блок 5 из ТЗ) — **уже есть**.
- Фронт-касса уже использует UI-кит и глобальный тост: `WalletPOS.tsx` тянет `useToast`,
  `errorMessage(e, t)`, `ConfirmModal` на «Наличные», `InfoHint`-заглушку на «Карту»,
  инвалидацию `wallet/client/clientEvents` после оплаты (No-F5 из Блока 5 — **готово**).

**Что на самом деле сломано / уродливо (это и чиним):**
1. **UI из 90-ых:** вся вёрстка `WalletTab/WalletCatalog/WalletPOS` — инлайновые стили,
   `…` вместо скелетонов, каталог валит абонементы и разовые в две простыни без табов,
   тумблер бонусов нарисован руками. Никаких CSS-модулей, отступы/тени/скругления не по
   дизайн-системе.
2. **Навигация:** вкладка называется «Кошелёк» (`clients.json:37`), нужно «Покупки».
   В режиме просмотра нет верхнего ряда `Активные | Архив | [Купить]` — секции просто
   висят вертикально. В каталоге нет верхних табов `Абонементы | Разовые` и кнопки «Все».
3. **Локализация ошибок — реальный пробел:** `errorMessage` (`api/errorMessage.ts:9`)
   показывает `detail` с бэка **как есть**, а все `detail` в `checkout/router.py`
   захардкожены по-русски («Пакет не найден», «Оплата картой пока недоступна»…). При
   английском UI юзер увидит русский текст ошибки. Переводим по коду ошибки, а не по строке.
4. **Оплата наличными «не работает» — найденный root cause:** кнопка оплаты `disabled`,
   если `cashAccount` не найден. `WalletPOS.tsx:52` ищет счёт `accounts.find(a => a.type
   === 'cash')`, но `Account.type` — свободная строка (`models/finances.py:15`,
   `String(20)`). Если касса заведена с другим `type` (напр. `'касса'`/label) — счёт не
   находится, `cashAccount` undefined, кнопка вечно disabled → «оплата не проходит».

**Честное ограничение (Zero Trust соблюдён, как в CL-6):** весь расчёт цены и вся
синхронизация — строго на бэке в одной транзакции, фронт только рисует результат. CL-7
**ничего не добавляет к деньгам** — только к пикселям, переводам и одному фиксу выбора счёта.
Эквайринг («Карта») остаётся заглушкой (MVP — только наличные).

Источник: тест-фидбэк от 2026-07-20 по вкладке покупок/кассы. Продолжение CL-6
(`EPIC_CL_6_WALLET_AND_BOOKING.md`). Все находки проверены по коду — файлы и строки в задачах.

**Зависимости:** 7.1 (CSS-модуль + скелетоны) — фундамент вёрстки, на нём стоят 7.2/7.4/7.5.
7.3 (rename «Покупки») независим. 7.6 (локализация ошибок бэка) — бэк+общий словарь,
независим. 7.7 (фикс cash-счёта) независим и критичен — делать первым, чтобы касса вообще
проводила оплату во время тестов редизайна.

**Обозначения сложности:** 🟢 простая · 🟡 средняя · 🔴 сложная, делать внимательно.

## Блок 0. Разблокировать кассу

### 7.7 Фикс выбора счёта наличных · 🟡 · 1:00
**Простыми словами:** кнопка «Подтвердить оплату» вечно `disabled`, когда касса не нашла
счёт наличных. `WalletPOS.tsx:52` берёт `accounts.find(a => a.type === 'cash')`, а
`Account.type` — свободная строка (`back/models/finances.py:15`). Если студия завела кассу
не с `type==='cash'` — счёт не находится, `cashAccount` undefined, `disabled` на `:121`
не снимается. Отсюда «наличные не работают».

**Где:** `front/src/pages/dashboard/Clients/components/WalletPOS.tsx:52` (`cashAccount`),
`:121` (`disabled`); проверить сидинг/создание счетов в
`back/routers/finances/router.py` (эндпоинт `POST /accounts`) и что реально лежит в
`Account.type` у студии.

**Что конкретно сделать:**
1. **Диагностика (5 мин):** посмотреть, какой `type` реально у кассового счёта студии
   (GET `/finances/accounts`). Зафиксировать канон: `type === 'cash'` — договорённость,
   заведённая при онбординге/сидинге. Если счета создаются с другим `type` — источник
   бага там.
2. **Фронт-фолбэк (ладдер: одна строка вместо новой ручки):** выбор счёта делаем
   устойчивым — `accounts.find(a => a.type === 'cash') ?? accounts[0]`. Уже есть на `:52`,
   но убедиться, что `accounts[0]` реально существует; если счетов нет вообще — вместо
   тихого disabled показать заглушку «Сначала создайте счёт в Финансах» (тост/`InfoHint`),
   а не молча гасить кнопку.
3. Если счета создаются некорректно (`type` не канон) — починить в точке создания
   (`POST /accounts` / сидинг), чтобы касса шла `type==='cash'`. Не плодить маппинги на
   фронте — канон один. ponytail: чиним источник значения, а не каждый `.find` по коду.
4. Кнопка `disabled` (`:121`): оставить `!cashAccount` в условии — но теперь пустой список
   даёт понятную заглушку, а не немую кнопку.

**Готово, когда:** на студии с корректно заведённой кассой «Наличные» → «Подтвердить» не
disabled и проводит оплату; если счетов нет — юзер видит внятную подсказку, а не мёртвую
кнопку.

**Простыми словами:** бэк кидает `HTTPException(detail="Пакет не найден")` — русская
строка; `errorMessage` (`api/errorMessage.ts:9`) показывает её **как есть**. При английском
UI юзер видит русский текст. Нужен машинный код ошибки, который фронт переводит через i18n.

**Где:** `back/routers/checkout/router.py` (все `raise HTTPException`, `:86/:95/:97/:143/
:151/:162`); `front/src/api/errorMessage.ts:9`; словари `front/src/locales/{ru,en}/
common.json` (неймспейс `errors`).

**Что конкретно сделать:**
1. **Бэк:** к каждому `HTTPException` в `checkout/router.py` добавить машинный код. Способ
   минимальный (ладдер — не заводить новый класс исключений на весь бэк ради одного
   роутера): `detail={"code": "checkout.package_not_found", "message": "Пакет не найден"}`
   — FastAPI сериализует dict как есть; `message` остаётся русским фолбэком для логов/старого
   фронта. Коды: `package_not_found`, `package_inactive`, `client_not_found`,
   `account_not_found`, `card_unavailable`, `promo_expired`.
2. **Фронт `errorMessage`:** если `detail` — объект с `code`, переводить
   `t('common:errors.' + code)`; если перевода нет — фолбэк на `detail.message`, затем на
   текущую логику по статусу. Сохранить обратную совместимость: `detail`-строка (остальной
   бэк) работает как раньше (`:9` уже это делает).
3. **Словари:** добавить `errors.checkout.*` в `common.json` ru **и** en с человеческими
   формулировками на обоих языках.
4. Фронт-валидацию инпутов кассы (пустой промо и т.п.) — тоже через i18n-ключи, не
   голые строки (в POS уже `t(...)`, проверить, что нигде не осталось хардкода).
5. Тест-чек `back/tests/`: `card` → 400 с `detail.code == "checkout.card_unavailable"`;
   несуществующий пакет → `checkout.package_not_found`.

**Готово, когда:** при английском UI ошибки кассы приходят на английском; при русском — на
русском; старые строковые `detail` других роутеров не сломаны; тосты глобальные (в POS уже
`useToast`), локальных алертов в разделе нет.
