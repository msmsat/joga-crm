# Эпик CL-6 — Кошелёк клиента, касса и контроль доступа к записи

Цель эпика простыми словами: сейчас записать клиента на занятие можно **вообще без
абонемента** — ни `POST /schedule/reservations` (Журнал), ни `POST /clients/{id}/booking`
(панель клиента) не проверяют, есть ли у клиента право посетить это занятие. Плюс два
UI-бага в панели клиента (недельная сетка записи обрезает субботу/воскресенье, вкладка
«События» тянет горизонтальный скролл всей страницы) и целый отсутствующий раздел —
**«Кошелёк»**: клиент не видит своих абонементов/разовых, и продать их из панели нельзя.
Закрываем всё это одним эпиком: строгая валидация доступа на бэке, фиксы вёрстки и
полноценный флоу кассы (каталог → POS → оплата) прямо в карточке клиента.

**Стартовая точка лучше, чем кажется — половина бэка кассы уже написана:**
- `attach_subscription` (`back/routers/clients/subscriptions.py:24`) уже делает всё, что
  нужно на «оплате»: создаёт абонемент, проводит доход в кассу (`Operation` + баланс
  счёта), пишет `ClientPayment`, начисляет баллы — и **не коммитит**, отдавая единый
  commit вызывающему. Готовый кирпич для `POST /checkout/pay`.
- `resolve_price` (`back/services/pricing.py:40`) уже считает финальную цену: студийная
  скидка + персональный оффер + промокод, «самая выгодная, без стека». Именно то, что
  просит `POST /checkout/calculate` — деньги не двигает, только цену.
- `find_valid_promo` (`back/routers/loyalty/promocodes.py`) валидирует промокод, отдавая
  объект для `resolve_price`.
- Баллы: `apply_points_change` (`loyalty.py:38`) списывает баллы и пишет транзакцию, в
  минус — 400. `accrue_points` (`loyalty.py:52`) начисляет за оплату. Оба не коммитят.
- **Матчинг типа занятия уже есть в данных:** `SubscriptionPackage.service_ids` (JSON,
  `null` = все занятия) (`back/models/loyalty.py:218`), а проданный абонемент хранит
  `package_id` (`back/models/schedule.py:61`). Значит «абонемент на Хатху ≠ Пилатес»
  проверяется через пакет — **без миграции**.
- `sell_subscription` (`subscriptions.py:93`) — готовый образец всего флоу продажи с
  ценой и промокодом; фронт-обёртка списка пакетов уже есть
  (`loyaltyApi.getSubscriptionPackages()`), эндпоинт — `GET /loyalty/subscriptions`
  (`back/routers/loyalty/packages.py:51`).
- События клиента уже логируются (`log_activity`, `back/activity.py`); `book_lesson`
  (`profiles.py:816`) уже пишет лог `booking` — образец для лога оплаты.
- Категория операции «Абонементы» уже триггерит баллы; финансовая часть кассы = тот же
  `Operation`, что везде.

**Честное ограничение (Zero Trust соблюдён):** вся валидация доступа, расчёт цены,
списание баллов и начисление продукта — **строго на бэке в одной транзакции**. Фронт
только рисует результат. Эквайринг («Карта») — заглушка, MVP принимает только наличные.

Источник: требования от 2026-07-20, эпик 6. Обзор — в `ROADMAP.md`. Все находки проверены
по коду — файлы и строки в задачах.
Зависимости: 6.1 (бэк-валидация) → 6.4 (выдача доступных занятий) → 6.2 (фронт скрывает
недоступные); касса 6.7 → 6.8 → 6.9; Кошелёк 6.6 стоит на эндпоинте 6.5.

**Обозначения сложности:** 🟢 простая · 🟡 средняя · 🔴 сложная, делать внимательно.

## Блок 1. Строгая валидация записи (Booking Access Control)

### 6.1 Единый гейт доступа `assert_can_book` · 🔴 · 2:30
**Простыми словами:** сейчас клиента можно записать без абонемента и без разового —
**две** ручки пишут `Reservation` независимо и ни одна не проверяет право:
`create_reservation` (`back/routers/schedule/reservations.py:16`, Журнал) и `book_lesson`
(`back/routers/clients/profiles.py:772`, панель). Лазить в обе и дублировать проверку —
плохо; ставим один гейт в общей функции, через которую проходят обе (ponytail: одна
проверка в shared-хелпере вместо двух копий в вызывающих).

**Где:** новый `back/services/booking_access.py`; вызов из `reservations.py:create_reservation`
и `profiles.py:book_lesson`; модель `SubscriptionPackage.service_ids` (`models/loyalty.py:218`),
`ClientSubscription.package_id` (`models/schedule.py:61`), `Lesson.service_id`
(`models/schedule.py:46`).

**Что конкретно сделать:**
1. `async def assert_can_book(db, studio_id, client_id, lesson) -> ClientSubscription | None`
   в `services/booking_access.py`. Логика (Zero Trust — только бэк):
   - Ищем активный незамороженный абонемент клиента с остатком:
     `status == "active" and not is_frozen and used_classes < total_classes`.
   - **Матч типа занятия:** для каждого такого абонемента берём его пакет по `package_id`;
     если у пакета `service_ids` не пуст и `lesson.service_id not in service_ids` — этот
     абонемент не подходит (абонемент на Хатху, занятие Пилатес). `service_ids is null`
     или пусто = универсальный, подходит под любое.
     `package_id is null` (старые абонементы до V5-4) — тоже считаем универсальным
     (не ломаем ретроактивно; ponytail-комментарий об этом).
   - Подходит хотя бы один → возвращаем его (списывать занятие — уже существующая логика
     `attend_reservation`, здесь только проверяем право).
   - Ничего не подходит → `HTTPException`:
     - есть абонемент, но не на этот тип → **400** «Абонемент клиента не подходит для
       этого занятия».
     - вообще нет доступного продукта → **403** «У клиента нет активного абонемента или
       разового занятия для записи».
   - «Разовое занятие»: MVP-модели отдельной сущности «разовое» нет. Разовое = запись,
     оплаченная тут же через кассу (6.9) — она создаёт `Reservation` напрямую, минуя гейт
     (её право — сам факт оплаты). Гейт проверяет только запись «по абонементу».
     ponytail: не заводим таблицу «пакет разовых», разовое = одна оплаченная запись;
     отдельный продукт «пул разовых» — в BACKLOG, если понадобится перенос.
2. Дёрнуть `assert_can_book` в обеих ручках **после** проверки мест и дубля, **до**
   создания `Reservation`. `book_lesson` (панель) и `create_reservation` (Журнал) —
   обе через `require_role("owner","admin")`, гейт применяется одинаково.
3. **Не** трогаем `sell_subscription` и кассовую запись (6.9) — там право = оплата.
4. Тест-чек: `back/tests/` — клиент без абонемента → 403; с абонементом на другой
   `service_id` → 400; с универсальным (`service_ids=null`) → запись проходит; с
   подходящим `service_ids=[lesson.service_id]` → проходит; замороженный / исчерпанный
   абонемент → 403.

**Готово, когда:** записать клиента без подходящего продукта нельзя ни из Журнала, ни из
панели; текст ошибки понятен; универсальные и старые абонементы работают как раньше.

### 6.4 `GET /schedule/lessons/{id}/eligible-clients` · 🟡 · 1:15
**Простыми словами:** при записи из Журнала нужен список клиентов, которых **можно**
записать именно на это занятие (Zero Trust — фронт не решает, кто подходит). Сейчас
Журнал берёт всех клиентов студии. Даём эндпоинт, который прогоняет каждого через тот же
гейт, что и 6.1.

**Где:** новый хендлер в `back/routers/schedule/lessons.py` (рядом с `get_lesson`);
использует `assert_can_book` из 6.1; фронт-обёртка — `front/src/api/schedule/schedule.api.ts`.

**Что конкретно сделать:**
1. `GET /schedule/lessons/{lesson_id}/eligible-clients` → список `{id, name, last_name,
   phone, avatar_color, subscription_hint}` только тех клиентов студии, для кого
   `assert_can_book` не бросает (оборачиваем вызов в try/except на подходящесть, без
   HTTP-исключения — переиспользуем ядро гейта как булеву-хелпер `can_book(...) -> bool`,
   чтобы не ловить исключения в цикле; вынести общее ядро, гейт 6.1 = ядро + raise).
   ponytail: N клиентов × запрос абонементов — приемлемо для MVP-объёмов; при росте —
   один JOIN. Помечаем комментарием.
2. Скоуп занятия — через `get_scoped_lesson` (404 чужая студия / 403 тренер на чужом),
   как в `get_lesson` (`lessons.py:137`).
3. Фронт-обёртка `scheduleApi.getEligibleClients(lessonId)` + тип в `schedule.types.ts`.

**Готово, когда:** эндпоинт для занятия Пилатес отдаёт только клиентов с подходящим
доступом; для тренера на чужом занятии — 403.

**Что конкретно сделать:**

   confirm ошибка 403/400 с бэка уже ловится (`confirmBooking:171`) — вывести её текст

   (видно из `wallet`/`profile`) — под кнопкой «Записать» показать заглушку
   «У клиента нет доступного абонемента для записи» и не блокировать (владелец может

   если пусто — заглушка «Нет клиентов с доступом на это занятие».
3. Никаких клиентских решений о цене/доступе — только отображение того, что вернул бэк.

**Готово, когда:** в Журнале нельзя выбрать клиента без доступа; в панели попытка записать
без абонемента даёт понятный тост, а не молчит; заглушки локализованы.

## Блок 2. UI/UX фиксы карточки клиента

### 6.3a Недельная сетка записи: все 7 дней без скролла · 🟡 · 1:00

**Где:** блок рендера `next7Days` в booking-подпанели `ClientProfileSlider.tsx`

**Что конкретно сделать:**
1. Контейнер дней — `display: grid; grid-template-columns: repeat(7, 1fr)` (нативный
   CSS Grid, ладдер: платформа вместо flex+фиксированной ширины). Ширину колонки не

2. Внутри ячейки: `min-width: 0`, день недели + число в столбик, `text-align: center`,

4. Проверить в тёмной теме и на узкой панели — горизонтального скролла ни у ряда, ни у

**Готово, когда:** все 7 дней видны целиком в панели, воскресенье не обрезано, всей
страницы горизонтальный скролл не появляется.

### 6.3b Вкладка «События»: скроллится только внутренний список · 🟢 · 0:45

   Использовать `Card`, `Button` из `components/ui/index` (UI-кит обязателен, CLAUDE.md).
3. Данные — `useClientWallet` через TanStack Query (инфраструктура готова, CL-2), ключ
   `wallet(clientId)` в `queryKeys.ts` (очередь миграции Клиентов уже там).
4. Кнопка `[Купить]` в шапке вкладки — переключает панель в режим каталога (6.8).
   Локальный стейт `walletMode: 'view' | 'catalog' | 'pos'` в `WalletTab`.
5. Валюта — `getCurrencySymbol(useStudioCurrency())` (уже используется в панели, `:371`).

**Готово, когда:** вкладка «Кошелёк» показывает реальные активные и архивные абонементы
клиента; переключение табов не дёргает панель (CL-1.3 уже починил перемонтирование);
кнопка «Купить» открывает каталог.

## Блок 4. Касса (Checkout) и финансы

### 6.7 `POST /checkout/calculate` — финальная цена · 🟡 · 1:30
**Простыми словами:** при выборе продукта в кассе нужно показать итог к оплате с учётом
промокода и бонусов. Всё считает бэк (Zero Trust) — фронт только присылает выбор.
`resolve_price` уже считает скидки+промо, `apply_points_change`-логика знает про баланс
баллов — собираем эндпоинт-калькулятор, деньги он **не двигает**.

**Где:** новый роутер `back/routers/checkout/` (`router.py` + подключить в `back/main.py`);
переиспользует `resolve_price` (`services/pricing.py:40`), `find_valid_promo`
(`routers/loyalty/promocodes.py`), баланс баллов из `ClientLoyaltyCard`; схемы —
`back/schemas/checkout.py`; фронт-обёртка — новый `front/src/api/checkout/`.

**Что конкретно сделать:**
1. `POST /checkout/calculate` body `{ client_id, product_id, product_type:
   "subscription"|"single", promo_code?, use_bonuses: bool }` (`require_role("owner","admin")`).
2. Базовая цена: для `subscription` — `SubscriptionPackage.price` (по `product_id`,
   скоуп студии, `is_active`); для `single` — `SubscriptionPackage.per_visit_price` или
   цена услуги (решить: MVP — `per_visit_price` уже есть на пакете, `models/loyalty.py:214`).
3. Промокод: `find_valid_promo` → `resolve_price(db, studio_id, client_id, base, promo)`
   → `final_price` (после скидок).
4. Бонусы: если `use_bonuses` — сколько баллов можно списать. Правило V1: 1 балл = 1 ₽,
   списываем `min(points_balance, final_price)` (не в минус). Возвращаем
   `bonuses_applied`. ponytail: списание бонусов полной суммой, без «максимум X% чека» —
   процентный лимит в BACKLOG.
5. Ответ: `{ base_price, discount, promo_valid, bonuses_available, bonuses_applied,
   total_price }`. Ничего не коммитим, ничего не помечаем использованным (промокод/оффер
   помечаются только при `pay`).

**Готово, когда:** калькулятор возвращает верный итог: база − скидка/промо − бонусы;
невалидный промокод не роняет запрос (флаг `promo_valid=false`); отрицательной цены не бывает.

### 6.8 Каталог + POS-экран внутри панели · 🔴 · 2:30
**Простыми словами:** по кнопке `[Купить]` (6.6) панель Кошелька превращается в каталог с
`[Назад]`; в каталоге 2 раздела (Абонементы / Разовые); выбор продукта открывает экран
кассы (POS) с ценой, промокодом, тумблером бонусов и выбором метода оплаты.

**Где:** `WalletTab.tsx` (стейт `walletMode`); подкомпоненты `WalletCatalog.tsx`,
`WalletPOS.tsx` (< 200 строк каждый); данные каталога — `loyaltyApi.getSubscriptionPackages()`
(обёртка уже есть); касса — `checkoutApi.calculate/pay` (6.7/6.9); UI-кит `Button`,
`Card`, `Input`, `ConfirmModal`, `useToast`.

**Что конкретно сделать:**
1. **Каталог** (`walletMode==='catalog'`): `[Назад]` возвращает в `view`. Два раздела —
   «Абонементы» (пакеты из `getSubscriptionPackages()`, `is_active`) и «Разовые»
   (`per_visit_price` пакетов, либо услуги — согласно решению в 6.7 п.2). Карточка
   продукта → клик → `walletMode='pos'` с выбранным `product_id/type`.
2. **POS** (`walletMode==='pos'`): при входе и при изменении промокода/тумблера бонусов
   дёргаем `checkout/calculate` (дебаунс на промокод) → показываем `base → скидка →
   бонусы → ИТОГ`. Инпут промокода (`Input`), тумблер «Списать бонусы» (переключатель
   кита), под ним выбор метода: `[Наличные] / [Карта]`.
3. **Метод «Карта»** → UI-заглушка (Stub) «Интеграция с эквайрингом в разработке»
   (`InfoHint`/дизейбл кнопки оплаты). Оплату не проводим.
4. **Метод «Наличные»** → кнопка `[Подтвердить оплату]` → `ConfirmModal` («Точно
   оплачено?», danger-режим замены `window.confirm`, CLAUDE.md UI-кит) → `checkout/pay`
   (6.9).
5. Успех оплаты: тост success, `walletMode='view'`, инвалидация `wallet(clientId)`,
   `client(clientId)` (баланс баллов), `clientEvents(clientId)` — панель обновляется без
   F5 (TanStack Query, CL-2).
6. Все строки — `clients.json`; цена — валюта студии; никакой цены/скидки фронт не считает
   сам, только рисует ответ `calculate` (Zero Trust).

**Готово, когда:** Купить → каталог → продукт → POS с реальной ценой из бэка; «Карта»
показывает заглушку; «Наличные» через Confirm проводят оплату; после оплаты Кошелёк и
баллы обновлены.

### 6.9 `POST /checkout/pay` — атомарная транзакция кассы · 🔴 · 2:15
**Простыми словами:** подтверждение оплаты наличными. Бэк в **одной транзакции**: доход в
Финансы, списание бонусов, начисление продукта клиенту, лог в События. Большая часть —
готовые некоммитящие кирпичи (`attach_subscription`, `apply_points_change`, `log_activity`).

**Где:** `back/routers/checkout/router.py`; переиспользует `attach_subscription`
(`routers/clients/subscriptions.py:24`), `resolve_price` (`services/pricing.py`),
`find_valid_promo`, `apply_points_change` (`routers/clients/loyalty.py:38`), `log_activity`
(`back/activity.py`); модели `Operation`, `ClientPayment`, `Reservation`.

**Что конкретно сделать:**
1. `POST /checkout/pay` body `{ client_id, product_id, product_type, account_id,
   promo_code?, use_bonuses, payment_method:"cash" }` (`require_role("owner","admin")`).
   `payment_method != "cash"` → 400 «Оплата картой пока недоступна» (эквайринг —
   заглушка, деньги не проводим на бэке тоже).
2. **Пересчёт цены на сервере заново** (не доверяем присланному итогу — Zero Trust):
   `find_valid_promo` + `resolve_price` → `final_price`. Бонусы к списанию считаем так же,
   как в 6.7.
3. Всё в одной транзакции, **единый commit в конце**:
   - **Абонемент:** `attach_subscription(db, studio_id, client_id, package, account,
     mark_paid=True, price=final_price − bonuses, payment_method="cash")` — он уже создаёт
     `Operation` (доход, категория «Абонементы», баланс счёта += ), `ClientPayment`,
     начисляет баллы за оплату. Для «Разового» — создать `Reservation`
     (`booking_channel="single"`) + `Operation` + `ClientPayment(action_type="single")`
     вручную по образцу (право на запись = сама оплата, гейт 6.1 обходится законно).
   - **Списание бонусов:** если `bonuses > 0` — `apply_points_change(client_id, studio_id,
     -bonuses, "Оплата бонусами", db)` (в минус → 400, но 6.7 уже ограничил ≤ баланса).
   - **Промокод/оффер:** пометить `used_count += 1` / `is_used = True` (образец —
     `sell_subscription:135–139`).
   - **Лог в События:** `log_activity(db, studio_id, "payment", title=…, entity_type=
     "operation"/"reservation", entity_id=…)` — попадёт в ленту событий клиента (образец —
     `book_lesson:816`).
   - `await db.commit()` один раз; при любой ошибке — откат всей транзакции (ничего не
     начислено, деньги не проведены).
4. `attach_subscription` **не** передавать уже-скидочную цену дважды: base для баллов —
   фактически заплаченное. Проверить, что баллы начисляются от реально уплаченной суммы
   (`final_price − bonuses`), не от прайса.
5. Тест-чек: `back/tests/` — оплата абонемента наличными → появился `Operation` (доход),
   баланс счёта вырос на уплаченное, `ClientSubscription` активен, баллы начислены,
   бонусы списаны, промокод `used_count+1`, лог `payment` есть; повтор с тем же промокодом
   сверх лимита → 400 и **ничего** не записано (атомарность); `payment_method="card"` → 400.

**Готово, когда:** одна кнопка «Подтвердить» проводит доход, начисляет продукт, списывает
бонусы и пишет событие — атомарно; сбой на любом шаге откатывает всё; карта отклоняется
заглушкой.

## Definition of Done эпика

- Записать клиента без подходящего активного продукта нельзя ни из Журнала, ни из панели
  (403/400 с понятным текстом); абонемент на один тип занятия не пускает на другой;
  универсальные и старые (без `package_id`) абонементы работают как раньше.
- В Журнале список клиентов для записи — только те, у кого есть доступ на это занятие.
- Недельная сетка записи показывает все 7 дней без горизонтального скролла; вкладка
  «События» скроллит только внутренний список, не распирая страницу.
- Вкладка «Кошелёк» показывает реальные активные/архивные продукты клиента текущей студии.
- Флоу кассы работает на реальных эндпоинтах (никаких моков): каталог → POS →
  `calculate` (цена с бэка) → `pay`; «Карта» — заглушка, «Наличные» через Confirm.
- `pay` в одной транзакции проводит доход, начисляет продукт, списывает бонусы, помечает
  промокод/оффер и пишет событие; сбой откатывает всё.
- Вся валидация доступа и весь расчёт цены — строго на бэке; фронт только отображает
  (Zero Trust). UI-кит переиспользован, ничего app-wide не построено; `npm run build &&
  npm run lint` и бэковые тесты — чисто.
- Отложено в `docs/BACKLOG/`: полноценный эквайринг (карта), процентный лимит списания
  бонусов, отдельный продукт «пул разовых», возвраты из кассы.
