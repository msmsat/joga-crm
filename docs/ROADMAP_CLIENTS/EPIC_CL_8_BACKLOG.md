# Эпик CL-8 — Клиенты: модалка добавления (создание и валидация)

Цель эпика простыми словами: модалка «Новый клиент» красивая, но врёт и дырявая:
обязательное только имя (телефон/email/город можно не заполнять), день рождения — свободный
текст, который **вообще не отправляется на сервер**, а шаг «Абонемент» — бутафория
(выдуманные пакеты 8/10/12 занятий, выбор никуда не уходит). Делаем жёсткую валидацию,
подключаем реальные пакеты абонементов и связываем оплату с кассой — всё одной транзакцией.

Источник: требования от 2026-07-17, эпик 8. Все находки проверены по коду.
Обзор и стартовая точка — в `ROADMAP.md`.

**Важно понимать про старт:**
- Реальный путь модалки: `front/src/pages/dashboard/Clients/components/modals/AddClientModal.tsx`
  + `hooks/useClientForm.ts` (не `front/src/components/clients/…`).
- `note` **уже работает**: схема принимает (`schemas/clients/clients.py:15`), эндпоинт
  сразу создаёт `ClientNote` (`profiles.py:469–475`). По бэку тут делать нечего.
- Эндпоинт списка пакетов **уже есть**: `GET /loyalty/subscriptions`
  (`back/routers/loyalty/packages.py:51`), фронт-обёртка тоже —
  `loyaltyApi.getSubscriptionPackages()` (`loyalty.api.ts:72`). Ловушка: он под
  `require_role("owner")`, а клиентов создаёт и **admin** → для админа будет 403 (задача 8.1).
- Образец «продать абонемент + деньги в кассу + баллы» уже написан — `sell_subscription`
  (`back/routers/clients/subscriptions.py:39–99`): ClientSubscription + Operation
  (категория «Абонементы») + `accrue_points`. В 8.2 переиспользуем его логику.
- `create_client` уже транзакционен (один commit на клиента + заметку + запись в ленту
  студии) — дополняем его, а не строим новый эндпоинт.
- `ClientCreate` используется **только** этим эндпоинтом (онлайн-запись создаёт `Client`
  напрямую, `booking/public.py:170`) — ужесточать поля безопасно.
- Попутные находки: `bday` не отправляется на сервер вовсе (`AddClientModal.tsx:184–192` —
  `birth_date` нет в payload); ошибка создания молча глотается (`:197–200`); тумблер
  «SMS-приветствие» на шаге 4 — декорация (ничего не шлёт).

**Обозначения сложности:** 🟢 простая · 🟡 средняя · 🔴 сложная, делать внимательно.

### 8.1 Схема ClientCreate: обязательные поля + новые поля · 🟢 · 0:45
**Простыми словами:** сейчас в `ClientCreate` обязательное только имя — телефон, email и
город `Optional` (`back/schemas/clients/clients.py:7–16`). Делаем их строго обязательными
и добавляем поля для абонемента. Заодно чиним доступ админа к списку пакетов.
**Где:** `back/schemas/clients/clients.py`, `back/routers/loyalty/packages.py:53`,
`front/src/api/clients/clients.types.ts`.
**Что конкретно сделать:**
1. `ClientCreate`: `phone: str`, `email: str`, `city: str` — без `Optional`/дефолтов
   (Pydantic вернёт 422 на пустоту); `name` уже обязателен. `note` уже есть — не трогать.
2. Добавить: `membership_id: Optional[int] = None`, `is_membership_paid: bool = False`.
3. `list_packages` (`packages.py:53`): `require_role("owner")` → `require_role("owner", "admin")`
   — только чтение списка; CRUD пакетов остаётся owner-only.
4. Фронт-тип `ClientCreate` в `clients.types.ts` синхронизировать.
**Готово, когда:** POST без телефона/email/города → 422; админ получает список пакетов без 403.

### 8.2 Транзакция создания: клиент + абонемент + оплата + события · 🟡 · 1:30
**Простыми словами:** эндпоинт уже создаёт клиента и заметку в одной транзакции
(`profiles.py:444–485`). Дополняем тот же единый commit: привязка абонемента и, если
оплачен, — деньги в кассу и баллы. Образец логики целиком лежит в `sell_subscription`
(`routers/clients/subscriptions.py:70–95`) — не копипастить, а вынести общий кусок
(например, `_attach_subscription(...)`) и звать из обоих мест.
**Где:** `back/routers/clients/profiles.py` (create_client), `back/routers/clients/subscriptions.py`.
**Что конкретно сделать:**
1. Если пришёл `membership_id`: найти `SubscriptionPackage` студии (чужой/неактивный → 404/400)
   → создать `ClientSubscription` (как `subscriptions.py:70–77`).
2. Если `is_membership_paid == True`: как в `sell_subscription` — `Operation` (type="in",
   категория «Абонементы», `client_id`, счёт) + `account.balance += price` +
   `accrue_points`. Плюс `ClientPayment` (amount, description=имя пакета, status="success") —
   именно из неё лента событий берёт «оплатил» (`profiles.py:321–333`) и считается
   «Потрачено» (`:61`).
3. Счёт для оплаты: счёт студии с `type == "cash"` (касса), нет — первый счёт.
   `# ponytail: счёт-касса по умолчанию; выбор счёта в модалке — после MVP.`
4. События: «новый клиент» уже пишется в ленту (`log_activity`, `:477`); добавить
   `log_activity(..., "client", title="Подключён абонемент «...»", entity_type="client",
   entity_id=client.id)` — в ленте клиента появится через механизм CL-5.1.
   Отдельной таблицы `client_events` не заводим — лента выводится из реальных таблиц (CL-5.1).
5. Всё — до единого `commit` в конце: упало что-то одно → не создалось ничего.
**Готово, когда:** создание клиента с оплаченным абонементом одним запросом даёт: клиента,
заметку, активный абонемент, операцию в Финансах (баланс кассы вырос), баллы лояльности,
события в ленте. При ошибке на любом шаге — в БД пусто.

### 8.3 Валидация шагов 1–2 + честный день рождения · 🟡 · 1:15
**Простыми словами:** дальше первого шага пускают с одним именем: телефон/email проверяются
только «если заполнены» (`useClientForm.ts:36–41`), город не проверяется вообще, а день
рождения — свободный текст «12 апреля 1990», который даже не отправляется на сервер.
**Где:** `front/src/pages/dashboard/Clients/hooks/useClientForm.ts`, `AddClientModal.tsx`
(goNext `:173`, canGoNext `:168`, шаги 1–2 `:329–355`, handleFinish `:182`).
**Что конкретно сделать:**
1. `validate(step)`: шаг 1 — имя (≥2 символа), телефон (обязателен + ≥6 цифр), email
   (обязателен + нормальная regexp, не `includes('@')`); шаг 2 — город обязателен.
   Пустые поля → красная рамка `#D88C9A` + текст под полем (компонент `Field` уже умеет
   `error` — `AddClientModal.tsx:84–111`).
2. `goNext`/`canGoNext`: не пускать на следующий шаг, пока шаг невалиден (сейчас
   `canGoNext` проверяет только имя).
3. День рождения: свободный текст → нативный `<input type="date">`; в `handleFinish`
   отправлять `birth_date` (сейчас теряется).
4. Тексты ошибок — через `t('clients:…')` (словарь из CL-1).
**Готово, когда:** с пустым телефоном/email на шаг 2 не пройти, с пустым городом — на шаг 3;
дата рождения сохраняется и видна в профиле клиента.

### 8.4 Шаг 3: заметка выше, реальные пакеты, «уже оплатил?» · 🟡 · 1:30
**Простыми словами:** на шаге 3 выдуманные карточки «8/10/12 занятий»
(`AddClientModal.tsx:360–383`), выбор (`abMax`) никуда не отправляется, а заметка стоит
ниже абонемента (`:385–403`). Показываем реальные пакеты студии и спрашиваем про оплату.
**Что конкретно сделать:**
1. Поле «Заметка» перенести выше секции абонементов.
2. Пакеты: `useQuery(subscriptionPackages)` → `loyaltyApi.getSubscriptionPackages()`
   (ключ добавить в `queryKeys.ts`); карточки — реальные `name` / `class_count` / `price`
   (цена — через `formatMoney` + валюта студии из CL-1.2), только `is_active`.
   Выбор опционален (кнопка «Без абонемента»); `abMax` из `useClientForm` →
   `membershipId: number | null`.
3. При выбранном пакете снизу появляется вопрос «Клиент уже оплатил этот абонемент?» —
   две радиокнопки Да / Нет (дефолт — Нет) → `is_membership_paid`.
4. Пустой список пакетов — подсказка «Создайте абонементы в Лояльности» со ссылкой.
5. Сводка шага 4: показывать имя выбранного пакета вместо «N занятий» (`:444`).
   Бонус-находка: декоративный тумблер «SMS-приветствие» (`:455–467`) ничего не шлёт —
   убрать до появления SMS-канала.
**Готово, когда:** на шаге 3 — только реальные пакеты студии; выбранный пакет и ответ
об оплате уходят на сервер; заметка — над абонементами.

### 8.5 Сохранение: мутация + success-тост + инвалидация · 🟢 · 0:30
**Простыми словами:** сейчас успех закрывает модалку молча, а ошибка молча глотается
(`AddClientModal.tsx:197–200` — пустой catch). После CL-2 создание — это `useMutation`;
довешиваем тост и инвалидацию.
**Что конкретно сделать:**
1. `handleFinish` → мутация create (из CL-2.2): `onSuccess` — success-тост
   («Клиент добавлен»), закрыть модалку, инвалидация `clients` + `clientCategories`
   (новый клиент виден без F5).
2. Ошибка → error-тост с `detail` сервера (`errorMessage`), модалка **не** закрывается,
   введённые данные не теряются (422 по обязательным полям сюда же).
3. 402 (лимит тарифа) обрабатывает глобальная модалка апселла — тостом не дублировать
   (как в Staff, V2-1 задача 7).
**Готово, когда:** успех — тост и клиент сразу в списке; занятый сервером кейс/ошибка —
понятный тост, данные формы целы.
