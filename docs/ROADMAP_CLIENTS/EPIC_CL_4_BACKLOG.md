### 4.1 График активности на реальных данных · 🟡 · 1:30
**Простыми словами:** эндпоинт `GET /clients/{id}/activity` уже считает визиты и оплаты
по месяцам за год (`profiles.py:385–439`) — а компонент `ActivityChart`
(`ClientProfileSlider.tsx:208–343`) рисует `Math.random`, вымышленные недели и текст
«посещает 2.5 раза в неделю, отток крайне низкий».
**Что конкретно сделать:**
1. `useQuery(clientActivity(id))`; периоды — «3 мес» / «6 мес» (по требованию) — срез
   последних N элементов массива. Режим «1 мес» по неделям убрать (данные месячные;
   недельная гранулярность — после MVP, в `docs/BACKLOG/`).
2. Мини-бары свёрнутого вида — из тех же реальных 12 месяцев, `Math.random` удалить.
3. Блок «инсайта» с выдуманным текстом: удалить, либо считать честно из данных
   (среднее визитов в месяц за период). Ничего выдуманного не оставлять.
**Готово, когда:** у клиента без визитов график пустой; числа совпадают с лентой событий.

### 4.2 Быстрая запись: листание дат ←/→ · 🟢 · 0:45
**Простыми словами:** панель записи уже показывает реальные занятия выбранного дня и
реально записывает. Не хватает только листания дальше ближайших 7 дней
(требование: «с возможностью листать даты вперед/назад»).
**Где:** `ClientProfileSlider.tsx:775–787` (карусель), `useClientActions.ts:35,134–143`.
**Что конкретно сделать:**
1. `bookingDate: number` (индекс 0–6) → абсолютная дата; стрелки ←/→ по краям карусели
   сдвигают окно на 7 дней. Назад раньше сегодняшнего дня не листаем.
2. Остальное (загрузка занятий дня, выбор, подтверждение) — уже работает, не трогать.
**Готово, когда:** можно записать клиента на занятие через 3 недели.
