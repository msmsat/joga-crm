# Эпик CL-5 — Клиенты: события, статусы, бонусы

Цель эпика простыми словами: лента событий уже собирается из реальных таблиц, но знает
только три типа (визит, оплата, заморозка абонемента) — «записался», «отменил» и «бонус»
выводимы из тех же таблиц без миграций. Кнопка «Удалить» сносит клиента мгновенно без
подтверждения. Из трёх бонусов реален один. Телеграм-уведомление о бонусе — делаем
честно: по `tg_id` (Bot API не умеет писать по номеру телефона).

**Стартовая точка лучше, чем кажется:**
- Кнопки «Заморозить» и «Удалить» в панели **уже есть и работают** — чинится только
  отсутствие подтверждения.

  (`loyalty.py:72–82`), начислений может быть сколько угодно.
- У `Reservation` уже есть `created_at` и `cancelled_at` (`back/models/schedule.py:67,69`).
- Лента студии `ActivityLog` + `log_activity` (`back/activity.py`) — готовая
  инфраструктура для логирования заморозки.
- `Client.tg_id` уже в БД (`back/models/client.py:14`), `TG_BOT_TOKEN` — в `.env.example`,
  `aiogram` — в `requirements.txt`.

Источник: требования от 2026-07-17, эпик 5. Обзор — в `ROADMAP.md`.

### 5.1 Лента событий: записался/отменил/бонус + лог заморозки · 🟡 · 2:00
**Простыми словами:** события уже собираются из реальных таблиц, но только три типа:
визит (attended), оплата, заморозка абонемента (`profiles.py:293–351`). «Записался» и
«отменил» выводимы из тех же резерваций — у `Reservation` уже есть `created_at` и
`cancelled_at`. Бонусы — из `LoyaltyPointTransaction`. Единственное, что сейчас нигде
не фиксируется по времени — заморозка самого клиента.
**Где:** `back/routers/clients/profiles.py` (get_client_events, freeze_client),
`back/schemas/clients/responses.py` (EventRecordOut), фронт: `clients.types.ts`,
`EventIcon` и вкладки фильтра в слайдере.
**Что конкретно сделать:**
1. В `get_client_events` добавить типы: `booking` (все Reservation по `created_at`),
   `cancel` (Reservation со `status == "cancelled"` по `cancelled_at`),
   `bonus` (LoyaltyPointTransaction: описание + баллы).
2. Заморозка клиента: в `freeze_client` (`profiles.py:507–520`) добавить
   `log_activity(db, studio_id, "freeze", title=…, entity_type="client", entity_id=id)`
   (инфраструктура `back/activity.py` уже есть) — и читать эти записи ActivityLog в events
   как тип `freeze`.
3. Фронт: расширить `EventRecord.type`, иконки и вкладки фильтра (`booking`/`cancel`/`bonus`);
   фильтр `event_type` на бэке принимает новые значения.
4. ponytail: лента — union по 5 таблицам в памяти; если объёмы вырастут — единый лог
   `client_events`.
**Готово, когда:** запись из панели, её отмена в Журнале, оплата, бонус и заморозка —
все видны в ленте с датами.

### 5.2 Удаление через ConfirmModal · 🟢 · 0:45
**Простыми словами:** кнопки «Заморозить» и «Удалить» в панели уже есть и работают, но
«Удалить» сносит клиента **мгновенно без подтверждения**: `onDelete` сразу зовёт
`clientsApi.delete` (`Clients.tsx:75–86`), а `DeleteClientModal` — мёртвый код
(`deleteTarget` из панели никогда не ставится, `:157–162`).
**Что конкретно сделать:**
1. Клик «Удалить» → общий `ConfirmModal` (`front/src/components/ui/ConfirmModal.tsx`,
   создан в V3-2) с именем клиента → подтверждение → мутация delete.
2. `DeleteClientModal.tsx` и стейт `deleteTarget` удалить.
**Готово, когда:** удалить клиента случайным кликом невозможно; после подтверждения —
success-тост, список и счётчики обновились.

### 5.3 Бонусы: убрать бутафорию, все варианты — в историю · 🟡 · 1:00
**Простыми словами:** `POST /clients/{id}/bonus` уже начисляет баллы и пишет транзакцию,
начислений может быть сколько угодно. Но в UI из трёх вариантов реален один
(«+500 баллов») — «скидка 10%» и «подарочное занятие» показывают тост и ничего не делают
(`useClientActions.ts:162–175`).
**Что конкретно сделать:**
1. Варианты с баллами (`BONUS_OPTIONS`) — все через `addBonus(amount, description)`;
   можно расширить пресеты (+200/+500/+1000).
2. «Скидка 10%» и «подарочное занятие»: механики скидок на странице нет и строить её
   здесь не будем (живёт в Лояльности) — либо убрать из списка, либо фиксировать
   транзакцией с 0 баллов и описанием (след в истории и ленте). Решить при реализации;
   полноценные механики — в `docs/BACKLOG/`.
3. Успех → инвалидация `client(id)` (баланс/уровень) + `clientEvents(id)`; стейт-костыль
   `bonusPoints` удалить.
**Готово, когда:** каждый выбранный бонус оставляет след в истории баллов и ленте событий;
баланс в панели обновляется без F5.

### 5.4 Телеграм-уведомление о бонусе · 🟡 · 1:30
**Простыми словами:** требование «отправлять в Telegram по номеру клиента» упирается в
честное ограничение: **Bot API не умеет писать по номеру телефона** — только по `chat_id`
пользователя, который сам начал диалог с ботом. Поле для этого уже есть — `Client.tg_id`
(наполняется клиентским ботом/мини-приложением второй очереди). MVP: есть `tg_id` — шлём,
нет — молча пропускаем.
**Где:** `back/services/notifier.py`, `back/routers/clients/loyalty.py` (add_bonus),
`.env` (`TG_BOT_TOKEN` — уже описан в `.env.example`).
**Что конкретно сделать:**
1. В `notifier.py` — функция `send_telegram(chat_id, text)`: один POST на
   `api.telegram.org/bot<token>/sendMessage` (aiogram уже в requirements — можно через
   `Bot.send_message`; либо `requests` через `asyncio.to_thread`, чтобы не блокировать loop).
   Вся отправка в try/except с логом — ошибка Телеграма не валит начисление.
2. В `add_bonus` после `commit`: если у клиента есть `tg_id` — отправить
   «Вам начислено N баллов: <описание>».
3. ponytail: один глобальный бот и прямая синхронная отправка; per-studio боты и
   очередь — после MVP.
**Готово, когда:** клиенту с заполненным `tg_id` приходит сообщение; без `tg_id`
начисление работает как раньше; упавший Телеграм не ломает запрос.
