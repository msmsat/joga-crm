## Аудит 3: Журнал (Journal) — обсуждено 2026-07-14

**Общее решение аудита (принято 2026-07-14): режим черновика удаляем.**
Журнал редактируют только владелец и администратор, тренер — только просмотр.
Все правки применяются сразу («онлайн»), с тостом-отменой (5 сек) и историей
Undo/Redo (стрелки + Ctrl+Z / Ctrl+Y). Подробности — п.3.

### 1. Языки: страница целиком на выбранном языке

**Сейчас:** Журнал — единственная большая страница **вообще без i18n**: ни одного
`t()` во всей папке `front/src/pages/dashboard/Journal/`. Русский хардкод везде:
- `Toolbar.tsx` — «Сегодня», «Тренеры», «Залы», «День», «Неделя»;
- `DaySummary.tsx` — «занятий сегодня», «записей клиентов», «средняя загрузка»,
  «ожидает подтв.», «тренеров активно», «Режим черновика», «Опубликовать» и т.д.;
- `NewBookingModal.tsx` — «Новое занятие», «Название программы», «Локация / Зал»,
  «Начало», «Конец», «Лимит группы», «Назначить тренера», «Создать занятие»,
  плейсхолдер «Например: Пилатес Реформер»;
- `BookingPopup.tsx` — «Подтверждено», «Ожидает», «мест занято», «Заполненность»,
  «Записаны», «Добавить», «Изменить», «Сохранить», тайтлы кнопок;
- `AddClientModal.tsx` — «Добавить клиента», «Поиск по имени или телефону...»,
  «Клиенты не найдены», «Выбрано: N клиент(а)»;
- `RightPanel.tsx` — «Залы», «Загрузка», «Ближайшие», + дата через
  `toLocaleDateString('ru-RU')` жёстко;
- `Grid.tsx` — «зан. · чел.», «Загрузка: N%», «занятий на сегодня», «Новое занятие»;
- `constants.ts` — `MONTH_NAMES`, `DAY_NAMES_SHORT` русскими строками;
- все тосты в `Journal.tsx` («Занятие добавлено», «Занятие перенесено»,
  «Неверный формат даты»…) — русские строки.

**Решение:**
- Новый namespace `journal.json` (ru + en) в `front/src/locales/`, все строки
  страницы (включая модалки, тосты, плейсхолдеры, title-атрибуты) → через `t()`.
- Месяцы/дни недели — не константами, а через `toLocaleDateString(i18n.language)`
  (или ключи в locales) — везде, где сейчас `MONTH_NAMES`/`DAY_NAMES_SHORT`/`'ru-RU'`.
- Ошибки API — по схеме Аудита 2 п.1: известные статусы переводим, осмысленный
  `detail` (409, валидация) показываем как есть.

### 2. Данные обновляются сами, без F5 — TanStack Query

**Сейчас:** `useSchedule.ts` — локальный стейт + разовый fetch:
- тренеры и залы грузятся один раз на маунте; поменял зал в Каталоге — Журнал
  не в курсе до F5;
- занятия перезагружаются только при смене видимого диапазона дат;
- второй администратор добавил занятие — у первого его нет до F5;
- все правки локально патчат стейт (`setBookings`) — источник правды расходится
  с сервером до следующего захода.

**Решение (продолжение общего решения V3 №1):**
- Ключи: `['journal-lessons', dateFrom, dateTo]`, `['staff']`, `['halls']` —
  общие с Сотрудниками/Каталогом, изменения там сами обновят Журнал.
- Мутации (создать/перенести/растянуть/изменить/отменить занятие, записать/снять
  клиента) — `useMutation` с оптимистичным апдейтом кэша + инвалидация ключа.
- `refetchInterval` ~30–60 с + refetch on window focus — чужие правки
  подтягиваются сами; этого достаточно, websocket не заводим.

### 3. Черновик удаляем: прямое редактирование + Undo/Redo + тост-отмена

**Сейчас:** `isDraftMode` — чисто клиентский тумблер (`DaySummary.tsx`):
- без черновика нельзя ничего: создание по клику в пустой слот, drag, resize,
  кнопки «Изменить»/«Удалить» в попапе — всё за гейтом `isDraftMode`;
- «Опубликовать» — **фейк**: просто выключает флаг и показывает тост «График
  успешно опубликован!», на сервер ничего не уходит;
- при этом drag/resize внутри «черновика» коммитятся на сервер сразу
  (`commitBookingChange`) — то есть «черновик» уже давно ложь, а не черновик.

**Решение (принято 2026-07-14):**
- `isDraftMode`, статус-бейдж «Режим черновика / Рабочий график», кнопки
  «Изменить график»/«Отменить»/«Опубликовать» — **удалить**. Все гейты
  `isDraftMode` в `Grid.tsx`, `BookingCard.tsx`, `BookingPopup.tsx` снять:
  владелец и админ всегда редактируют напрямую.
- **Тренер — только просмотр**: для роли teacher скрыть создание/drag/resize/
  редактирование/отмену в UI **и ужесточить бэк** (сейчас
  `back/routers/schedule/lessons.py` позволяет тренеру создавать и править свои
  занятия — для POST/PATCH/cancel роли teacher вернуть 403).
- **Тост-отмена:** после каждой правки — единый тост (UI-кит) с кнопкой
  «Отменить» и таймером 5 сек (полоска прогресса). «Отменить» = undo этой правки.
- **Undo/Redo:** стрелки ← → на месте удалённого статус-бейджа в `DaySummary`
  + Ctrl+Z / Ctrl+Y (и Ctrl+Shift+Z). История — in-memory стек пар
  {undo, redo} на сессию страницы; уход со страницы просто очищает стек —
  всё уже сохранено на сервере, терять нечего.
- Механика undo по типам операций:
  - перенос / растягивание / редактирование — обратный PATCH со старыми полями;
  - создание — новый эндпоинт `DELETE /schedule/lessons/{id}` (жёсткое удаление,
    только если нет активных записей; нужен именно для undo создания);
  - отмена занятия — каскад снимает записи клиентов и шлёт им уведомления,
    обратной операции нет → **отложенный коммит**: запрос уходит после
    истечения тоста (5 с), «Отменить» в тосте просто гасит таймер;
  - запись/снятие клиента — обратные `cancelReservation`/`createReservation`.

### 4. «Ничего не сохраняется» — где Журнал лжёт про сохранение

**Сейчас:** создание, перенос, растягивание и отмена реально сохраняются
(`scheduleApi` → PATCH/POST, после F5 на месте). Не сохраняются две вещи:
- **Главный баг:** «Сохранить» в форме редактирования попапа
  (`BookingPopup.tsx:382-398`) меняет **только локальный стейт** (`setBookings`)
  — ни название, ни время, ни зал, ни лимит мест не уходят на сервер. После F5
  всё откатывается. Это и есть «создал/поправил — перезапустился — ничего нет».
- «Опубликовать» — фейковый тост (закрывается п.3 — кнопка удаляется).

**Решение:**
- Сохранение формы попапа → `scheduleApi.updateLesson` с diff-полями
  (`name`, `start_time`, `duration_min`, `hall_id`, `total_spots`) — по образцу
  уже написанного `commitBookingChange` в `Journal.tsx:191`; при ошибке — откат
  + error-тост. Встроить в undo-историю (п.3).
- Заодно: в edit-форме зал хардкодом `'Зал 1'` как дефолт (`BookingPopup.tsx:40`)
  — инициализировать реальным залом занятия.

### 5. Модалка вылезает за рамки / рисуется под панелями

**Сейчас:** `NewBookingModal` и `BookingPopup` рендерятся внутри дерева Журнала,
а контент-обёртка `DashboardLayout.tsx:583-584` имеет `position: relative;
z-index: 1` — это **запирает модалку в контексте наложения уровня 1**, и её
`z-index: 9999` бессилен против сайдбара/топбара (z-index 100). Плюс расчёт
позиции (`usePopupPosition.ts`) считает от `window.innerWidth`, не зная про
левый сайдбар: при одном тренере колонка широкая, форма выбирает левую сторону
и уезжает в зону меню (`finalX` клампится к `SAFE=16` — это координата под
сайдбаром).

**Решение:**
- Обе плавающие панели (`NewBookingModal`, `BookingPopup`) рендерить через
  `createPortal(…, document.body)` — вне контекста `.content`.
- В `usePopupPosition` ограничивать X не от 0, а от правого края сайдбара
  (реальная ширина через `getBoundingClientRect()` элемента меню, не константой),
  и учесть уже существующие константы `RIGHT_PANEL_W`/`TOOLBAR_H` так же честно
  (сейчас захардкожены числами — брать из DOM).
- Проверить и Journal-тост: он тоже в дереве `.content` — глобальный тост из
  UI-кита (п.9) снимает вопрос.

### 6. Живое превью данных при создании/редактировании

**Сейчас — проводка оборвана:** превью-карточка в сетке (`Grid.tsx:289`) рисует
`newForm.title`, но из `Journal.tsx:426` ей передаётся **статичный объект**
`{ title: '', hall: …, maxClients: '8' }` — реальный стейт формы живёт локально
внутри `NewBookingModal`. Что бы ты ни печатал, превью показывает «Новое занятие».
Время (начало/конец) при этом синхронизируется — через `newBookingSlot`.

**Решение:**
- Поднять стейт формы: `NewBookingModal` отдаёт изменения наверх (колбэк
  `onFormChange` или стейт формы целиком в `Journal.tsx`), превью в сетке рисует
  название/зал/лимит по мере ввода.
- То же для редактирования: пока открыта edit-форма попапа, карточка занятия в
  сетке живо отражает вводимые название/время/зал (сейчас карточка меняется
  только после «Сохранить»).

### 7. Аудит честности: эндпоинты и моки

**Эндпоинты — проверены, все реальные и корректно скоупятся** (студия — 404 на
чужое, тренер — 403/только свои, через `get_scoped_lesson`):

| Фронт (`schedule.api.ts`) | Бэк | Статус |
|---|---|---|
| GET `/schedule/lessons?date_from&date_to` | `lessons.py` list_lessons | ✅ окно дат, тренер видит только свои, booked_count без N+1 |
| GET `/schedule/lessons/{id}` | get_lesson | ✅ + список записанных |
| POST `/schedule/lessons` | create_lesson | ✅ валидация мест 1–50, тренер в студии, зал в студии |
| PATCH `/schedule/lessons/{id}` | update_lesson | ✅ diff-поля, мест не меньше записанных |
| PATCH `/schedule/lessons/{id}/cancel` | cancel_lesson | ✅ каскад отмены записей + уведомления |
| GET `/schedule/halls` | halls.py | ✅ только чтение, активные залы студии |
| POST `/schedule/reservations` | reservations.py | ✅ лимит мест, 409 на дубль |
| PATCH `/schedule/reservations/{id}/cancel` | ✅ | освобождает место |
| PATCH `/schedule/reservations/{id}/attend` | ✅ | идемпотентно, списывает абонемент, last_visit |
| `createHall`/`updateHall` в `schedule.api.ts` | ❌ **нет на бэке** | мёртвые методы фронта — удалить (залы живут в Каталоге) |

**Найденные моки и ложь на странице:**
- **AI-чат журнала (`AiChat.tsx`) — имитация**: захардкоженный ответ «Пока я
  работаю в демо-режиме 🤖» через `setTimeout`. Решение: убрать локальный чат и
  открывать глобальный `AIDrawer` (он уже ходит в реальный `/ai`), либо
  подключить `aiApi` сюда же. Дубли чатов не держим.
- **Мини-календарь: фейковые точки событий** — `hasEventDays = [3, 7, 9, …]`
  захардкожен (`RightPanel.tsx:27`). Решение: точки от реальных занятий месяца
  (данные уже грузятся при `visibleRange('month')` — либо лёгкий счётчик
  занятий по дням), иначе убрать.
- **Метрика «ожидает подтв.» всегда 0** — бэк создаёт занятия только со
  статусом `confirmed`, `pending` не существует. Решение: метрику убрать
  (статусы подтверждения — не в этой итерации), не рисовать декорацию.
- **`CLIENTS_DB` в `constants.ts` — мёртвый мок** (нигде не используется,
  `AddClientModal` давно на реальном `clientsApi`). Удалить.
- **Вид «Месяц» — недоделан и недостижим**: в `Toolbar.tsx` кнопки нет (только
  День/Неделя), но тип `'month'` протащен через все компоненты, а
  `visibleRange('month')` грузит занятия месяца, которые сетка свалит в один
  день. Решение: выпилить `'month'` из типов и `visibleRange` — до тех пор,
  пока месячный вид не спроектирован по-настоящему.
- **Колонки журнала = все сотрудники**, включая владельца и админов без занятий
  (`staffApi.getList()` без фильтра). Решение: фильтровать по роли
  тренер/имеющие занятия в диапазоне; владельца-тренера не терять.
- Занятия вне сетки 07:00–22:00 не отобразятся (константа `TIMES` + капы `15`
  в drag). Пока фиксируем как известное ограничение; диапазон часов брать из
  часов работы студии — отдельной задачей.

### 8. Переключение страниц без «лага» — мгновенно и плавно

**Сейчас:** заход в Журнал = пустая сетка без колонок → тренеры/залы приехали →
колонки появились → занятия впрыгнули. Три волны вёрстки. Листание дней тоже
мигает пустотой (250 мс `withAnimation` + fetch наперегонки).

**Решение (общее V3 №11 + журнальное):**
- TanStack Query кэш: повторный заход рисует прошлые данные мгновенно, обновление
  тихо в фоне.
- Первый заход — скелетон сетки в дизайн-системе: колонки-плейсхолдеры + пара
  призрачных карточек, никакой пустоты.
- Листание дней/недель — `placeholderData: keepPreviousData`: старая сетка
  остаётся до прихода новой, анимация `withAnimation` играет поверх данных,
  а не поверх пустоты.
- Префетч соседних дней (вчера/завтра или соседняя неделя) — листание почти
  всегда мгновенное.

### 9. Ошибки видимы всегда; тост/модалки/селекты — единые (UI-кит)

**Сейчас:**
- Ошибки **загрузки** глотаются: оба `catch` в `useSchedule.ts` →
  `console.error` — при 403/500/сети пользователь видит просто пустой журнал.
- Тост Журнала — **шестая** локальная реализация в проекте (после Finances,
  Settings, Profile, Staff-inline, Catalog); только success-вид, ошибки
  показываются тем же зелёным чеком.
- Валидации в формах нет: пустое название просто молча не создаёт (return без
  сообщения), лимит мест можно ввести 0 или 999 — узнаешь 400-й ошибкой сервера.
- Подтверждение отмены занятия отсутствует вовсе (клик по корзине — сразу
  отмена с каскадом по клиентам).
- Выпадающие списки времени (`kp-time-dropdown`) — свои, отличаются от
  селектов AI/Booking/Settings.

**Решение (распространение V3 №5/№12 на Журнал):**
- Локальный тост Журнала удалить → глобальный `Toast` из UI-кита
  (`front/src/components/ui/`), варианты success / error / **undo (с таймером
  5 с и кнопкой)** — undo-вариант добавить в кит, он понадобится и другим
  страницам.
- Все ошибки API (400 «мест меньше записанных», 409 «уже записан», 403, 404,
  сеть) → error-тост с текстом сервера; ошибки загрузки — тост + состояние
  «не удалось загрузить, повторить» на сетке.
- Клиентская валидация до отправки: название обязательно, лимит группы 1–50
  (зеркало бэка), конец позже начала — подсветка поля + сообщение.
- Отмена занятия с записанными клиентами — единый `ConfirmModal` из кита
  («Отменить занятие? N клиентов получат уведомление»); без записанных —
  достаточно undo-тоста.
- Дропдауны времени и прочие мелкие селекты — на единый `Select`/стили кита.

## Аудит 3.1: Журнал — жизненный цикл занятия (дополнение, обсуждено 2026-07-19)

Находки ручного тестирования после завершения V4-1…V4-4. Нумерация продолжает
аудит 3.

### 10. Правила времени: расписание нельзя дёргать «впритык»

**Сейчас:** бэк принимает любые даты — занятие можно создать задним числом,
изменить или отменить за минуту до начала.

**Решение (MVP, только бэк):** три простые проверки в
`back/routers/schedule/lessons.py` — создание минимум за 3 часа до начала,
изменение и отмена (перенос, длительность) минимум за 2 часа; позже — 400 с
человекочитаемым `detail`. Фронту ничего не надо: error-тост и откат
оптимистичных правок уже работают (V4-2).

### 11. Отменённые занятия: видимость, причина, статус уведомлений

**Сейчас:** отменённые занятия приходят с API и карточка получает класс
`.booking-card.cancelled`, но CSS-правил под него нет — карточка выглядит
живой, drag/resize работают, причины отмены не существует в принципе.

**Решение:** поля `cancel_reason` + `clients_notified` у Lesson (зеркало
`Reservation.cancellation_reason`); модалка отмены с опциональной причиной;
отменённая карточка — полупрозрачная серая, без drag/resize; клик — инфо-вид
в попапе: причина (редактируемая) + «Уведомления отправлены: Да/Нет»; метрики
дня отменённые не считают.

### 12. Умные confirm и один тост на правку

**Сейчас:** confirm есть только при отмене занятия с клиентами; перенос,
resize и правка формы проходят молча. Плюс баг: на drag & drop выстреливают
**два** тоста (`useDragAndDrop` тостит сам и второй раз — `commitBookingChange`
после успеха).

**Решение:** guard в единой воронке `commitBookingChange` (через неё идут все
три пути правки) — занятие с клиентами правится только после подтверждения
«Клиентам будет отправлено уведомление»; тост из `useDragAndDrop` убрать,
точный текст прокинуть в `onCommit` — один тост после ответа сервера.

### 13. Уведомления об изменении занятия + честный флаг

**Сейчас:** фасад `notify()` (`back/services/notifier.py`) уже существует и
зовётся при отмене (событие c3), но изменение занятия клиентов не уведомляет,
и результат рассылки нигде не фиксируется.

**Решение (без нового фасада — дублей не заводим):** событие c11 «Изменение
занятия» (шаблон в `_render` + строка в матрице на фронте), вызов в
`update_lesson` при смене времени/длительности/зала; `notify()` возвращает
bool → агрегат в `Lesson.clients_notified` (и для отмены тоже).

### 14. Занятие строго ссылается на услугу (Data Integrity)

**Сейчас:** название занятия — свободный текст; `service_id` в модели есть,
но опционален и квик-формой не заполняется — аналитика по услугам слепая.

**Решение:** `service_id` обязателен в схемах создания/редактирования (в БД
колонка остаётся nullable — старые занятия легальны), `name` денормализуется
из услуги (как `teacher_name`), выдача расписания отдаёт цвет услуги через
JOIN. На фронте в обеих формах вместо текстового поля — `Select` услуг студии
(кэш общий с Каталогом) с пунктом «+ Создать услугу» → ConfirmModal →
`useNavigate` в Каталог (пункт только владельцу — Каталог за `OwnerRoute`).
