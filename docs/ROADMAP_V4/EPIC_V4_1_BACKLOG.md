# Эпик V4-1 — Честный Журнал: сохранение, прямое редактирование, роли

Цель эпика простыми словами: Журнал должен перестать врать. Сегодня форма
редактирования занятия «сохраняет» только в память браузера (после F5 всё
откатывается), кнопка «Опубликовать» показывает победный тост, за которым ничего
нет, а «режим черновика» — фикция: перенос и растягивание внутри «черновика» и
так коммитятся на сервер сразу. Эпик чинит сохранение, выкидывает черновик
целиком (решение владельца 2026-07-14: редактирование всегда прямое), закрывает
журнал от правок тренера (UI и сервер) и выметает мёртвые моки.

Источник: аудит 3 в `docs/ROADMAP_V4/ROADMAP.md`, пп. 3, 4, 7 (обсуждено 2026-07-14).

**Важно понимать про старт:**
- Главный баг подтверждён кодом: «Сохранить» в `BookingPopup.tsx:382-398` делает
  только `setBookings(...)` — ни одного вызова `scheduleApi`. При этом рядом в
  `Journal.tsx:191` уже есть правильный образец (`commitBookingChange`: diff →
  PATCH → откат при ошибке) — его и переиспользуем, не изобретаем второй путь.
- «Черновик» — чисто клиентский флаг `isDraftMode`, серверного состояния у него
  нет (подтверждено `docs/TZ/STATUS.md`: «черновик остался чисто клиентским
  режимом»). Удаление флага ничего не ломает на бэке.
- Бэкенд сейчас **разрешает** тренеру создавать/править/отменять свои занятия
  (`back/routers/schedule/lessons.py:174, 233`) — по новому решению это
  закрывается (тренер — только просмотр).
- Этот эпик первый в очереди V4: он меняет те же компоненты, которые дальше
  трогают V4-2 (Query) и V4-4 (порталы/превью) — сначала убираем лишнее, потом
  строим поверх.

**Обозначения сложности:** 🟢 простая · 🟡 средняя · 🔴 сложная, делать внимательно.

## Фаза 1 — Сохранение

### 1. Форма попапа реально сохраняет · 🔴 · 1:30
**Простыми словами:** «Сохранить» в форме редактирования занятия должен отправлять
изменения на сервер — сейчас он молча меняет только локальный стейт, и после F5
название, время, зал и лимит мест откатываются.
**Зачем:** это и есть жалоба владельца «занятие создал/поправил — перезапустился —
ничего нет» (аудит 3, п. 4). Создание при этом сохраняется честно — ложь именно
в редактировании.
**Где:** `front/src/pages/dashboard/Journal/components/BookingPopup.tsx:382-398`
(обработчик «Сохранить»), `front/src/pages/dashboard/Journal/Journal.tsx:191`
(`commitBookingChange` — образец и место расширения).
**Что конкретно сделать:**
1. Расширить `commitBookingChange` полями `name` (title) и `total_spots`
   (maxClients) — сейчас он умеет только время/длительность/тренера/зал.
2. Из попапа не дёргать API напрямую: пробросить колбэк `onSave(prev, next)`
   из `Journal.tsx` (попап знает залы только по именам — конвертация имя → id
   живёт наверху, рядом с `halls`).
3. Оптимистично применить в стейт (как сейчас), при ошибке сервера — откатить
   карточку и показать тост с текстом ошибки (образец отката уже в
   `commitBookingChange`).
4. Ошибка 400 «Мест не может быть меньше числа записанных (N)» от бэка
   (`lessons.py:225`) должна доходить до пользователя дословно.
**⚠️ Подводные камни:**
- `parseTimeToIndex` не ограничивает ввод рамками сетки: «06:00» даст
  отрицательный индекс и `start_time` вне диапазона — заклампить в границы
  `TIMES` при коммите (полноценная валидация форм — эпик V4-4, здесь только
  защита от заведомо кривого PATCH).
- В payload включать только реально изменённые поля (diff), как это уже делает
  `commitBookingChange` — иначе PATCH перетрёт `teacher_name` и прочее без нужды.
**Готово, когда:** правишь название/время/зал/лимит → F5 → всё на месте; при
падении сервера карточка откатывается и виден тост с причиной.

## Фаза 2 — Без черновика

### 3. Удалить `isDraftMode` и все гейты; выкинуть фейк «Опубликовать» · 🟡 · 1:30
**Простыми словами:** сейчас без включённого «черновика» нельзя ничего: ни
создать занятие кликом в пустой слот, ни перетащить, ни растянуть, ни открыть
редактирование. Убираем тумблер — владелец и админ всегда редактируют напрямую.
**Зачем:** решение владельца (аудит 3, п. 3): «просто начинаешь редактировать и
у тебя автоматом всё меняется». Плюс «Опубликовать» — фейковый тост «График
успешно опубликован!», на сервер ничего не уходит — такого в продукте быть
не должно.
**Где:** `Journal.tsx:40` (стейт), `DaySummary.tsx:53-113` (бейдж
«Режим черновика / Рабочий график», кнопки «Изменить график» / «Отменить» /
«Опубликовать»), `Grid.tsx:212-228` (гейты hover и создания по клику),
`BookingCard.tsx:47, 65, 76, 104` (гейты drag/resize и курсора),
`BookingPopup.tsx:408-426` (кнопки «Изменить»/«Удалить» за гейтом).
**Что конкретно сделать:**
1. Удалить стейт `isDraftMode` и проп по всей цепочке (Journal → DaySummary,
   Grid, BookingCard, BookingPopup) — TypeScript сам покажет все места.
2. Все условия `isDraftMode && …` заменить на `canEdit` (см. задачу 4; для
   владельца/админа = `true` всегда).
3. Из `DaySummary` удалить статус-бейдж и все три кнопки управления черновиком.
   Переключатель «Шаг сетки» оставить, но показывать всегда (сейчас виден только
   в черновике — `DaySummary.tsx:54`). Освободившееся место справа займут
   стрелки Undo/Redo (эпик V4-3 — оставить понятный TODO-якорь недопустимо,
   просто знать, что место зарезервировано).
4. Тост «График успешно опубликован!» удалить вместе с кнопкой.
**⚠️ Подводные камни:** в `Grid.tsx:224` гейт заодно защищает от создания слота
поверх открытого попапа/драга (`showNewForm || popupBooking || drag ||
wasDragging`) — эту часть условия сохранить, убирается только `isDraftMode`.
**Готово, когда:** клик в пустой слот сразу открывает форму создания; drag,
resize, «Изменить», «Удалить» доступны без всяких режимов; слов «черновик» и
«опубликовать» на странице больше нет.

### 4. Тренер — только просмотр (UI) · 🟡 · 0:45
**Простыми словами:** тренер видит своё расписание, может открыть занятие и
посмотреть записанных, но не может ничего менять: ни создать, ни перетащить,
ни отредактировать, ни отменить, ни записать/снять клиента.
**Зачем:** решение владельца (аудит 3): «тренер редактировать не сможет журнал,
этим будет заниматься только админ и владелец».
**Где:** те же компоненты, что в задаче 3; роль пользователя берём из того же
источника, из которого `DashboardLayout` скрывает пункты меню по ролям.
**Что конкретно сделать:**
1. В `Journal.tsx` завести `canEdit = role !== 'teacher'` и пробросить вместо
   бывшего `isDraftMode` (вся проводка уже сделана задачей 3).
2. Для тренера: карточка занятия открывает попап в режиме просмотра — без
   «Изменить», «Удалить», «Добавить» и без крестиков снятия клиентов; галочку
   «Пришёл» (отметить посещение) тренеру **оставить** — это его рабочий
   инструмент, а не правка графика.
3. Курсор карточки для тренера — `pointer` (не `grab`), пустые слоты не
   подсвечиваются и не кликаются.
**Готово, когда:** под аккаунтом тренера журнал полностью читается, отметка
посещения работает, всё остальное недоступно и не намекает на доступность.

### 5. Тренер — только просмотр (сервер: 403 на правки) · 🟡 · 0:45
**Простыми словами:** ограничение из UI закрепить на сервере — иначе оно
декоративное (ТЗ: «получить чужие данные в обход интерфейса нельзя», это же
касается правок).
**Зачем:** сейчас `back/routers/schedule/lessons.py` разрешает тренеру создавать
занятия на себя (`:174`), править свои (`get_scoped_lesson`) и отменять свои;
`reservations.py` — записывать/снимать клиентов на своих занятиях. По новому
решению всё это — только владелец и админ.
**Где:** `back/routers/schedule/lessons.py` (create/update/cancel),
`back/routers/schedule/reservations.py` (create/cancel; **attend не трогаем**).
**Что конкретно сделать:**
1. В `create_lesson`, `update_lesson`, `cancel_lesson`: роль `teacher` → 403
   («Расписание меняют владелец и администратор»). Ветки «тренер может только
   на себя» (`:174, :233`) при этом умирают — удалить.
2. В `create_reservation`, `cancel_reservation`: аналогичный 403 для `teacher`.
3. `attend_reservation` оставить как есть — тренер отмечает посещения своих
   занятий (скоуп уже проверяет `get_scoped_lesson`).
4. GET-эндпоинты не трогать: тренер по-прежнему видит только свои занятия.
**⚠️ Подводные камни:** `back/routers/staff/schedule.py` («Добавить событие»
сотруднику со страницы Сотрудников) — проверить, что он уже owner-only и не
открывает тренеру обходной путь создания занятий.
**Готово, когда:** прямой curl под токеном тренера на POST/PATCH lessons и
reservations возвращает 403; attend — 200 на своём занятии, 403 на чужом.

### 6. Выпилить недостижимый вид «Месяц» · 🟢 · 0:30

**Где:** `hooks/useSchedule.ts:8-26` (`visibleRange`), типы `'day' | 'week' |

**Что конкретно сделать:** тип сузить до `'day' | 'week'`, ветку `month` из
`visibleRange` удалить, массив слайдера — из двух значений. Месячный вид, если
понадобится, проектируется отдельно как настоящий (сетка по дням, а не колонки
тренеров).
**Готово, когда:** grep `'month'` по папке Journal — пусто; переключатель

### 7. Колонки журнала — тренеры, а не весь штат · 🟡 · 0:45
**Простыми словами:** сейчас `useSchedule` берёт `staffApi.getList()` целиком —
владелец и администраторы становятся колонками журнала, даже если не ведут
занятия. По ТЗ 2.3 колонки — тренеры.
**Где:** `hooks/useSchedule.ts:41-43`, маппер `staffToTrainer` в `utils.ts:172`.
**Что конкретно сделать:**
1. Фильтровать сотрудников по роли доступа «Тренер» **плюс** любой сотрудник
   (включая владельца), у которого есть занятия в загруженном диапазоне —
   владелец, ведущий классы, колонку не теряет.
2. Тонкость порядка: список занятий приходит позже списка сотрудников — колонка

цвет от того, что уволили администратора.
**Готово, когда:** админ без занятий не занимает колонку; тренер с занятиями и
владелец-тренер — на месте; цвета занятий стабильны между заходами.

## Definition of Done эпика

- Любая правка занятия из любого места UI сохраняется на сервер и переживает F5.
- Понятия «черновик» в коде и интерфейсе не существует; фейковых кнопок нет.
- Тренер — строго просмотр + отметка посещений, на уровне UI **и** сервера.
- Мёртвый код удалён: `CLIENTS_DB`, `createHall`/`updateHall`, вид `month`.
- Колонки — только тренеры со стабильными цветами; метрик-декораций нет.
- Сборка и линт чистые.
