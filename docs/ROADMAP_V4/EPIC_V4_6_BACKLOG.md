# Эпик V4-6 — Жизненный цикл занятия: правила времени, отмена с причиной, услуги

Цель эпика простыми словами: Журнал научился сохранять и откатывать правки
(V4-1…V4-4), но бизнес-логика жизненного цикла занятия дырявая: расписание
можно менять за минуту до начала, отменённое занятие выглядит как живое и не
несёт причины, правка занятия с записанными клиентами проходит молча, на один
drag прилетает два тоста, а название занятия — свободный текст, который ломает
аналитику по услугам. Эпик закрывает пять находок тестирования от 2026-07-19
максимально прагматично: простые проверки на бэке, одна воронка confirm на
фронте, переиспользование уже написанного (notify-фасад, ConfirmModal,
отложенный cancel) — без новых подсистем.

Источник: аудит 3.1 в `docs/ROADMAP_V4/ROADMAP.md`, пп. 10–14 (обсуждено
2026-07-19).

**Важно понимать про старт:**
- Выполнять **до V4-5** (языки — строго последним): эпик добавляет новые
  строки UI (модалка отмены, confirm-тексты, инфо-вид отменённого занятия) —
  их переводит V4-5, повторный проход не нужен.
- **Фасад уведомлений уже существует** — `back/services/notifier.py`,
  `notify(db, studio_id, role, event_id, context)`: читает матрицу
  «событие × канал», шлёт email, глушит ошибки. Он уже вызывается в
  `cancel_lesson` (событие c3). Никакого нового `notifications.py` не
  создавать — дублей фасада быть не должно; задача 3 только расширяет его.
- **Confirm при отмене занятия с клиентами уже есть** (`Journal.tsx`,
  `confirmCancelBooking` + `ConfirmModal`) — задача 8 расширяет этот путь до
  модалки с причиной, а не строит второй.
- Отложенный cancel с undo-тостом (V4-3) уже живёт (`startDeferredCancel`,
  `deferredCancelRef`, страховки на unmount/beforeunload/листание) — причина
  отмены встраивается в него, механику не трогаем.
- У `Reservation` уже есть `cancellation_reason` — для `Lesson` заводим
  зеркальное поле, имена и размер совпадают.
- Отменённые занятия **уже приходят** с API (`list_lessons` не фильтрует по
  статусу) и карточка **уже получает** класс `.booking-card.cancelled`
  (`BookingCard.tsx:48`) — просто CSS-правил под него нет, поэтому карточка
  выглядит живой, drag/resize работают. Половина задачи 9 — дописать то, что
  почти готово.

**Обозначения сложности:** 🟢 простая · 🟡 средняя · 🔴 сложная, делать внимательно.

## Фаза 1 — Бэк: правила и данные

### 1. Правила времени · 🟡 · 0:45
**Простыми словами:** студия не должна дёргать расписание «впритык»: создать
занятие можно минимум за 3 часа до начала, изменить или отменить — минимум за
2 часа. Позже — блокировка с понятной причиной.
**Сейчас:** бэк принимает любые даты — хоть создание задним числом, хоть
отмену за минуту до начала.
**Где:** `back/routers/schedule/lessons.py` — `create_lesson`,
`update_lesson`, `cancel_lesson`.
**Что конкретно сделать:**
1. `create_lesson`: `body.start_time < now + 3h` → 400 «Создавать занятие
   можно не позднее чем за 3 часа до начала».
2. `update_lesson`: занятие уже начинается через <2 ч **или** новое
   `start_time` попадает в окно <2 ч → 400 «Изменять занятие можно не позднее
   чем за 2 часа до начала».
3. `cancel_lesson`: старт через <2 ч → 400 «Отменять занятие можно не позднее
   чем за 2 часа до начала».
4. Исключение: PATCH, в котором из полей только `cancel_reason` (правка
   причины у уже отменённого/прошедшего занятия, задача 9) — правило времени
   **пропускает**; для остальных полей отменённого занятия PATCH блокировать.
5. `DELETE /lessons/{id}` (undo создания, V4-3) не трогаем — он и так требует
   нуля записей, клиентам от него ничего не приходит.
6. Фронту ничего специально делать не надо: 400 с `detail` уже показывается
   error-тостом, оптимистичные правки уже откатываются (V4-2).
**⚠️ Подводные камни:** `start_time` хранится naive — сравнивать с
`datetime.now()` (сервер и студия в одном поясе — то же осознанное упрощение
MVP, что и во всём бэке). Undo/Redo возле границы: обратный PATCH получит
400 → error-тост, история выкидывает entry (`handleHistoryError` уже есть) —
это ожидаемое поведение, не чинить.
**Готово, когда:** маленький pytest в `back/tests/` (создание <3 ч → 400,
изменение и отмена <2 ч → 400, PATCH только `cancel_reason` — проходит);
в UI попытка утащить занятие, начинающееся через час, откатывается с тостом.

### 2. Поля `cancel_reason` + `clients_notified` + миграция · 🟡 · 0:45
**Простыми словами:** занятию нужно уметь хранить причину отмены и простой
флаг «уведомления клиентам отправлены».
**Где:** `back/models/schedule.py` (Lesson), `back/schemas/schedule/lessons.py`,
`back/routers/schedule/lessons.py`, миграция alembic; зеркало типов —
`front/src/api/schedule/schedule.types.ts`.
**Что конкретно сделать:**
1. `Lesson`: `cancel_reason: Mapped[Optional[str]] = String(300)` (зеркало
   `Reservation.cancellation_reason`) и `clients_notified: Mapped[bool] =
   default False`. Одна autogenerate-миграция.
2. `cancel_lesson` принимает опциональное тело `{reason?: str}` (маленький
   Pydantic-класс `LessonCancelRequest` в тот же `schemas/schedule/lessons.py`)
   и пишет его в `lesson.cancel_reason`.
3. `LessonUpdateRequest`: + `cancel_reason` — правка причины постфактум из
   инфо-вида (задача 9).
4. `LessonRead`: + `cancel_reason`, `clients_notified` — попадут и в список,
   и в деталь. Зеркалить в `schedule.types.ts` и `Booking` (`Journal/types.ts`).
**⚠️ Подводные камни:** тело `cancel` было пустым — фронтовый
`scheduleApi.cancelLesson` должен слать `{}` по умолчанию, чтобы не сломать
существующие вызовы (undo-коммит шлёт причину, страховки — ту же замкнутую).
**Готово, когда:** `alembic upgrade head` проходит; отмена с причиной
сохраняет её в базе, GET отдаёт оба поля.

### 3. Событие «занятие изменено» + честный флаг · 🟡 · 1:00
**Простыми словами:** при переносе/изменении занятия клиентам должно уходить
уведомление (пока — через существующий email-фасад, остальные каналы молчат),
а занятие — запоминать, отработала ли рассылка. Новый фасад не строим —
расширяем `notify()`.
**Где:** `back/services/notifier.py`, `back/routers/schedule/lessons.py`
(`update_lesson`, `cancel_lesson`),
`front/src/pages/dashboard/Notifications/constants.ts` (`NOTIF_EVENTS.client`).
**Что конкретно сделать:**
1. `notify()` возвращает `bool`: `True`, только если письмо реально ушло
   (`send_email` без исключения); все ранние `return` → `False`.
2. `_render`: шаблон `c11` — «Занятие изменено»: название, новое время
   («Занятие „{lesson}“ перенесено — новое время: {when}»).
3. `NOTIF_EVENTS.client`: + строка `c11` (icon `Icon.Clock`, «Изменение
   занятия», «Перенос времени, зала или тренера») — иначе владелец не сможет
   включить событие в матрице.
4. `update_lesson`: если среди изменённых полей есть `start_time`,
   `duration_min` или `hall_id` — собрать `client_id` активных резерваций
   (образец — `cancel_lesson`), вызвать `notify(..., "c11", ...)` по каждому;
   `lesson.clients_notified = any(результатов)`.
5. `cancel_lesson`: результат уже существующих `notify(..., "c3", ...)` тоже
   агрегировать в `clients_notified` (сейчас выбрасывается).
**⚠️ Подводные камни:** `notify` глушит ошибки внутри — это правильно
(рассылка не должна валить PATCH), поэтому флаг честный: «Нет» означает и
«канал выключен в матрице», и «нет email у клиента» — для MVP это одно и то
же «клиенты не уведомлены». `clients_notified` проставлять и коммитить после
цикла notify (он идёт после основного commit — нужен второй короткий commit
только этого поля).
**Готово, когда:** перенос занятия с записанным клиентом при включённом email
в матрице шлёт письмо и ставит `clients_notified = true`; при выключенном —
`false`; PATCH не падает при недоступном SMTP.

### 4. Обязательный `service_id` на бэке · 🟡 · 1:00
**Простыми словами:** занятие больше не «свободный текст», а всегда ссылка на

**Где:** `back/schemas/schedule/lessons.py` (`LessonCreateRequest`,
`LessonUpdateRequest`, `LessonRead`), `back/routers/schedule/lessons.py`
(`create_lesson`, `update_lesson`, `list_lessons`, `get_lesson`).
**Что конкретно сделать:**

2. `create_lesson`: загрузить `Service` по `id + studio_id` (нет/чужая → 404

5. Миграцию NOT NULL на `lessons.service_id` **не делать** — старые строки
   без услуги легальны; обязательность живёт на уровне схемы запроса.
**⚠️ Подводные камни:** сидеры/тесты, создающие занятия через POST, начнут
получать 422 — поправить вместе с задачей. Публичная запись

**Готово, когда:** POST без `service_id` → 422; созданное занятие несёт имя

### 6. Дебаг двойного тоста при drag & drop · 🟢 · 0:20
**Простыми словами:** на перенос занятия выстреливают два тоста. Оставить
один точный, и показывать его после ответа сервера, а не до.
**Сейчас:** `useDragAndDrop.ts:123-126` — `apply()` зовёт `showToast(msg)`
сам **и** `onCommit` → `commitBookingChange` → второй `showToast('Занятие
обновлено')` при успехе (`Journal.tsx:287`).
**Что конкретно сделать:**
1. Из `useDragAndDrop` вызов `showToast` убрать (проп больше не нужен —
   почистить интерфейс и зависимости эффекта).
2. Точный текст прокинуть третьим аргументом: `onCommit(prev, next, label)` —
   «Занятие перенесено», «Время начала изменено», «Время окончания изменено».
3. `commitBookingChange` показывает один тост с этим label после успеха

### 7. Confirm при правке занятия с записанными клиентами · 🟡 · 0:45
**Простыми словами:** любая попытка изменить занятие, где есть клиенты
(форма, drag & drop, resize), — сначала явное предупреждение: «В этом занятии
есть клиенты. Вы уверены, что хотите внести изменения? Клиентам будет
автоматически отправлено уведомление об этом».
**Где:** `Journal.tsx` — `commitBookingChange` (:281): это **единая воронка**,
через которую идут все три пути правки (drag, resize, «Сохранить» формы) —
один guard закрывает всё, по отдельным путям не размазывать.
**Что конкретно сделать:**
1. Стейт `confirmChange: { prev, next, label } | null` (образец —
   `confirmCancelBooking`).
2. В `commitBookingChange`: если `prev.clients > 0` — не запускать мутацию, а
   положить параметры в `confirmChange`; существующий `ConfirmModal` из кита
   (не danger) → подтверждение выполняет отложенную мутацию, отказ — просто
   закрывает (превью drag уже снято, кэш не тронут, откатывать нечего).
3. Занятия без клиентов идут как раньше — без лишнего клика.
**⚠️ Подводные камни:** undo/redo зовут `mutations.updateLesson` напрямую,
мимо `commitBookingChange` — confirm при откатах **не** выскакивает, это
правильно (пользователь уже подтвердил исходную правку). Текст обещает
уведомление — оно реально уходит (задача 3), не врать в модалке до того.
**Готово, когда:** правка занятия с клиентами любым из трёх способов — через
подтверждение; без клиентов — без него; отказ не оставляет следов в кэше.

### 8. Модалка отмены: предупреждение + причина · 🟡 · 1:00
**Простыми словами:** отмена занятия всегда открывает модалку с опциональным

предупреждает об этом.
**Сейчас:** при `clients > 0` — `ConfirmModal` без причины, без клиентов —
мгновенная отмена с undo-тостом. Причину ввести негде.
**Где:** `Journal.tsx` (`deleteBooking`, `confirmCancelBooking`,
`startDeferredCancel`), новая
`components/modals/CancelLessonModal.tsx`, `hooks/useJournalMutations.ts`
(`commitDeferredCancel`), `front/src/api/schedule/schedule.api.ts`.
**Что конкретно сделать:**
1. `CancelLessonModal` на `ModalShell`/`ModalHeader`/`Body`/`Footer` из
   UI-кита: при `clients > 0` — предупреждение («Внимание: на занятие
   записаны N клиентов — они получат уведомление об отмене»), всегда —
   `Input` «Причина отмены (необязательно)», кнопки «Назад» / danger
