# Эпик V4-2 — Журнал без F5: TanStack Query, скелетоны, плавное листание

Цель эпика простыми словами: данные Журнала должны жить сами — обновляться без
F5, не мигать пустотой при листании дней и мгновенно рисоваться при повторном
заходе. Сейчас `useSchedule` — локальный стейт с разовым fetch'ем: поменял зал в
Каталоге — Журнал не в курсе; второй админ создал занятие — у первого его нет;
каждое листание дня — гонка fetch'а с 250-мс анимацией поверх пустоты. Переводим
Журнал на TanStack Query по паттерну, заложенному эпиком V3-1 для Каталога.

Источник: аудит 3 в `docs/ROADMAP_V4/ROADMAP.md`, пп. 2, 8, 9 (частично)
(обсуждено 2026-07-14); общее решение V3 №1 (TanStack Query).

**Важно понимать про старт:**
- **Координация с V3-1:** провайдер и дефолты QueryClient — задача 1 эпика V3-1.
  Если V3-1 ещё не выполнен к старту — провайдер создаётся здесь по спеке V3-1
  (задача 1), а в V3-1 отмечается «сделано в V4-2». Дублей быть не должно.
- Выполнять **после V4-1**: он удаляет черновик и мёртвый вид `month` из тех же
  файлов — мигрировать на Query проще уже почищенный код.
- Ключи `['staff']` и `['halls']` — общие с Сотрудниками и Каталогом: когда те
  страницы мигрируют (их аудиты), инвалидация свяжет всё автоматически.
- Ошибки загрузки сейчас глотаются: оба `catch` в `useSchedule.ts:47, 60` →
  `console.error`, пользователь видит молча пустой журнал. Query отдаст ошибку
  наружу — показываем по-настоящему (задача 6).
- Эпик V4-3 (Undo/Redo) строится **поверх** мутаций из этого эпика — сначала
  мутации, потом история.

**Обозначения сложности:** 🟢 простая · 🟡 средняя · 🔴 сложная, делать внимательно.

## Фаза 1 — Чтение

### 2. `useSchedule` → три `useQuery` · 🟡 · 1:30
**Простыми словами:** переписать ручные `useState`+`useEffect` на три квери;
интерфейс хука для `Journal.tsx` сохранить (рефакторинг = реорганизация,
поведение UI не меняется — кроме исчезновения лагов).
**Где:** `front/src/pages/dashboard/Journal/hooks/useSchedule.ts`.
**Что конкретно сделать:**
1. Ключи: `['staff']` (список сотрудников → тренеры), `['halls']` (залы),
   `['journal-lessons', dateFrom, dateTo]` (занятия видимого диапазона).
   Диапазон уже считает `visibleRange` — оставить как есть.
2. Занятия: `enabled: staff и halls загружены` (сейчас — флаг `loaded`);
   маппер `lessonToBooking` — через `select`, чтобы кэш хранил сырой ответ API,
   а сетка получала готовые `Booking`.
3. Защита от гонки при быстром листании (`stale`-флаг в `useSchedule.ts:54`)
   умирает — Query сам отменяет устаревшие запросы по смене ключа.
4. Вернуть наружу тот же контракт `{ trainers, halls, bookings, … }`; локальный
   `setBookings` на этом шаге оставить как тонкую обёртку (мутации заберут его
   в задаче 4).
**⚠️ Подводные камни:** `useEffect` в `Journal.tsx:58-59` (автовключение всех
тренеров/залов в фильтры) завязан на приход данных — проверить срабатывание при
мгновенной отдаче из кэша (тот же камень, что в V3-1 задача 2).
**Готово, когда:** Журнал работает как раньше; повторный заход — мгновенный;
изменение зала в Каталоге после его миграции подтянется само (проверяется
позже, ключи уже общие).

### 3. Листание без пустоты: `keepPreviousData` + скелетон · 🟡 · 1:30
**Простыми словами:** при листании дней старая сетка остаётся на экране, пока
едет новая — анимация `withAnimation` играет поверх данных, а не поверх пустоты.
При самом первом заходе (кэш пуст) — скелетон сетки вместо голых колонок.
**Зачем:** п. 8 аудита: «переключаешься — оно начинает данные грузить, выглядит
будто лагает, я хочу чтобы всё было плавно и обалденно».
**Где:** `useSchedule.ts` (`placeholderData: keepPreviousData` для ключа
занятий), `Journal.tsx` / `Grid.tsx` / `Journal.css` (скелетон).
**Что конкретно сделать:**
1. `placeholderData: keepPreviousData` на квери занятий — листание дней и недель
   больше никогда не показывает пустую сетку.
2. Скелетон первой загрузки: колонки-плейсхолдеры (шапка с кружком-аватаром +
   2–3 призрачные карточки на разной высоте), мягкий shimmer, цвета
   `rgba(26,26,26,0.04–0.07)`, радиусы 12–16px (CLAUDE.md §6). Показывать
   только при `isLoading && !data`.
3. Проверить связку с `withAnimation` (250 мс slide в `Journal.tsx:71-78`):
   анимация запускается по клику, данные подъезжают из кэша/сети — если данных
   ещё нет, старые остаются (п. 1), «прыжка» быть не должно.
**Готово, когда:** листание дней с заторможенной сетью (devtools → Slow 3G) —
без единого кадра пустой сетки; первый заход — скелетон, повторные — мгновенно.

## Фаза 2 — Запись

### 4. Мутации → `useMutation` с оптимизмом и откатом · 🔴 · 2:00
**Простыми словами:** все восемь путей записи (создать, перенести, растянуть,
изменить из попапа, отменить занятие; записать, снять, отметить клиента)
перевести на мутации: мгновенный оптимистичный апдейт кэша → запрос → при
ошибке откат + тост; после успеха инвалидация ключа.
**Зачем:** сейчас каждая правка руками патчит локальный стейт (`setBookings`)
в пяти разных местах — источник правды расходится с сервером до следующего
листания; после миграции чтения на Query это вообще перестанет работать.
**Где:** `Journal.tsx` (`commitBookingChange:191`, `deleteBooking:276`,
`createLessonFromModal:287`, `confirmAddClients:320`), `BookingPopup.tsx`
(`markAttended:66`, `removeClient:78`), сервисные вызовы — `scheduleApi`.
**Что конкретно сделать:**
1. Хук `useJournalMutations` рядом с `useSchedule`: мутации поверх
   `scheduleApi`, оптимизм через `onMutate` → `setQueryData(['journal-lessons',
   from, to], …)` + снапшот для отката в `onError`, `invalidateQueries` в
   `onSettled`.
2. `setBookings` из компонентов исчезает полностью — все правки идут через
   мутации (это же зачистит ручную синхронизацию `popupBooking` ↔ `bookings`
   в `removeClient`).
3. Контракт мутаций сохранить async-функциями, кидающими `ApiError` — обработка
   ошибок остаётся в вызывающем коде (тосты — задача 6 и эпик V4-3).
4. **Каждая мутация возвращает данные для undo** (prev-снапшот занятия) — эпик
   V4-3 строит на этом историю, закладываем сигнатуру сразу: `mutate(payload,
   { prev })` или возврат `{ prev, next }` из хука.
**⚠️ Подводные камни:**
- `confirmAddClients` пишет пачкой через `Promise.allSettled` с частичным
  успехом («Записано 2 из 3») — оптимизм применять по одному, а не на всю пачку.
- Список записанных в попапе (`booked` в `BookingPopup.tsx:52`) — отдельный
  серверный стейт; либо тоже на квери `['lesson', id]` с инвалидацией от
  мутаций записей, либо оставить локальным, но обновлять из `onSuccess` (первое
  правильнее и дешевле в поддержке).
**Готово, когда:** все правки применяются мгновенно, при обрыве сети
откатываются с тостом; после любой мутации данные в сетке, попапе и правой
панели согласованы без F5.

**Простыми словами:** чужие правки подтягиваются сами (второй админ добавил
занятие — у первого оно появилось), а соседние дни уже загружены к моменту клика.
**Где:** `useSchedule.ts`, боковое меню в `DashboardLayout.tsx` (карта префетча
из V3-1 задачи 5).
**Что конкретно сделать:**
1. На квери занятий: `refetchInterval: 60_000` (минута — достаточно, websocket
   не заводим — YAGNI), `refetchOnWindowFocus` уже в дефолтах провайдера.
2. Префетч соседей: после успешной загрузки дня — `prefetchQuery` на день −1 и
   +1 (в недельном виде — соседние недели). Листание становится мгновенным
   почти всегда.
3. Пункт «Журнал» в карте hover-префетча меню: `['staff']`, `['halls']`,
   занятия сегодняшнего дня.
**Готово, когда:** две вкладки: правка во второй появляется в первой ≤ минуты
без действий; листание вперёд-назад по загруженным дням — без сети.

### 6. Ошибки загрузки видимы · 🟡 · 0:45
**Простыми словами:** упавшая загрузка перестаёт выглядеть как «пустой журнал»:
error-тост + на месте сетки состояние «Не удалось загрузить — Повторить».
**Зачем:** п. 9 аудита («ошибки пусть выскакивают все»); сейчас `catch →
console.error` — тишина.
**Где:** `useSchedule.ts` (ошибки квери наружу), `Journal.tsx`/`Grid.tsx`
(состояние ошибки), тост — общий из UI-кита.
**Что конкретно сделать:**
1. **Координация с V3-2:** используем общий `useToast()`; если V3-2 ещё не
   выполнен — временно локальный тост Журнала (он пока жив, его снос — эпик
   V4-3 вместе с undo-тостом).
2. Ошибка квери занятий/тренеров/залов → error-тост с `ApiError.message` +
   плашка по центру сетки с кнопкой «Повторить» (дёргает `refetch`).
3. Фоновая ошибка (данные в кэше есть, refetch упал) — только тост, сетку
   не ломать.
**Готово, когда:** выключенный бэкенд превращает заход в Журнал в понятный тост
и кнопку «Повторить», а не в тишину с пустыми колонками.

## Definition of Done эпика

- В папке Journal нет ни одного ручного `fetch`-стейта: чтение — `useQuery`,
  запись — `useMutation` с оптимизмом и откатом.
- Чужие правки видны ≤ минуты; свои — мгновенно и согласованно во всех виджетах.
- Листание никогда не показывает пустую сетку; первый заход — скелетон.
- Ошибки загрузки и мутаций видимы всегда; молчаливых `console.error` нет.
- Мутации отдают prev/next-снапшоты — фундамент для Undo/Redo (V4-3) готов.
- Сборка и линт чистые.
