# Эпик V4-3 — Undo/Redo и тост-отмена: правки, которые не страшно делать

Цель эпика простыми словами: раз черновика больше нет (V4-1) и каждая правка
сразу уходит на сервер (V4-2), пользователю нужна страховка. Две механики,
решённые владельцем 2026-07-14: после каждой правки — тост с кнопкой «Отменить»
и таймером 5 секунд; плюс полноценная история Undo/Redo — стрелки ← → на месте
удалённого статус-бейджа черновика и горячие клавиши Ctrl+Z / Ctrl+Y. История
живёт в памяти страницы: уход из Журнала просто очищает стек — всё уже
сохранено на сервере, терять нечего.

Источник: аудит 3 в `docs/ROADMAP_V4/ROADMAP.md`, пп. 3, 9 (обсуждено 2026-07-14).

**Важно понимать про старт:**
- Выполнять **после V4-2**: undo/redo — это обратные вызовы тех же мутаций;
  их сигнатура с prev/next-снапшотами заложена там (задача 4, пункт 4).
- **Координация с V3-2 (UI-кит):** undo-тост — новый вариант общего `Toast`.
  Если V3-2 к старту не выполнен — общий Toast создаётся здесь по спеке V3-2
  (задача 1) сразу с undo-вариантом, а в V3-2 отмечается «сделано в V4-3».
  Локальный тост Журнала (`Journal.tsx:149-167`, шестая реализация в проекте)
  в любом случае сносится здесь.
- Одна операция необратима на сервере: **отмена занятия** каскадом снимает
  записи клиентов и шлёт им уведомления (`lessons.py:249-288`). Для неё —
  отложенный коммит (запрос уходит после истечения тоста), решение из аудита.
- Для undo **создания** занятия нужен настоящий DELETE на бэке — сейчас есть
  только `cancel` (occupies id, оставляет мусор при цикле «создал-передумал»).

**Обозначения сложности:** 🟢 простая · 🟡 средняя · 🔴 сложная, делать внимательно.

## Фаза 0 — Фундамент

### 1. Undo-вариант общего Toast · 🟡 · 1:30 · ✅ сделано (2026-07-19)
**Простыми словами:** третий вариант тоста в UI-ките (после success/error):
сообщение + кнопка «Отменить» + тонкая полоска-таймер, утекающая за 5 секунд.
Клик по «Отменить» вызывает переданный колбэк и мягко закрывает тост.
**Зачем:** решение владельца: «как ты делаешь изменения — автоматом появляется
тост и предлагает отменить с таймером в 5 секунд».
**Где:** `front/src/components/ui/Toast.tsx` (общий компонент из V3-2 — см.
координацию в шапке), `DashboardLayout.tsx` (монтирование провайдера, если ещё
не смонтирован).
**Что конкретно сделать:**
1. API: `toast.undo(msg, { onUndo, duration = 5000 })`. Внутри — та же плашка
   `#1A1A1A`, кнопка «Отменить» акцентом `--peach`, прогресс-полоска по нижней
   кромке (CSS-анимация `width 100%→0` за duration, линейная).
2. Клик «Отменить» → `onUndo()` → exit-анимация. Истечение таймера → просто
   exit (коммит уже случился, для отложенных операций — см. задачу 5: там
   `onExpire`).
3. Наведение мыши на тост ставит таймер на паузу (стандарт UX для undo-тостов —
   пользователь целится в кнопку, тост не должен сбежать).
4. Новый undo-тост при живом старом: старый немедленно коммитится (его
   `onExpire`, если есть) и уходит — «Отменить» всегда относится к последней
   операции, очередь не копим.
5. Все требования базового Toast из V3-2 (плавный exit с текстом, тёмная тема)
   действуют и здесь.
**Готово, когда:** демо-цепочка «правка → тост → Отменить» и «правка → 5 с →
сам ушёл» работает плавно; hover держит тост; двойная правка не оставляет два
тоста.

### 2. Бэк: `DELETE /schedule/lessons/{id}` · 🟢 · 0:45 · ✅ сделано (2026-07-19)
**Простыми словами:** настоящее удаление занятия — только для отката только что
созданного. Не путать с «отменой» (cancel остаётся основным пользовательским
действием: занятие остаётся в базе со статусом cancelled).
**Зачем:** undo создания через `cancel` оставлял бы в базе мусорные отменённые
занятия при каждом «создал-передумал».
**Где:** `back/routers/schedule/lessons.py`, роут в `back/routers/schedule/
__init__.py` (по образцу соседей).
**Что конкретно сделать:**
1. `DELETE /schedule/lessons/{lesson_id}` → 204. Скоуп — `get_scoped_lesson`
   (чужая студия 404); роль: владелец и админ (тренер — 403, как во всём V4-1).
2. Защита: удалять можно только занятие **без активных записей** (иначе 409
   «На занятие записаны клиенты — сначала снимите их или отмените занятие») —
   DELETE существует для отката свежесозданного, а не для стирания истории.
3. Фронт: `scheduleApi.deleteLesson(id)` в `schedule.api.ts`.
**Готово, когда:** созданное и тут же удалённое занятие не оставляет следов в
базе; попытка удалить занятие с записями — 409 с понятным текстом.

## Фаза 1 — История

### 3. Undo-менеджер: стек пар {undo, redo} · 🔴 · 2:00 · ✅ сделано (2026-07-19)
**Простыми словами:** маленький хук `useUndoHistory` в Журнале: каждый успешный
коммит правки кладёт в стек пару функций — «как откатить» и «как повторить».
Ctrl+Z снимает верхушку и зовёт undo, Ctrl+Y двигает указатель обратно и зовёт
redo. Никакой библиотеки — это массив и указатель.
**Зачем:** решение владельца: «стрелки назад-вперёд… шаги можно держать во
временной памяти, если выйдешь — всё сохранится автоматом» (сохранится — потому
что каждый шаг уже закоммичен на сервер; стек хранит только обратные вызовы).
**Где:** новый `front/src/pages/dashboard/Journal/hooks/useUndoHistory.ts`,
интеграция — `useJournalMutations` из V4-2.
**Что конкретно сделать:**
1. Структура: `{ past: Entry[], future: Entry[] }`, `Entry = { label, undo:
   () => Promise<void>, redo: () => Promise<void> }`. Новая операция очищает
   `future` (классика: после отката и новой правки «вперёд» некуда).
2. Каждая мутация V4-2 после успеха регистрирует entry:
   - перенос/растяг/правка → undo = PATCH с prev-полями, redo = PATCH с next;
   - создание → undo = `deleteLesson(id)`, redo = `createLesson(payload)`
     (⚠️ redo создаст **новый id** — entry обновляет id внутри себя, чтобы
     последующий undo удалял уже новый);
   - записи клиентов — задача 6.
3. Undo/redo сами являются мутациями Query (оптимизм + инвалидация) — сетка
   реагирует мгновенно.
4. Выполнение undo показывает обычный success-тост («Отменено: перенос
   занятия»), **не** undo-тост — иначе рекурсия «отмени отмену» через тост;
   для этого есть Ctrl+Y.
5. Стек в памяти компонента: смена дня/недели стек **не** чистит (операции
   привязаны к id занятий, не к видимому дню; undo правки вчерашнего дня из
   сегодняшнего — валиден, инвалидация сама обновит нужный диапазон). Размонт
   страницы — стек умирает, это решение владельца.
6. Ограничение глубины — 50 записей (не память ради памяти).
**⚠️ Подводные камни:** undo может упасть (409: место уже занято клиентом,
занятие отменено вторым админом) — показать error-тост с причиной и **выкинуть
битую entry из стека**, не зацикливая пользователя на неисполнимом откате.
**Готово, когда:** цепочка «перенёс → растянул → создал → Ctrl+Z ×3 → Ctrl+Y ×3»
приводит сетку и сервер ровно в исходное/конечное состояние (проверить F5 после
каждой точки).

### 4. Стрелки ← → и горячие клавиши · 🟡 · 0:45 · ✅ сделано (2026-07-19)
**Простыми словами:** две кнопки-стрелки там, где раньше жил статус-бейдж
черновика (правый край сводки дня), плюс Ctrl+Z / Ctrl+Y / Ctrl+Shift+Z.
**Где:** `DaySummary.tsx` (место освобождено задачей 3 эпика V4-1),
`Journal.tsx` (обработчик клавиш).
**Что конкретно сделать:**
1. Кнопки в стиле `btn-icon`: ← undo, → redo; задизейблены при пустом стеке
   (`past`/`future` соответственно); `title` с меткой операции («Отменить:
   перенос занятия») — метки уже в entry (`label`).
2. `keydown` на `document` при смонтированном Журнале: Ctrl/Cmd+Z → undo,
   Ctrl+Y и Ctrl/Cmd+Shift+Z → redo. Игнорировать, когда фокус в
   input/textarea (пользователь правит текст — Ctrl+Z принадлежит полю).
3. Для тренера (read-only, V4-1) стрелки не показываются вовсе.
**Готово, когда:** стрелки и клавиши работают синхронно с одним стеком;
дизейбл честный; ввод текста в форме не перехватывается.

## Фаза 2 — Особые случаи

### 5. Отмена занятия: отложенный коммит + ConfirmModal · 🟡 · 1:15 · ✅ сделано (2026-07-19)
**Простыми словами:** отмена занятия — единственная необратимая операция
(каскад по клиентам + уведомления), поэтому она **не** уходит на сервер сразу:
карточка мгновенно гаснет, запрос стреляет после истечения undo-тоста. «Отменить»
в тосте возвращает карточку — на сервер не ушло ничего.
**Зачем:** решение аудита: «обратной операции нет → отложенный коммит»;
уведомление «занятие отменено» клиентам нельзя отозвать.
**Где:** `Journal.tsx` (`deleteBooking`), undo-тост (`onExpire` из задачи 1),
`ConfirmModal` из UI-кита V3-2 (координация как в шапке).
**Что конкретно сделать:**
1. Клик «Отменить занятие»: если записанных > 0 — сначала `ConfirmModal`
   («Отменить занятие? N клиентов получат уведомление об отмене», данные о
   записанных уже есть в попапе); без записанных — сразу к шагу 2.
2. Оптимистично пометить занятие cancelled в кэше → undo-тост «Занятие
   отменено» → `onExpire` → реальный `scheduleApi.cancelLesson(id)`;
   `onUndo` → вернуть кэш из снапшота, на сервер не ходить.
3. Страховки отложенного коммита: уход со страницы / закрытие вкладки /
   листание дня при живом таймере — коммитим немедленно (`beforeunload` +
   cleanup-эффект), отмена не должна «потеряться».
4. В общий undo-стек (задача 3) отложенная отмена **не** кладётся до коммита;
   после коммита не кладётся тоже (необратима) — Ctrl+Z её не трогает. Это
   осознанная асимметрия, отразить комментарием в коде.
5. При ошибке коммита (сервер упал через 5 с) — откатить кэш + error-тост.
**Готово, когда:** отмена с клиентами спрашивает подтверждение; «Отменить» в
тосте полностью бескровен (сервер не узнал); истечение — честный cancel с
каскадом; F5 в любой точке не оставляет полу-состояний.

### 6. Undo записи/снятия клиента · 🟡 · 0:45 · ✅ сделано (2026-07-19)
**Простыми словами:** запись клиента на занятие и снятие с занятия — тоже
операции с «Отменить» в тосте и местом в истории.
**Где:** `useJournalMutations` (V4-2), `confirmAddClients` в `Journal.tsx:320`,
`removeClient` в `BookingPopup.tsx:78`.
**Что конкретно сделать:**
1. Запись клиента: undo = `cancelReservation(reservation_id)` (id есть в ответе
   `createReservation`); redo = повторный `createReservation` (с обновлением
   id в entry, как у создания занятия).
2. Пачка из `AddClientModal` («Записано: 3 клиента») — одна entry на пачку:
   undo снимает всех успешно записанных, redo записывает обратно.
3. Снятие клиента: undo = `createReservation(client_id, lesson_id)` заново.
   ⚠️ Если между снятием и undo место заняли — сервер ответит 400 «Все места
   заняты»: показать error-тост, entry выкинуть (правило из задачи 3).
4. Отметку посещения (attend) в историю **не** включать — идемпотентная
   галочка факта, «отменять посещение» — не журнальная операция (нет и
   эндпоинта). Тост у неё обычный success.
**Готово, когда:** «записал → Отменить» и «снял → Отменить» возвращают точное
состояние мест и списка «Записаны» в попапе; конфликт мест даёт понятную ошибку.

- Локальный тост Журнала удалён — страница на общем Toast из UI-кита.
- Сборка и линт чистые.
