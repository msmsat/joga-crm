# Velora CRM — Roadmap до релиза MVP

## Контекст

Аудит по `docs/TZ.md`, `docs/STATUS.md` и коду (`back/main.py`, `back/routers/**`, `back/dependencies.py`):

- **Фронтенд** визуально готов почти на 100% по 14 страницам, но работает на захардкоженных массивах из 5–10 элементов — на реальных объёмах данных без пагинации/виртуализации он деградирует (см. сквозные задачи).
- **Бэкенд** подключён для 9 разделов; реально работают только auth, studio (частично), clients, staff. Пять (schedule, finances, settings, ai, analytics) подключены, но пусты. Разделов Онлайн-запись, Лояльность, Уведомления, Тариф/оплата на сервере нет.
- **Дашборд (`Overview`)** не делает ни одного запроса — всё захардкожено; своего серверного раздела нет, обслуживается агрегатами аналитики (Эпик 3).
- **Критический долг:** проверка роли есть только в auth-роутерах; остальные не проверяют отправителя — тренер/админ в обход UI могут получить чужие данные.
- БД уже спроектирована под весь функционал впрок.
- Точечные кнопки-имитации в рабочих Клиентах/Сотрудниках («Позвонить», «Сообщение», «Записать», «Бонус», «Написать», «Добавить событие») расписаны внутри профильных эпиков (Эпик 2 и 4), а не отдельно.

**Мультистудийность — важное уточнение:** `studio_id`-скоуп (изоляция данных студий в общей БД) обязателен и делается в Эпике 1. А переключение пользователя между несколькими студиями (своя роль в каждой, активная студия, «Сменить CRM») — дорогой multi-tenancy, вынесен за MVP (см. «Вне рамок»).

Порядок эпиков: фундамент → ядро (Журнал) → данные/аналитика (Финансы/Отчёты/Дашборд) → каналы роста → биллинг → ИИ/полировка.

## Эпик 1: Технический фундамент (безопасность, скоуп, контракт данных)

**Цель:** Прочное и безопасное основание для всей дальнейшей работы: серверная проверка ролей, изоляция по студии, и контракт списочных API (пагинация/фильтры) — чтобы потом не переписывать ни бэкенд, ни фронтенд.

**Точка старта:** Роль проверяется только на входе и при удалении последнего владельца. Остальные роутеры не проверяют отправителя. В Каталоге не хватает CRUD филиала/зала и всего раздела услуг; Профиль показывает заглушку вместо реального пользователя. Списочные эндпоинты (там где есть) не пагинированы.

**Ключевые работы:**
- Backend: переиспользуемая зависимость проверки роли и области видимости (владелец/администратор/тренер; тренер — только свои данные) + фильтрация всех запросов по `studio_id` (изоляция студий). Применить на clients, staff, studio, schedule, finances, settings, analytics.
- Backend: **контракт списочных эндпоинтов** — пагинация (`items` + `total` + `offset`/cursor), серверные фильтры и сортировка с первого дня. Все будущие списки (клиенты, занятия, операции, сотрудники) следуют этому контракту.
- Backend + БД: доделать Каталог — PATCH/DELETE филиала, POST/PATCH/DELETE зала; роутер + модель + CRUD услуг.
- Backend + Frontend: реальные данные пользователя в Профиле вместо хардкода.
- Frontend: связать формы правки/удаления филиала, зала, услуг с API; обработка 403 в UI (сейчас не ожидается).
- Проверка: чек-лист «тренер/админ не может...» по каждому разделу.

## Эпик 2: Журнал — ядро операционного цикла

**Цель:** Перевести Журнал из макета в реально работающий центр записи на занятия.

**Точка старта:** `schedule` подключён, но пуст. Вся интерактивность (виды, фильтры, drag&drop, растягивание, модалки) работает на моках; после перезагрузки всё сбрасывается. БД под занятия и записи готова.

**Ключевые работы:**
- БД/Backend: эндпоинты занятий (создание/перенос/растягивание/удаление) и записи клиентов (добавить/отменить) с проверкой лимита мест и запретом двойной записи.
- **Прямое сохранение (без серверного черновика):** перенёс/изменил карточку → запрос → сохранено. Режим черновика (ТЗ 4.2) остаётся **клиентским** — фронтенд копит изменения в локальном стейте (как сейчас), «Опубликовать» отправляет батч обычных сохранений, конфликты — last-write-wins. Никакого серверного хранилища черновика и разрешения конфликтов в MVP.
- Frontend: заменить моки на реальные запросы (тренеры/залы переиспользовать из Сотрудников/Каталога); подключить создание/перенос/удаление и мастер «Добавить клиента».
- Frontend: подгружать только видимый диапазон (день/неделя), не всё расписание разом — окно данных вместо полного списка.
- Frontend + Backend: сводка дня из реальных агрегатов.
- Backend: ролевой скоуп — тренер видит/меняет только свои занятия.
- Backend: кнопки «Записать» (Клиенты) и «Добавить событие» (Сотрудники) → те же эндпоинты записи/создания занятия.

## Эпик 3: Финансы, Отчёты и Дашборд — реальные данные и аналитика

**Цель:** Три разных среза одних и тех же реальных данных о клиентах, занятиях и деньгах — поверх общего агрегирующего слоя.

**Точка старта:** `finances` и `analytics` пусты; Дашборд не делает запросов. БД под счета/операции/зарплаты/контрагентов/документы/цели готова. Экспорт CSV/PDF механически работает, данные заготовленные.

**Ключевые работы:**
- Backend: CRUD Финансов — Счета, Операции (пагинированный список по контракту Эпика 1), Зарплаты, Контрагенты, Документы, Цели (авто/ручной режим).
- Backend: события «оплата»/«визит» из Журнала и Клиентов формируют операции — единый источник фактов.
- Backend: агрегирующий слой (выручка/расходы/прибыль по периодам, retention, по тренерам/услугам, средний чек, загрузка) — обслуживает и Отчёты, и Дашборд.
- Backend + Frontend: **Дашборд** — метрики с трендами, график, лента событий, три сводки → к агрегатам. Виджет «Задачи на сегодня» — отдельная модель + CRUD.
- Frontend: подключить вкладки Финансов и Отчётов, сохранив интерактивность графиков.
- Backend: реальные данные в существующий экспорт CSV/PDF.

## Эпик 4: Каналы роста — Онлайн-запись, Лояльность, Уведомления

**Цель:** Контур привлечения и удержания поверх уже реальных данных (Эпики 2–3).

**Точка старта:** Серверных разделов нет — только вёрстка на локальном стейте. БД под лояльность есть. Плюс долг: «Бонус», «Позвонить»/«Сообщение» (Клиенты), «Написать»/«Позвонить» (Сотрудники) — имитация.

**Ключевые работы:**
- Backend/БД: раздел «booking» — правила записи, подключение каналов (Telegram-токен, WhatsApp/Instagram/веб), публичный эндпоинт клиентского потока «услуга → время → подтверждение».
- Backend/БД: «лояльность» — 5 программ, начисления/списания, привязанные к оплатам/визитам (Эпик 3).
- Backend/БД: «уведомления» — матрица «событие × канал» сохраняется; минимум один реальный канал (Telegram/Email) для сквозной проверки; остальные — сохранение конфигурации без отправки.
- Frontend: связать карточки каналов, панель лояльности, матрицу событий с API.
- Backend: события Журнала/Клиентов/Финансов триггерят уведомления и начисления лояльности.
- Backend/Frontend: «Бонус» → начисление баллов; «Позвонить»/«Сообщение»/«Написать» → каналы связи (звонок — `tel:`/WhatsApp-ссылка на фронте, сообщение — через подключённый канал).

## Эпик 5: Биллинг и монетизация (отдельно, до ИИ)

**Цель:** Принимать реальные деньги. Вынесен в отдельный эпик перед ИИ, потому что это самая рискованная часть (реальные платежи, вебхуки провайдера, статусы, возвраты) и требует запаса на тестирование — её нельзя делать «в остаток сил».

**Точка старта:** Тариф/оплата считает по заготовленным ценам; смена тарифа и привязка карты никуда не уходят; истории платежей нет. Провайдер — **Fondy** (в Настройки → Интеграции пока заявлена ЮKassa — карточку интеграции переименовать в Эпике 6).

**Ключевые работы:**
- Backend/БД: тарифы (Старт/Pro/Business), периоды и скидки, текущая подписка студии, история платежей.
- Backend: интеграция с платёжным провайдером (Fondy) — редирект на Fondy Checkout, привязка карты токеном (rectoken), списание, **вебхуки статусов оплаты с проверкой подписи**, возвраты (reverse); идемпотентность по order_id и корректная обработка повторных колбэков.
- Backend: применение лимитов тарифа (напр. Старт — до 3 сотрудников / 100 клиентов).
- Frontend: подключить «Тариф и оплата» (калькулятор периода, смена тарифа, история, способ оплаты) к API.
- Тестирование: тестовый режим провайдера, проверка сценариев успех/отказ/возврат до боевого ключа.

## Эпик 6: Velora AI и финальная полировка (может уехать за MVP)

**Цель:** Заменить имитацию ИИ настоящим ассистентом и закрыть остаточные имитации в Настройках. **Наименее критичный эпик:** мок ассистента работает для демо, поэтому при нехватке времени этот эпик частично или полностью переносится на пост-MVP без блокировки релиза.

**Точка старта:** Velora AI отвечает заготовленной фразой. Настройки свёрстаны, но логотип/бэкапы/интеграции/безопасность — имитация.

**Ключевые работы:**
- Backend: интеграция с реальной LLM для чат-ассистента (системный промпт из настроек, контекст студии) и двух агентов (Telegram/Instagram).
- Backend + Frontend: единая серверная история AI-диалогов, общая для раздела AI, глобальной панели и AI-строки (ТЗ 12.4).
- Backend: доработка Настроек — загрузка логотипа, бэкапы/экспорт данных, 2FA, реальные сессии и API-токены.
- Backend: обработчики Настройки → Интеграции (WhatsApp/Telegram/Instagram/Google Calendar/1С/Fondy) — сохранение конфигурации и статус.
- Frontend: остаточные вкладки к API; проверка i18n-покрытия (`i18next` уже установлен) и сохранения темы/языка; финальный аудит фронт↔бэк по `docs/TZ.md`.

## Сквозные задачи (во всех эпиках)

- **Производительность списков.** Все списочные эндпоинты — пагинированы (контракт Эпика 1). Фронт: дебаунс поиска, виртуализация/lazy-load тяжёлых сеток (Журнал, Клиенты, Операции). Не оставлять рендер тысяч элементов разом.
- **Миграции БД (Alembic)** при каждом изменении схемы (`alembic revision --autogenerate` → `alembic upgrade head`).
- **Лента событий студии** — общий поток (запись/оплата/заморозка/новый клиент), наполняется каждым эпиком, а не в конце.
- **Состояния загрузки/ошибок в UI** (включая 403) при подключении каждого раздела.
- **Бейджи-счётчики в меню** — к реальным данным вместе с разделом.
- **Синхронизация типов Frontend ↔ Backend** по `CLAUDE.md` §8: TS-интерфейсы строго под Pydantic-схемы, производные UI-данные — на лету.

## Вне рамок этого roadmap

- **Клиентское мини-приложение** (раздел 17 ТЗ) — постMVP, отдельный roadmap.
- **Мультистудийность (переключение между студиями)** — пользователь в нескольких студиях со своей ролью в каждой, активная студия, «Сменить CRM», создание доп. CRM (ТЗ 2.4, 14.10). Полноценный multi-tenancy режем из MVP как непропорционально дорогой: `studio_id`-изоляция данных (Эпик 1) остаётся, но UI ограничивается одной студией на аккаунт. Возвращается в v2.
