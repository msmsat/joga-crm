# Эпик 6 — Velora AI (self-hosted LLM с инструментами) + финальная полировка

> Настоящий ИИ-ассистент на собственной модели: видит расписание и статистику с учётом роли, переводит по страницам, меняет данные через подтверждение. Плюс добивка Настроек, тёмная тема, i18n и финальный аудит.
> Источник: [`docs/ROADMAP.md`](ROADMAP.md), Эпик 6. Модель — **self-hosted** (сервер с GPU 24 ГБ), без зависимости от внешних API.

## Выбор модели и сервинга (решение зафиксировано)

Критерии: русский язык, нативный tool calling, помещается в 24 ГБ VRAM, коммерческая лицензия, живое комьюнити для дообучения.

- **Основная: Qwen3-14B** (Apache 2.0) в 4-бит квантовании — лучший баланс русского языка и function calling для 24 ГБ; линейка Qwen3/3.5 — топ мультиязычных open-моделей 2026 года, Qwen3-14B называют оптимумом «производительность/эффективность» для русского.
- **Альтернатива «от Google»: Gemma 3 27B** в Q4 — нативный function calling (модель обучена выдавать структурные tool-вызовы); с прицелом на Gemma 4, когда стабилизируется в квантованном виде.
- **Русскоязычные тюны для этапа дообучения:** T-pro/T-lite (Т-Банк, Qwen-based), Saiga, Vikhr — если «натуральность» русского окажется важнее.
- **Сервинг:** старт — **Ollama** (OpenAI-совместимый API из коробки); прод при нагрузке — **vLLM** (тот же контракт, бэкенд не меняется).
- **Дообучение — НЕ начинать с него.** Системный промпт + инструменты закрывают ~90 % задач. LoRA (Unsloth) на накопленных реальных диалогах CRM — отдельный шаг после MVP.

## Что уже есть в коде (точка старта)

- **Модели БД готовы** (`back/models/ai.py`): `StudioAISettings` (model/language/system_prompt + полные конфиги TG/IG-агентов со статистикой handled_count/avg_rating), `AIChatSession`, `AIChatMessage`. Миграции не нужны.
- **Бэкенд — чистое поле:** `back/routers/ai/chat.py` и `back/schemas/ai/*` — заглушки, ноль эндпоинтов.
- **Контракт фронта готов:** `front/src/api/ai/ai.api.ts` — 7 методов (`/ai/sessions` CRUD, messages, `/ai/settings`). Гэп: тип `AISettings` не содержит `*_max_length` и полей статистики из модели.
- **Четыре независимых мок-чата на фронте:** страница `AI/` (`useAIChat.ts` + `MOCK_AI_RESPONSES`), глобальный дровер `components/AIDrawer/` (`useDrawerChat.ts`), AI-строка в шапке (`DashboardLayout.tsx:73-82` — «Подключите сюда реальный вызов»), чат Журнала (`Journal/components/AiChat.tsx:33` — «демо-режим»). Общий только `AIDrawerContext` (open/close).

- **Настройки:** роутеры `back/routers/settings/*` — заглушки; вкладки фронта — локальные моки; `settingsApi` уже типизирован (general, working-hours, security, integrations). Логотип в `GeneralTab.tsx` — base64 в стейт; паттерн загрузки готов в `back/routers/studio/media.py` (**`POST /upload-logo` сейчас без auth — дыра**). Модели `UserSession`, `ApiToken`, `StudioIntegration`, `StudioBackupSettings`, `StudioWorkingHours` — готовы.
- **Тёмной темы не существует** (нет data-theme/персиста; `AppearanceTab` — косметика). CSS-переменные есть в `App.css :root`.
- **i18n:** i18next настроен, но дефолт `en` (баг для RU-продукта), язык не персистится; переведены только Staff/Profile/Catalog.

## Жёсткие правила эпика (MVP)

1. **Никакой привязки к вендору:** бэкенд говорит по OpenAI-совместимому протоколу; смена модели/сервера = `LLM_BASE_URL`/`LLM_MODEL` в .env (Ollama ↔ vLLM ↔ что угодно).
2. **Ассистент = агентный цикл с инструментами**, и каждый инструмент выполняется **с ролью и студией текущего пользователя** (get_studio_context/require_role из Эпика 1): тренер через ИИ физически не достанет чужие данные.
3. **Изменяющие действия — только через подтверждение:** mutating-инструменты не исполняются внутри цикла; ассистент возвращает «предложение действия», человек подтверждает кнопкой — галлюцинация модели не испортит расписание.
4. **Одна серверная история чатов для всех поверхностей** (раздел AI, дровер, AI-строка, Журнал — ТЗ 12.4): четыре мок-движка заменяются одним хуком.
5. **Стриминг SSE** — токены + статусы «вызываю инструмент…»: локальная модель отвечает секундами, без стрима ассистент ощущается зависшим.

**Обозначения сложности:** 🟢 простая · 🟡 средняя · 🔴 сложная, делать внимательно.

## Фаза 0 — Подготовка (бизнес-задача, не код)

### 0. Выбрать и арендовать GPU-сервер (24 ГБ VRAM) · 🟡 · 1:00
**Простыми словами:** для self-hosted модели нужен сервер с видеокартой на 24 ГБ (уровень RTX 3090/4090/A5000) — его надо выбрать, арендовать и получить SSH-доступ.
**Зачем:** задача 1 разворачивает Ollama — разворачивать негде, пока сервера нет. Это ежемесячные деньги — решение бизнес-уровня.
**Что конкретно сделать:**
1. Сравнить 2–3 GPU-хостинга по цене за месяц аренды карты 24 ГБ (ориентир: заметная статья расходов, закладывать в юнит-экономику тарифов).
2. Учесть: география (пинг до основного VPS), почасовая vs помесячная оплата (на этапе разработки почасовая дешевле).
3. Арендовать, получить SSH-доступ, проверить `nvidia-smi`.
**⚠️ Подводные камни:** для разработки и демо хватит почасовой аренды с включением по необходимости; постоянная аренда нужна только с первыми реальными пользователями. Ассистент должен присутствовать в юнит-экономике: стоимость сервера / число студий.
**Готово, когда:** SSH работает, `nvidia-smi` показывает ≥24 ГБ; стоимость в месяц записана в этот файл.

## Фаза 1 — Модель и сервинг

### 1. Поднять self-hosted LLM (Qwen3-14B + Ollama) · 🔴 · 2:00
**Простыми словами:** развернуть свою модель на сервере из задачи 0 и убедиться, что она держит русский и вызовы инструментов.
**Зачем:** независимость от внешних API — базовое требование; всё остальное строится на этом.
**Где:** сервер (Ollama); `back/.env.example` (`LLM_BASE_URL`, `LLM_MODEL`); `docs/TZ/STATUS.md`.
**Что конкретно сделать:**
1. Установить Ollama; `ollama pull qwen3:14b` (Q4).
2. Дымовой тест: русский диалог + tool calling через OpenAI-совместимый `/v1/chat/completions`; замерить скорость токенов.
3. Запасной вариант — `gemma3:27b-q4`; зафиксировать выбор и замеры в `docs/TZ/STATUS.md`.
**⚠️ Подводные камни:** порт Ollama наружу не открывать всем — только бэкенду (firewall/private network); vLLM — прод-опция позже без смены кода.
**Готово, когда:** модель отвечает по-русски; корректно вызывает тестовый инструмент; endpoint доступен бэкенду по LLM_BASE_URL.

### 2. LLM-клиент бэкенда · 🟡 · 1:00
**Простыми словами:** один модуль общения с моделью: чат, инструменты, стриминг — без привязки к конкретному серверу.
**Зачем:** смена модели или переезд Ollama→vLLM = одна переменная .env; код не трогаем.
**Где:** новый `back/services/llm.py`; `back/.env.example`.
**Что конкретно сделать:** `chat(messages, tools, stream)` по OpenAI-совместимому протоколу через aiohttp (уже в requirements); base_url и model из .env (паттерн `back/security.py`); таймауты и понятная ошибка, если LLM-сервер лежит.
**Готово, когда:** из консоли бэкенда уходит запрос и приходит стрим-ответ; при выключенном Ollama — внятная ошибка, а не зависание.

### 4. Реестр инструментов с ролевым скоупом · 🔴 · 2:30
**Простыми словами:** набор инструментов ассистента: расписание, клиенты, статистика, настройки, навигация — каждый работает от имени и роли пользователя.
**Зачем:** «взаимодействует с графиком, знает роль, помогает со статистикой» — сердце фичи; безопасность: тренер через ИИ не достанет чужое.
**Где:** новый `back/services/ai_tools.py`.
**Что конкретно сделать:**
1. Описать инструменты в формате OpenAI tools: `get_schedule(date_from, date_to)`, `create_lesson(...)`, `get_clients(search)`, `get_client(id)`, `book_client(...)`, `get_stats(period)` → агрегаты Эпика 3, `get_settings`/`update_setting`, `navigate(page)` — enum страниц CRM.
2. Каждый инструмент вызывает ту же логику, что обычные эндпоинты, с get_studio_context и require_role текущего пользователя.
3. У каждого флаг `mutating: true/false`.
**⚠️ Подводные камни:** инструменты НЕ должны обходить ролевые проверки — они проксируют существующую логику, а не пишут свои запросы к БД.
**Готово, когда:** вызов get_schedule тренером возвращает только его занятия; get_stats отдаёт цифры /analytics/summary; mutating-флаги проставлены.

### 5. Агентный цикл: LLM ↔ инструменты · 🔴 · 2:00
**Простыми словами:** обработка сообщения: модель думает, зовёт инструменты, получает результаты, отвечает по-русски.
**Зачем:** превращает чат-болталку в ассистента, который реально смотрит в данные студии.
**Где:** `back/routers/ai/chat.py`.
**Контракт:**
```
POST /ai/sessions/{id}/messages   { content, current_page? }
Ответ: сообщение ассистента (стрим — задача 7)
```
**Что конкретно сделать:**
1. Системный промпт = роль + имя студии + сегодняшняя дата + текущая страница (передаёт фронт) + system_prompt из StudioAISettings.
2. Цикл: ответ модели → если tool_calls — исполнить safe-инструменты → результаты обратно в модель; лимит 6 итераций.
3. Mutating-инструменты НЕ исполнять (задача 6); сохранить user+assistant сообщения в AIChatMessage; обновить preview сессии.
**Готово, когда:** вопрос «сколько занятий у меня завтра» отвечает реальными данными из БД с учётом роли; история сохраняется.

### 6. Подтверждение изменяющих действий · 🔴 · 1:30
**Простыми словами:** ИИ не меняет данные молча: предлагает действие, человек подтверждает кнопкой — только тогда исполняется.
**Зачем:** «менять что-то» без этого опасно — галлюцинация модели не должна портить расписание студии.
**Где:** `back/routers/ai/chat.py`, `back/services/ai_tools.py`.
**Контракт:**
```
В ответе цикла: action_proposal { tool, args, description }
POST /ai/actions/execute  { tool, args }  → исполнение с теми же ролевыми проверками
Ошибки: 400 tool не из реестра / args не валидны по схеме
```
**Готово, когда:** «Создай занятие завтра в 10» возвращает предложение, а не создаёт; execute создаёт; тренер не может выполнить чужое действие.

### 7. Стриминг ответов (SSE) · 🟡 · 1:30
**Простыми словами:** ответ прилетает токен за токеном плюс статусы «смотрю расписание...».
**Зачем:** локальная модель отвечает секундами — без стрима ассистент ощущается зависшим.
**Где:** `back/routers/ai/chat.py`.
**Что конкретно сделать:** StreamingResponse (text/event-stream) поверх задачи 5: события `token`, `tool_status` (имя инструмента), `action_proposal`, `done`; полное сообщение в БД по завершении.
**Готово, когда:** в curl виден постепенный поток событий; между вызовами инструментов приходят статусы.
