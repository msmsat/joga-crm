# Эпик 7 — Запуск и продажа (деплой, юридика, мониторинг)

> Всё, что превращает работающий на localhost код в продукт, за который можно брать деньги: боевой сервер, домен, HTTPS, юридические страницы (без них Fondy не верифицирует мерчанта), бэкапы, мониторинг и финальный смоук на проде.
> Источник: аудит ROADMAP на «продаваемость». **Ни одна задача этого эпика раньше нигде не значилась** — код мог быть готов на 100%, а продавать было бы нельзя.

- Тестировать вебхуки Fondy удобнее на реальном домене, чем через ngrok.
- Ранний деплой = ранняя обкатка: баги окружения всплывут, пока их дёшево чинить.

**Обозначения сложности:** 🟢 простая · 🟡 средняя · 🔴 сложная, делать внимательно.

## Фаза 1 — Инфраструктура

### 1. Купить домен и настроить DNS · 🟢 · 0:30
**Простыми словами:** выбрать и купить домен продукта (например velora-crm.com), направить A-записи на будущий VPS.
**Зачем:** домен нужен всем дальше: SSL, OAuth, Fondy (redirect и вебхук), почте (SPF/DKIM).
**Что конкретно сделать:** купить домен у регистратора; создать A-запись на IP VPS (задача 2); при желании — поддомен `api.` для бэкенда.
**⚠️ Подводные камни:** DNS распространяется до суток — покупать домен первым, не последним.
**Готово, когда:** `ping домен` отвечает IP VPS-а.

### 2. Арендовать VPS и базовая настройка · 🟡 · 1:30
**Простыми словами:** арендовать обычный (без GPU) сервер под бэкенд+фронт+БД и подготовить его.
**Зачем:** это дом продукта; GPU-сервер из Эпика 6 — отдельная машина только для LLM.
**Что конкретно сделать:**
1. Арендовать VPS (ориентир: 2 vCPU / 4 ГБ RAM / 60 ГБ SSD — хватит на старт).
2. Создать непривилегированного пользователя, вход по SSH-ключу, отключить парольный вход.
3. Firewall: открыть только 22/80/443.
4. Установить: python 3.12+, node LTS, nginx, postgresql, certbot, git.
**⚠️ Подводные камни:** не работать под root; порт PostgreSQL (5432) наружу НЕ открывать.
**Готово, когда:** SSH по ключу работает; `ufw status` показывает только 22/80/443.

### 3. Прод-PostgreSQL и миграции · 🟡 · 1:00
**Простыми словами:** создать боевую базу, отдельного пользователя БД с паролем и накатить все миграции.
**Где:** VPS; `back/alembic/`.
**Что конкретно сделать:**
1. `CREATE DATABASE velora; CREATE USER velora_app WITH PASSWORD '...'` + права только на эту БД.
2. Прописать DATABASE_URL в прод-.env (задача 7).
3. `alembic upgrade head` — вся схема из миграций, никаких ручных CREATE TABLE.
**⚠️ Подводные камни:** если `alembic upgrade head` на чистой базе падает — цепочка миграций битая, чинить сейчас, а не при первом клиенте.
**Готово, когда:** чистая база поднимается одной командой `alembic upgrade head`; подключение по DATABASE_URL работает.

### 4. Деплой бэкенда (uvicorn + systemd + nginx) · 🔴 · 2:00
**Простыми словами:** FastAPI работает на сервере постоянно, сам поднимается после перезагрузки, наружу смотрит через nginx.
**Что конкретно сделать:**
1. Клонировать репозиторий, venv, `pip install -r requirements.txt`.
2. systemd-юнит: `uvicorn main:app --host 127.0.0.1 --port 8000 --workers 2`, `Restart=always`.
3. nginx: `api.домен` (или `/api`) → proxy_pass на 127.0.0.1:8000; `client_max_body_size` под загрузку логотипов.
**⚠️ Подводные камни:** uvicorn слушает ТОЛЬКО 127.0.0.1 — наружу его выставляет nginx; `--reload` на проде не включать.
**Готово, когда:** `systemctl status velora-api` — active; `curl https://api.домен/docs` отвечает; после `reboot` сервис поднялся сам.

### 5. Сборка и деплой фронтенда · 🟡 · 1:00
**Простыми словами:** собрать React-приложение и отдавать статику через nginx.
**Что конкретно сделать:**
1. `cd front && npm ci && npm run build` (в `front/.env.production` — боевой `VITE_API_URL`).
2. nginx: корень домена → `dist/`; `try_files $uri /index.html` (иначе F5 на внутренних страницах даст 404 — React Router).
**⚠️ Подводные камни:** забытый `try_files` — классическая причина «404 после обновления страницы» в SPA.
**Готово, когда:** сайт открывается по домену; F5 на /dashboard/clients не даёт 404.

### 6. SSL-сертификат и HTTPS · 🟢 · 0:30
**Простыми словами:** бесплатный сертификат Let's Encrypt, весь трафик — только HTTPS.
**Что конкретно сделать:** `certbot --nginx` на домен и поддомены; проверить автопродление (`certbot renew --dry-run`); редирект 80 → 443.
**Готово, когда:** замочек в браузере; `http://` редиректит на `https://`; dry-run продления зелёный.

### 7. Боевой .env: секреты, CORS, URL-ы · 🟡 · 1:00
**Простыми словами:** собрать все боевые настройки в прод-.env и убрать dev-допущения.
**Где:** `.env` на VPS; `back/main.py` (CORS).
**Что конкретно сделать:**
1. Сгенерировать новый SECRET_KEY для JWT (не dev-овский!).
2. CORS: только боевой домен (убрать localhost:5173 из прода).
3. DATABASE_URL, SMTP_* (Эпик 4), FONDY_* (боевые ключи из Эпика 5 задачи 0a), `LLM_BASE_URL` (GPU-сервер Эпика 6), server_callback_url/response_url Fondy — на боевой домен.
4. Проверить, что `.env` в `.gitignore` и не попадал в git-историю.
**⚠️ Подводные камни:** секреты, засветившиеся в git-истории, считаются скомпрометированными — перегенерировать, а не «удалить файл».
**Готово, когда:** приложение работает на боевых настройках; в git нет ни одного секрета.

### 8. Скрипт деплоя одной командой · 🟡 · 1:00
**Простыми словами:** обновление продукта = одна команда, а не 10 ручных шагов по памяти.
**Где:** `deploy.sh` в корне репозитория (или Makefile).
**Что конкретно сделать:** скрипт: `git pull` → `pip install -r requirements.txt` → `alembic upgrade head` → `npm ci && npm run build` → `systemctl restart velora-api`. Вывод каждого шага виден, падение любого шага останавливает скрипт (`set -e`).
**Готово, когда:** правка в main выкатывается на прод одной командой за пару минут.

## Фаза 3 — Внешние сервисы

### 9. Боевые Google OAuth креды · 🟢 · 0:45
**Простыми словами:** вход через Google должен работать с боевого домена, а не только с localhost.
**Зачем:** сейчас OAuth-клиент настроен на localhost — на проде кнопка «Войти через Google» сломается.
**Что конкретно сделать:**
1. В Google Cloud Console: добавить боевой домен в Authorized JavaScript origins и redirect URIs (или создать отдельный прод-клиент).
2. Заполнить OAuth consent screen (название, логотип, ссылки на приватность/условия — страницы из задачи 11).
3. Прод-client_id в конфиги бэка и фронта.
**⚠️ Подводные камни:** consent screen со статусом Testing ограничен списком тестовых пользователей — для продаж перевести в Production.
**Готово, когда:** «Войти через Google» работает на боевом домене под любым Google-аккаунтом.

### 10. SPF/DKIM для почтового домена · 🟢 · 0:45
**Простыми словами:** DNS-записи, которые говорят почтовым серверам «эти письма действительно от нас» — без них письма CRM летят в спам.
**Зачем:** подтверждение регистрации и уведомления клиентам (Эпик 4) обязаны доходить до входящих.
**Что конкретно сделать:** включить SPF и DKIM у почтового провайдера (из Эпика 4, задачи 0a); добавить выданные TXT-записи в DNS; DMARC-запись `p=none` для мониторинга.
**Готово, когда:** письмо с кода регистрации приходит во «Входящие» (не спам) на Gmail и Яндекс; проверка SPF/DKIM в заголовках письма — pass.

## Фаза 4 — Юридика (без неё Fondy не пропустит)

### 11. Оферта, приватность, возвраты, контакты · 🔴 · 2:00
**Простыми словами:** четыре публичные страницы, которые обязаны существовать до приёма денег: публичная оферта (договор с клиентом), политика конфиденциальности (мы храним персональные данные клиентов студий!), политика возвратов, страница контактов с реквизитами.
**Зачем:** (1) Fondy проверяет наличие этих страниц при верификации мерчанта — без них боевых ключей не будет; (2) продавать подписку без оферты юридически нельзя; (3) CRM хранит персональные данные — политика обязательна.
**Где:** статические страницы на сайте (`/oferta`, `/privacy`, `/refund`, `/contacts`) + ссылки в футере лендинга/страницы входа.
**Что конкретно сделать:**
1. Взять типовые шаблоны для SaaS-подписки, адаптировать под Velora (наименование ФОП/юрлица, тарифы, порядок возвратов из Эпика 5).
2. Сверстать 4 статические страницы (простая вёрстка, без дизайн-подвигов).
3. Ссылки из футера; чекбокс «Принимаю условия» на регистрации.
**⚠️ Подводные камни:** тексты — шаблон + здравый смысл, но финальную вычитку сделать владельцу (это его юридическая ответственность); реквизиты должны совпадать с заявкой в Fondy.
**Готово, когда:** все 4 страницы открываются по прямым ссылкам; регистрация содержит согласие; ссылки отправлены в заявку Fondy.

## Фаза 5 — Надёжность

### 12. Бэкапы прод-БД по крону · 🟡 · 1:00
**Простыми словами:** каждую ночь база автоматически сохраняется; потеря сервера ≠ потеря бизнеса всех клиентов.
**Зачем:** продавать CRM без бэкапов = принять на себя риск чужих бизнес-данных. Экспорт студии из Эпика 6 — это фича для владельца студии, а не защита от потери сервера.
**Что конкретно сделать:**
1. Скрипт: `pg_dump -Fc velora > /backups/velora-$(date +%F).dump`; хранить 14 дней (`find -mtime +14 -delete`).
2. Cron на 03:00; копию — на внешнее хранилище (объектное хранилище или хотя бы второй сервер), не только локальный диск.
3. **Проверить восстановление:** `pg_restore` в пустую тестовую базу — бэкап, который никогда не восстанавливали, не считается бэкапом.
**Готово, когда:** в /backups появляются свежие дампы; restore в тестовую базу прошёл; копия уходит на внешнее хранилище.

### 13. Мониторинг: Sentry + uptime · 🟡 · 1:00
**Простыми словами:** об ошибках и падениях узнаёшь ты — раньше, чем платящий клиент напишет «у вас всё сломалось».
**Что конкретно сделать:**
1. Sentry (free tier): SDK в FastAPI (`sentry-sdk`) и в React (`@sentry/react`); DSN в .env.
2. Внешний uptime-пинг (любой бесплатный сервис) на `https://домен` и `https://api.домен/docs` с алертом на email/Telegram.
3. Логи uvicorn — в journald (уже даёт systemd), ротация по умолчанию.
**Готово, когда:** тестовое исключение видно в Sentry (бэк и фронт); остановка сервиса приводит к алерту в течение 5 минут.

## Фаза 6 — Приёмка

### 14. Финальный смоук-прогон на проде · 🟡 · 1:30
**Простыми словами:** пройти путь реального клиента от и до на боевом окружении, своими руками.
**Чек-лист (все шаги на проде, не на localhost):**
1. Регистрация нового аккаунта → письмо с кодом пришло во «Входящие».
2. Вход через Google.
3. Онбординг 5 шагов → CRM открылась.
4. Создать филиал, зал, услугу, сотрудника, клиента.
5. Создать занятие в Журнале, записать клиента, отметить посещение.
6. Создать операцию → Дашборд и Отчёты показывают цифры.
7. Публичная бронь без логина (`/public/booking/...`) → клиент и запись появились; письмо-подтверждение пришло.
8. Оплата тарифа Pro тестовой картой → подписка активна; вебхук дошёл по HTTPS без туннелей.
9. F5 на каждой странице — ничего не падает и не сбрасывается.
10. Зафиксировать результат в `docs/TZ/STATUS.md`: «прод проверен, дата, что работает».
**Готово, когда:** все 10 пунктов зелёные. **После этого продукт можно показывать и продавать первым студиям.**

## Порядок работы (что за чем)

Фаза 1 → 2 → (3, 4, 5 в любом порядке) → 6. Задачи 1–6 + 11 — предусловие боевой заявки Fondy (Эпик 5, задача 0a), поэтому стартовать эпик после Эпика 3, параллельно Эпикам 4–5. Задача 14 — строго последняя, после Эпика 6 (или без AI-пунктов, если Эпик 6 уехал за MVP).

## Что специально НЕ входит в Эпик 7

- **CI/CD-пайплайн** (GitHub Actions и т.п.) — deploy.sh достаточно для одного разработчика; автоматизация — когда появится команда.
- **Docker/Kubernetes** — прямой деплой на VPS проще и дешевле на этом масштабе.
- **Балансировщик, вторая нода, реплика БД** — post-MVP, при реальной нагрузке.
- **Лендинг-маркетинг** (продающая страница с фичами/ценами) — минимум для Fondy — оферта и контакты; полноценный лендинг — отдельная работа вне этого roadmap.
