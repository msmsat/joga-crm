### 2.0. Каркас приложения
Постоянная оболочка, внутри которой открываются все разделы CRM:
- **Боковое меню:** логотип и название студии; разделы — Дашборд, Сотрудники, Каталог, Клиенты, Отчёты, Онлайн-запись, Финансы, Уведомления, Лояльность, Velora AI, Настройки, Тариф и оплата. Внизу закреплены кнопка «Журнал» и профиль пользователя. Администратор и тренер видят в меню только доступные им разделы.
- **Верхняя панель:** заголовок текущего раздела; по центру — AI-строка «Спросите AI или поставьте задачу...»: пользователь задаёт вопрос и получает ответ ассистента в выпадающей панели, не покидая страницу; справа — кнопка AI (открывает панель чата с ассистентом) и кнопка «+ Создать» (переход к созданию записи в Журнале).
- **Панель AI-чата** доступна с любой страницы: новый диалог, история диалогов.
- Пользователь может переключать светлую и тёмную тему; интерфейс мультиязычный.
- Каждая роль имеет доступ только к своим разделам и данным — ограничение действует и в интерфейсе, и на сервере (получить чужие данные в обход интерфейса нельзя).

### 2.1. Авторизация и вход
Гостевого режима нет — продукт доступен только после регистрации.
- Пользователь может зарегистрироваться по email с подтверждением кодом — или войти через Google.
- Пользователь может войти в аккаунт, восстановить забытый пароль, сменить пароль.
- После регистрации владелец проходит онбординг из 5 шагов: название и тип бизнеса → направления занятий → контакты и адрес → часы работы → региональные настройки (часовой пояс, валюта, язык). Без онбординга работа в CRM не начинается.
- Пользователь может состоять в нескольких студиях и переключаться между ними.

### 2.2. Дашборд (Обзор)
Доступ: все роли.
- Пользователь видит 4 ключевые метрики: выручка за месяц, активные клиенты, записи на сегодня, уровень удержания — каждая с трендом к прошлому периоду; клик по метрике ведёт в соответствующий раздел.
- График аналитики с переключением периодов.
- Сводные виджеты, лента последних событий студии, список задач на сегодня.

### 2.3. Журнал
Доступ: владелец и администратор — всё расписание; тренер — только свои занятия.
- Сетка расписания дня: колонки — тренеры, строки — часы; карточка занятия показывает название, зал, заполненность (записано/мест), цвет тренера и статус (подтверждено / ожидает / отменено).
- Пользователь может листать дни и фильтровать расписание по тренеру и залу.
- Может перенести занятие перетаскиванием на другое время или к другому тренеру.
- По клику на занятие открываются детали: можно записать клиента на занятие или отменить занятие.
- Может создать новое занятие (название, тренер, зал, время, число мест, цена, уровень, оборудование) и завести нового клиента, не покидая журнал.
- Правая панель: сводка дня (загрузка, выручка) и AI-чат.

### 2.4. Клиенты
Доступ: владелец и администратор — все клиенты; тренер — только свои.
- Таблица клиентов с поиском и фильтрами-категориями: Все, VIP, Активные, Новые, С абонементом, Неактивные, День рождения — со счётчиками по каждой.
- Пользователь может добавить клиента (имя, фамилия, телефон, email, дата рождения, город, источник, теги) и удалить клиента с подтверждением.
- **Карточка клиента:**
  - личные данные и статус (новый / активный / VIP / неактивный) с возможностью смены;
  - теги: добавить, удалить;
  - абонемент: тип, остаток занятий, срок действия, **заморозка и разморозка**;
  - баллы лояльности, сумма всех покупок, число визитов;
  - история событий (оплаты, посещения, заморозки) с фильтрами;
  - заметки: создать, изменить, удалить;
  - график активности клиента;
  - действия: позвонить, написать, **записать на занятие прямо из карточки**;
  - настройки: уведомления и напоминания вкл/выкл.

### 2.5. Сотрудники
Доступ: только владелец.
- Карточки сотрудников, статистика по команде, фильтры и поиск.
- Владелец может добавить, изменить и удалить сотрудника (последнего владельца удалить нельзя).
- У сотрудника два независимых поля: **должность** (свободный текст с готовыми пресетами по направлениям — пилатес, йога, стретчинг, массаж, администрирование и т.д.) и **роль доступа** (Администратор / Тренер) — см. Целевую аудиторию.
- **Профиль сотрудника:** контакты, показатели (записи, посещаемость, загрузка, выручка), расписание на сегодня / неделю / месяц, рабочие часы по дням недели, привязка к залам, действия позвонить/написать.

### 2.6. Каталог
Доступ: только владелец.
- **Филиалы:** название, адрес, контакты, часы работы по дням недели, фото; создание и редактирование.
- **Залы** внутри филиала: название, вместимость, площадь, цвет (этим цветом занятия подсвечиваются в журнале), оборудование, стоимость часа; поддерживаются онлайн-залы.
- **Услуги студии:** название, длительность, цена, уровень сложности, оборудование; создание и редактирование.

### 2.7. Онлайн-запись
Доступ: только владелец. Центр управления каналами, через которые клиенты записываются в студию (настройка клиентского мини-приложения).
- **4 канала:** веб-виджет для сайта, Telegram, WhatsApp, Instagram — владелец может подключить и настроить каждый.
- **Правила записи:** за сколько часов до начала закрывается запись, на сколько дней вперёд открыто расписание, до какого времени клиент может отменить запись.
- **Брендинг:** язык и фирменный цвет мини-приложения; живое превью — владелец сразу видит, как приложение выглядит для его клиентов.

### 2.8. Финансы
Доступ: только владелец. Вкладки:
- **Счета:** касса, расчётный счёт, онлайн-эквайринг — балансы и изменение за день; можно добавлять свои счета.
- **Операции:** доходы и расходы с категориями (абонементы, услуги, сертификаты, аренда, зарплата, возвраты), способом оплаты (карта, наличные, QR, перевод), статусом и привязкой к клиенту и счёту.
- **Контрагенты:** юр. лица, ИП, физ. лица — реквизиты, категория, баланс, число сделок.
- **Документы:** акты, счета, договоры, УПД, отчёты — со статусами (подписан / ожидает / черновик).
- **Онлайн-платежи:** каналы приёма оплат и их показатели.
- **Способы оплаты:** какие методы оплаты доступны в студии.
- **Цели:** финансовые цели студии с прогрессом выполнения.
- **Отчёты:** финансовые сводки.
- **Зарплаты:** расчёты и выплаты тренерам.

### 2.9. Лояльность
Доступ: только владелец.
- Сводка эффекта программ: выручка от лояльности, удержание, средний чек, динамика по месяцам.
- **5 программ** — каждую можно включить и настроить:
  1. **Карты лояльности** — накопительные баллы и уровни, правила начисления и списания;
  2. **Скидки**;
  3. **Подарочные сертификаты** — выпуск и погашение;
  4. **Абонементы** — пакеты занятий для продажи;
  5. **Реферальная программа** — «приведи друга», бонусы обеим сторонам.

### 2.10. Отчёты
Доступ: только владелец.
- Выбор периода (день / неделя / месяц / год) и диапазона дат; экспорт отчётов.
- Сводные метрики и графики: выручка, доходы/расходы, посещаемость, средний чек.
- Разрезы: основные показатели, по продажам, по тренерам (занятия, выручка, рейтинг, удержание каждого), по услугам (выручка, доля, тренд), события; детальная таблица.

### 2.11. Velora AI
Доступ: чат-ассистент — все роли; настройка агентов — только владелец.
- **Чат-ассистент по бизнесу:** пользователь задаёт вопросы о своей студии и получает ответы с учётом контекста бизнеса; история диалогов; выбор модели ИИ и языка общения; настраиваемая инструкция для ассистента.
- **ИИ-агенты для клиентов:** автоответчики в Telegram и Instagram — включение, тон общения, максимальная длина ответа, для Instagram — режим «отвечать только в нерабочее время»; статистика: сколько обращений обработано, средняя оценка ответов.
- ИИ доступен по всему продукту: панель чата на каждой странице, AI-строка в верхней панели, AI-чат в журнале.

### 2.12. Уведомления
Доступ: только владелец.
- **6 каналов доставки:** Telegram, Instagram, WhatsApp, Email, SMS, Push — подключение каждого.
- Настройка по получателям: клиент, тренер, администратор, владелец — у каждого свой набор событий.
- **Матрица «событие × канал»:** владелец включает, какие события в какие каналы отправляются. Примеры событий для клиента: подтверждение записи, напоминание о занятии (за 24 часа и за 2 часа), отмена занятия, успешная оплата, «осталось 1–2 занятия», абонемент истекает, день рождения со спец-предложением, запрос отзыва после визита, возврат средств, задолженность.

### 2.13. Настройки
Доступ: владелец — всё; администратор и тренер — только персональные вкладки.
- **Общие:** название, описание, контакты, сайт, адрес, логотип студии.
- **Часы работы** студии по дням недели.
- **Внешний вид:** светлая / тёмная тема.
- **Команда:** список сотрудников студии с ролями и статусами, приглашения.
- **Уведомления:** личные настройки оповещений.
- **Интеграции:** WhatsApp, Telegram, Instagram, Google Calendar (двусторонняя синхронизация), 1С, ЮKassa.
- **Безопасность:** активные сессии, ключи доступа для интеграций.
- **Данные:** экспорт данных, удаление данных, резервные копии.
- **Тариф:** краткая сводка и переход к оплате.
- Переключение между студиями пользователя.

### 2.14. Тариф и оплата
Доступ: только владелец.
- Три тарифа: Старт (990 ₽), Pro (2490 ₽), Business (5990 ₽) — сравнение возможностей каждого.
- Калькулятор периода: 1 / 6 / 12 / 24 месяца со скидками до 40 % и наглядным расчётом экономии.
- Смена тарифа, история счетов, привязанная карта для оплаты.

### 2.15. Профиль
Доступ: все роли (каждый видит своё).
- Личные данные: имя, email, телефон.
- Смена пароля.
- Привязанные аккаунты (Google), активные сессии, переключение между аккаунтами.

### 2.16. Клиентское мини-приложение (вторая очередь — после MVP CRM)
Отдельное приложение для клиентов студии; клиент попадает в него по ссылке из Telegram, Instagram или с сайта студии. Брендинг (цвет, логотип, язык) и правила записи задаёт студия в разделе «Онлайн-запись».
- **Быстрая запись без входа в кабинет** — через главную страницу студии.
- **Личный кабинет клиента:** мои записи (запись и отмена по правилам студии), мой абонемент (остаток занятий, срок), карта лояльности (баллы, уровень), история посещений, отзывы и оценки занятий.

## 🏗 5. Architecture & Structure Rules

**Frontend (Feature-Sliced Modular Design):**
- Keep files small (< 200-300 lines). If a file grows, split logic into `hooks/` and UI into `components/`.
- **Feature Structure:** `src/pages/dashboard/<FeatureName>/`
  - `index.ts` (barrel export)
  - `<Feature>.tsx` (Orchestrator/Controller, lean)
  - `types.ts` (Interfaces) & `constants.ts` (Mocks/Static data)
  - `components/` (Local UI) & `hooks/` (Local logic)
  - `components/modals/` (Strictly 2-column premium grid, high-quality animations).

**Backend (Layered API):**
- `routers/` (Endpoints) -> `schemas.py` (Pydantic validation) -> `models.py` (SQLAlchemy ORM) -> `database.py`.

- **Inputs:** Minimalist. Light gray border. On focus: Soft glow with Primary Accent (`#FCAE91`) and light shadow.

## 🐛 7. Debug & Dev Protocol
- **CORS / Auth:** If UI gets 401/403/CORS, verify JWT format (`Bearer ${token}`) and FastAPI `CORSMiddleware` config (ports 5173 <-> 8000).
- **Types:** Strictly resolve TS errors. Use Optional Chaining (`client?.phone`) and type guards to prevent undefined crashes.
- **Backend Debug:** Use `engine = create_async_engine(URL, echo=True)` to debug SQLAlchemy queries. Rely on Pydantic `ValidationError` logs for bad inputs.

## 🧩 8. Data Flow & API Integration (Frontend-Backend Sync)
- **Single Source of Truth:** Бэкенд (Pydantic схемы) всегда диктует структуру данных. TypeScript интерфейсы на фронтенде (`types.ts`) ДОЛЖНЫ строго совпадать с JSON-ответами API (например, `first_name`, `last_name`, `total_spent`).
- **No UI-specific Keys in Core Types:** ЗАПРЕЩЕНО использовать в базовых интерфейсах сокращенные или специфичные для UI ключи (вроде `n` для имени, `i` для инициалов, `spent` для отформатированных денег).
- **Adapters & Computed Data:** При переходе с тестовых данных (mocks) на реальный API, полностью переписывай интерфейсы под бэкенд. Любые производные данные для UI (инициалы, форматирование валюты типа "₽15K", перевод статусов) должны вычисляться "на лету" внутри компонента или через хелперы (например, `getInitials(name, last_name)`, `formatCurrency(total_spent)`).
- **Graceful Fallbacks:** При маппинге данных с бэкенда всегда используй optional chaining и дефолтные значения (например, `data.last_name || ''`), чтобы UI не падал при отсутствии части полей.
